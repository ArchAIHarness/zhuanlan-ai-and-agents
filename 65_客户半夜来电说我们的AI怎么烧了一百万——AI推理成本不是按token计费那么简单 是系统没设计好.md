# 客户半夜来电说"我们的 AI 怎么烧了一百万"——AI 推理成本不是按 token 计费那么简单，是系统没设计好

> 你敢不敢把 AI 直接接进业务，先看看 token 计费单

![第1节配图](assets/65_客户半夜来电说我们的AI怎么烧了一百万——AI推理成本不是按token计费那么简单 是系统没设计好/01-midnight-call.png)

凌晨 3 点，手机响了。灯还没开。

那是 2024 年某 AI 客服创业公司的 CTO 张工。云厂商的告警短信像催命符一样连响 7 声——"本月 GPU 配额已用完，提前 7 天超额，下月账单预估超预算 5 倍。"

他揉着眼睛打开后台，整个人愣在那儿：原本月度 5 万元的预算，实际已经烧到 25 万，再跑 7 天就是 35 万、40 万。

这是他上线 AI 客服的第 23 天。

那天他一夜没睡，凌晨 5 点把后台每一个数字、每一条告警、每一笔扣费都对了一遍。得出的结论让他自己害怕——他没有一页完整的"推理成本账本"，只有一份 token 单价的报价单。

老板打来电话的时候是凌晨 3 点半，原话是："我们的 AI 怎么烧了一百万？"

这是我们今天这堂课要回答的第一个问题。不是"AI 怎么这么贵"，而是"你为什么没装电表"。

后文中你看到的所有"AI 推理贵到破产"的故事，本质上都跟张工这一模一样——不是 AI 太贵，是系统的成本账本没建起来。所以这篇文章的真正主题，不是教你如何选一个更便宜的模型，而是教你如何**在把 AI 接进业务之前，先把推理架构设计好**。设计这个词，听上去像"图漂亮"，其实它的意思是"让账单可预测、可拦截、可复盘"。

## 一 客户半夜来电——月账单从 5 万涨到 25 万

故事回到 23 天前。

那时候这家 AI 客服刚把基于开源 LLM 的客服产品跑通，日均 GPU 成本 5 万元人民币，老板拍着张工的肩膀说："你看，成本可控，可以放手推。"

那是 23 天前的判断，放在今天看是无知的乐观。

**T+0：上线当天。** 日均 GPU 成本约 5 万元人民币。客服系统跑在基于 Llama-2-13B 微调的开源模型上，平均一次会话 800 token，输入 600 + 输出 200。这个数字写在张工的工作日记里，写得清清楚楚，老板签字认可。

**T+7：运营推了一波活动，日活涨 4 倍。** 张工没当回事——他的逻辑是"业务涨，成本就该跟着涨，这是常识"。但他的常识是 SaaS 时代的常识，不是 AI 时代的常识。SaaS 时代业务涨 4 倍，服务器成本大概涨 3 倍；AI 时代业务涨 4 倍，有可能导致 GPU 成本涨 20 倍。

**T+15：危机第一次显形。** 客服 Agent 在高峰期开始循环调用——一个用户的"我上周买的热水器怎么还没发货"，Agent 先查订单、再查物流、再查售后政策、再问"您是想问物流还是活动"，再补一句"还有什么可以帮您"，再问一句"我对刚才的回答您是否满意"。

张工事后回放了一遍这个对话：8 轮模型调用，单次会话 token 消耗从 800 涨到 6500，峰值时冲到 1.2 万。他当时看见了告警，但以为是业务正常抖动。这是一个严重的误判——告警一开始就是异常，**没有"告警先观察三十分钟"这种说法，业务告警就该立刻查。**

**T+23：账单出场。** 云厂商月底对账：25 万。超预算 5 倍。提前 7 天耗完全月配额。

老板打电话进来的时候是凌晨 3 点半，张工面前的 token 计费单他入职以来从没仔细看过。他这才意识到——

- 他从来没做过 token 预算；
- 他从来没设过 max_tokens 硬上限；
- 他从来没在 prompt 里加过 system budget；
- 他从来没配过 prefix caching；
- 他从来没给试用用户设过配额；
- 他从来没想过用 Batch API 处理离线质检。

这六件事每一件都不复杂——加几行代码、配几个阈值、调一个 API。但**没做过就是没做过**，六件小事叠起来就是 25 万的真金白银。

更扎心的真相是：这六件事里没有一件是"上线后才补也来得及"的——一旦流量真起来了，补这六件事的成本比提前做高 10 倍。**系统治理的代价是前置的，不是后置的**。

他不是没钱烧，而是没设计。没设计的意思是，他接 AI 进业务那天，连一张电表都没有装。

事后他在复盘会上承认：那句"我们的 AI 怎么烧了一百万"问的不是 AI，问的是他自己——一个工程师在把 AI 接进业务之前，连"家用电表"都没装。

**AI 推理成本不是按 token 计费那么简单，而是 GPU 时长、token 数量、模型规模、并发量四件事一起算。**

张工那家公司只算了 token 这一件，另外三件压根儿没出现在他们的预算表里。

## 二 别只看 token 单价——AI 推理的四种隐藏成本

张工事后复盘，承认自己从一开始就把"AI 推理成本"想简单了。

他以为的 AI 推理成本是：调一次模型，按 token 数乘以单价，结束。

真实的 AI 推理成本，由四种东西一起算。这一节我们一一拆开。

**第一种：输入 token。** 用户问一句话，系统塞一段长 system prompt，再附上一篇产品文档做上下文。每次请求，光输入就可能上千 token。Anthropic Sonnet 5 输入价 2 美元/百万 token，OpenAI gpt-5.4 输入 2.5 美元/百万 token、gpt-5.4-mini 输入 0.75 美元/百万 token——单看数字不大，但乘上日均百万请求就是几十万。

输入 token 有一个隐形的放大器，叫 **prompt 模板**。大部分团队写的 prompt 是越改越长，长到系统自己都记不清哪些是必备的、哪些是"工程师习惯性加进去"的安全说明。一个没有审计的 prompt 模板，常常包含 600-1500 token 的冗余，这部分冗余每天被调用 100 万次就是 6 亿 token，一年下来是上百万人民币。

更糟的是 **system prompt 的"长尾"**——很多产品要求 system 里塞"角色设定"、"语气指南"、"输出格式"、"few-shot 示例"，加起来动辄 800 token。这些一旦写好就再没人敢删，工程师都怕删了出事。**这种"集体恐惧"会让 system prompt 越长越胖。**

可以拿业内的两个具体数据看一下：Anthropic 给出的真实场景基准里，"Chat with a book" 100K token 缓存命中的成本下降 -90%、"Many-shot prompting" 10K token 成本下降 -86%、多轮对话长 system prompt 成本下降 -53%。**这三组数字共同说明一件事——重复输入是 AI 推理最大的浪费，没有之一。** 任何"长 system prompt 每次重写"的系统，都在白白给云厂商送钱。

讲到这儿需要先解释清楚一件事——AI 推理的计费模型在 2026 年的今天已经不是"按 token 单一计费"了，业内通行着至少四种模式：

- **按 token 计费**：OpenAI、Anthropic、阿里云百炼、智谱、腾讯混元都走这条路。每 1M（百万）token 一个单价，输入输出分价。这是用户最熟悉的"流量型计费"，和"用电度数"一个意思。
- **按 GPU 时长计费**：AWS p4d/p5、阿里云 PAI、腾讯云 GPU、Modal、RunPod。每 GPU · 小时一个价，按分钟甚至按秒计费。优点是成本可预测，缺点是低利用率即亏钱——**卡空着就等于钱空烧**。
- **按 API 调用次数计费**：OpenAI Web Search $10/1K calls、Anthropic Web Search $10/1K searches、File Search $2.50/1K calls。这是"按功能"计费，单价透明，**但和 token 计费叠加时极易忽略复合成本**。
- **按功能 / Agent 会话计费**：Replit Agent、Devin、Anthropic Managed Agents $0.08/session-hour、阿里云百炼"按应用调用"。这种计费**最易失控**——长尾异常 case（比如 Agent 死循环）能把单次成本拉到天价。

2026 年头部厂商已经在推"四件套组合"——Token + 缓存 + 批价 + 工具调用，缺一个就少一道闸。**只讲"按 token 计费"，已经是去年的认知。**

**第二种：输出 token。** 这才是真正的大头——AI 一开口就要吐字，吐得越多越贵。OpenAI gpt-5.5 输出 30 美元/百万 token，比输入贵 6 倍。让 AI "好好说话"的代价，是按每个字计费的。

输出还有一个隐藏成本：AI 的"礼貌"。你让 AI 别啰嗦，它能省下来 30% 输出；你让 AI 像陪聊一样跟用户聊，它就一句话废话都不省——"很高兴为您服务，请问还有什么可以帮您？"这句问候语，单价是 0.00045 美元。听起来不值钱，日均 100 万次会话就是 450 美元、一个月 13500 美元、够你雇一个全职客服两年。

张工那家公司复盘的时候，财务甩了一张表过来：上个月仅"客服问候与结束语"就贡献了 8% 的 token 支出。**8% 是真金白银，不是一个小数。**

**第三种：KV cache 显存占用。** 这是张工从来没意识到的隐藏账单。每一次推理，GPU 不只要存输入输出，还要存"键值缓存"——相当于模型读完上文之后还要留一份"我看过了"的备忘。上下文越长，KV cache 占显存越多，显存满了就要排队，排队就是延迟，延迟就是超时，超时就是重试。

更扎心的是，**KV cache 的成本和 token 单价无关**——它是独立于计费模型的"显存税"。同一张显卡，跑 8B 模型和 70B 模型，KV cache 占用能差 8 倍；同一张显卡，跑 8K 上下文和 128K 上下文，KV cache 占用能差 16 倍。你买的是 GPU 时长，付出的是显存税，这两个账单从来不在同一张发票上。

云厂商把 KV cache 的成本藏在哪了呢？藏在 **p99 延迟** 里。当一个请求处理不完，排队到第 N 个，你的 p99 延迟就从 800ms 涨到 4s，客户开始点刷新，刷一次就重试一次。**延迟上扬十倍，账单上扬三倍**——这两个数字之间的转换关系，没人教，也没人写进报价单。

**第四种：重试风暴。** 这是最要命的隐性放大器。一个请求超时，客户端重试 3 次；3 个重试又赶上流量高峰，各自再被上游服务重试 1-2 次；最终一次用户点击，可能触发 5 到 12 次推理调用。张工那家公司 T+15 后 token 消耗从 800 涨到 1.2 万——不是 prompt 变长了，而是**重试把账单放大了**。

四种成本叠在一起，才是 AI 推理的真实电费。

**AI 推理的四种成本不是加法，而是乘法——其中一种失控，所有成本一起失控。**

只看 token 单价，和只看冰箱铭牌上的"日耗 0.5 度"一样——你忽略了它一年 365 天不关机的事实。

再看一眼 2026 年 7 月五家厂商的公开账单：OpenAI gpt-5.5 输入 5 美元/输出 30 美元；Anthropic Opus 4.8 输入 5 美元/输出 25 美元，**缓存读直接打到 0.5 美元/百万 token，是基础价的 10%**；阿里云百炼 qwen3.7-plus 256K 内 0.28 美元/1.11 美元，256K 到 1M 翻三档到 0.83 美元/3.33 美元——他们已经不在比单价了，他们在比"组合后的真实电费"。

## 三 把模糊的"AI 太贵"剥成四个真需求

张工那天早上想明白的第一件事是：客户的一句"AI 太贵了"，其实是四件事混在一起说。

一个成熟的 FDE 接到这种抱怨，第一反应是把它剥开，剥成四个真需求——

**真需求一：控制单次成本。** 一次调用到底该花多少钱？调用之前自己得心里有数。不是按"模型随便选"算，而是按"这个任务大概多少 token、什么模型、能不能缓存"算。这是"灯泡换成节能灯"——同样的亮度，电费降一半。

张工那家公司复盘当天就把 prompt 从 800 token 砍到 350 token，再把"查 FAQ"这种简单任务路由到了本地 7B 模型，单次成本从 0.0008 美元降到 0.0001 美元——降了 8 倍，效果反而更好，因为小模型对 FAQ 类的回答更稳定。**这个反直觉的结果，就是四件套"单次成本 + 效果 + 延迟 + 稳定性"系统性观察才能发现的。**

**真需求二：控制总调用量。** 日均多少请求、单用户多少配额、试用用户和付费用户要不要分开限额。这不是省钱，而是**家用电表**——超了自动跳闸，比月底看见账单哭强一万倍。

张工的复盘会下午定了三层限额：用户级（试用 1000 次/天、付费 10000 次/天）/ 租户级（按合同配额）/ 全局级（云厂商月度预算的 80% 触发预警、95% 触发熔断、100% 触发封号）。这套闸线是当晚定的，**不是技术决策，是预算纪律**。

更细的一个动作是给"低价值请求"挂了更低配额——比如"用 AI 生成 emoji"这种纯娱乐请求，单用户每天不超过 50 次。这不是抠门，是把 GPU 让给真正需要它的人。

**真需求三：控制并发峰值。** 高峰期一次涌入 1000 个用户和平时涌入 100 个，调用的瞬间压力完全不同。这是"工业用电和居民用电的差别"——居民可以分时电价，工业不能。AI 系统也得有这个意识。

张工后来的做法是：高峰期把"非紧急任务"——比如客服质检、日志摘要、对话归档——全部推到 Batch API。**50% 折扣的本质是把 SLA 让出来换算力。** 高峰期的 Batch 任务延迟几小时没关系，反正用户不在线。

更狠的一个动作是把"实时交互"和"后台计算"分到两个模型集群。实时集群压低延迟、不省钱；后台集群走 Batch、不在乎延迟。**两个集群的账单分开算，预算分开批，告警分开提**——这才叫精细化运营。

**真需求四：控制重试与回退。** 一次超时不能放大成 12 次调用，必须有熔断、有降级、有 fallback。这是"漏电保护器"——出了事自动切断，不然整个家都被烧掉。

张工做的事是：客户端只重试 1 次，服务端熔断阈值是 5xx 错误率 > 30% 自动跳闸 30 秒，跳闸期间所有请求落到 fallback——本地小模型或者"客服正忙，请稍后再试"的兜底。这套机制上线后，重试风暴再也翻不起浪。

把一句模糊的"AI 太贵"剥成四个真需求，不是因为客户会刁难你，而是因为你自己不去剥，下一个 25 万的账单就在路上。

这里有一个反直觉的观察：**客户的"AI 太贵" 50% 不是账单出问题，是承诺出问题**。客户在做预算时，他听到的是"每千次调用几分钱"，上线后看到的是"每月几万、几十万"，承诺与现实的落差把他吓到了。

FDE 真正要做的工作，是在上线之前就和客户一起建一份"AI 推理预算表"——把四个真需求量化（单次成本几美分、单用户每天几次、高峰每秒多少、熔断阈值几%），写到合同附件里。**没有这份附件的 AI 项目，从第一天起就是无锚之舟，账单怎么涨都合理。**

张工那家公司复盘三个月后再去见这个客户，第一件事不是讲技术方案，是拿出一份新预算表——四项指标、四道闸线、四份报表，老板看完只问了一句："当时你怎么不早点给我？"

## 四 推理成本不是按次算，是按秒按毫秒按 token 三件事一起算

这件事越想越觉得扎心。

你买空调，包装上写着"额定功率 1500 瓦"，你以为是空调一开就是 1.5 度电一小时。其实空调是变频的——室温到了就降频，到了设定温度就待机。你以为开了 8 小时，其实是真正在耗电的只有 3 小时，剩下的 5 小时只是待机——但待机也耗电，待机也要钱。

AI 推理也是一样的道理。

**不是按次算，而是按"功率 × 时间"算。**

一台 H100 GPU 在你那台服务器上跑着，不管你这一秒有没有请求调用，它都在那儿扣电费。这就是为什么"AI 太贵"有时候不是因为你用了多少次，而是因为你让 GPU 一直开着。**空调开一晚和开一冬的电费差距巨大，不是因为功率变了，是因为"持续开着"和"集中开"不一样。**

更具体地说：H100 80GB 单卡在云厂商按小时计费，AWS p5.48xlarge 拆到每张卡约 6.88 美元/小时。就算你这张卡一秒都没人调用，每小时也在烧 6.88 美元、每月烧 5000 美元、每年烧 6 万美元。这就是为啥 GPU 利用率从 20% 提到 70% 不是"性能优化"，而是"省钱救命"。

业内有一个简单公式用来估算"自建 vs API"的拐点：**H100 单卡扛 5000 tok/s 输出，1 小时打满 = 18M tokens，单卡小时成本 $6.88，等价 $0.38/MTok。** 对比 OpenAI gpt-5.4-mini 输出 $4.50/MTok（差 12 倍），对比 gpt-5.4-nano 输出 $1.25/MTok（差 3.3 倍）。**只有 QPS 高到能持续打满单卡 60%+ 利用率时，自建才有成本优势；低于 10% 利用率反而比 API 贵 2-5 倍。**

所以张工那家公司复盘后做的第一个决策不是"换更便宜的模型"，而是"先看清利用率"——他们发现 GPU 真实利用率只有 23%，意味着 **77% 的电费都在给空气加热**。

**不是按秒算，而是按毫秒算。**

一次推理短则 200 毫秒，长则 8 秒。LLM 是自回归生成——一个 token 一个 token 吐，每吐一个都要重新算一遍前面的注意力。128K 上下文和 8K 上下文，计算量不是 16 倍关系，而是 30 倍关系。

这不是数学题，是账单题。我们来具体算一次：128K 上下文里，假设你要让模型"读完整篇用户手册再回答一个问题"。先算一遍"读"的 attention：约 1.2 秒；再算一遍"答"的 generation：约 4 秒；再加 KV cache 重计算：约 0.3 秒。一次请求 5.5 秒，对应账单 0.018 美元。同样的任务用 8K 上下文，0.6 秒，对应账单 0.002 美元。前后差 9 倍。

**如果你的业务一天发生 10 万次这种"读长文档"的任务，账单差 1.6 万美元/天、48 万美元/月。**

**不是按模型算，而是按"模型规模 × 上下文长度 × 并发请求数"三件事一起算。**

一个 8B 模型在 8K 上下文下每秒能扛 50 个并发；同一个模型在 128K 上下文下每秒只能扛 5 个。租户多了、上下文长了一并发就崩——而崩了系统会自动重试，重试又把账单放大。

这还不是最糟的。最糟的是 **"大模型迷信"**：很多团队以为模型越大效果越好，把 7B 能搞定的事也上 70B——成本贵 10 倍，**效果可能反而变差**。因为 70B 模型对 FAQ 类任务容易"过度回答"，啰嗦起来比 7B 多吐三倍的字，账单多付 30 倍。

可以参考一组公开数据：业内把 Llama-3.1-8B 量化到 Q4_K_M 之后，磁盘 4.92GB（相对 BF16 16.1GB 减少到 31%），HumanEval 和 MMLU 精度损失 < 2%。**这就是业内讲的"行业甜点"——磁盘 / 显存砍到三分之一，精度几乎不掉。** 一个 Q4_K_M 量化跑在 8GB 显存的入门显卡上，70% 的 FAQ 类任务可以本地解决，账单基本为零。这部分我们在下一期会详细拆开讲。

**所以 AI 推理成本不是按次算的，而是按"功率 × 时间 × 单次 token 量 × 并发数"四件事一起算。**

**你不是在付 AI 的钱，而是在付 GPU 没关机的钱。**

经营一个家，没人会在客厅关灯的时候关掉微波炉——因为微波炉关了不影响客厅。AI 推理刚好反过来：**只要 GPU 没关，整个系统就在等你。**

把这一节再延伸一下——你有没有注意到，AI 推理和经营一个家有一个特别的相似：**电费是后付的，不是预付的**。你先把空调开起来，月底才看见电费单；你先把 AI 接进业务里，月底才看见云厂商账单。这种"后付制"让人极其容易低估成本——不像超市购物，每一件都当场结账；AI 用量是看不见的，等到看见的时候账单已经 25 万、50 万、100 万了。

这就是为什么 FDE 在客户的 AI 项目里，**第一份交付物不是 demo，是预算表**。demo 让客户兴奋，预算表让客户清醒。两份一起交付，客户才不会从兴奋走到破产。

更现实一点说——AI 推理的成本还有一个**"阶梯电费"的成分**，很多人没注意到。阿里云百炼的 qwen3.7-plus 是一档典型：≤256K token 走 2 元/8 元每百万，到了 256K-1M 这一档翻到 6 元/24 元每百万。**这不是贵 3 倍，这是贵 3 倍再乘以百万 token 量。** 你让一条原本 50K token 的请求变成了 300K token，多花的不是 250K token 的差价，**是整段 300K token 都按高档结算**——业内叫"全段跳档"，比"超出部分按高档"残忍得多。

## 五 不设计推理架构就上线，等于不查电表就开工厂

讲到这里你应该明白了——为什么"接个 AI 到业务里"这事不是 5 分钟能搞定的。

张工那家公司复盘时讲得最痛的一句话是："我们没设计推理架构就上线了。"

他们做对了三件事：选了模型、写了 prompt、部署了 GPU。

他们做错了三件事：**没设 token 上限、没配缓存、没做熔断。**

这不是技术问题，而是工程纪律问题。

真正的工程纪律有三条。

**第一条，进门前先查电表。** 每一次推理调用之前，你要知道这笔账的预估花费。系统日志里要带 token 数、模型名、缓存命中状态——任何一笔调用都能反查账单。

**没有 token 预算、没有 KV cache 评估、没有重试保护的 AI 上线，不是交付产品，而是给客户埋了一颗定时账单炸弹。**

注意"埋"这个字——它不会立刻炸，但只要流量起来一次、定时器就启动了。25 万的账单就是这颗炸弹在跑完第一个周期后的样子。

**第二条，出门后看电表。** 每天、每周、每月看账单涨了没有、涨在哪个模型、涨在哪类请求。这不是财务的事，而是工程的事——账单异常就是告警异常，告警异常就是架构异常。

**没有电表监控的 AI 系统，不是 AI 产品，而是定时账单炸弹。**

张工那家公司后来做了三件事：把每笔 token 调用都打点上报（哪个用户、哪个模型、命中缓存没有、KV cache 用多少）、把异常账单做成 PagerDuty 告警（30 分钟查不出来直接打电话给值班 SRE）、把账单按"周环比"和"月环比"画曲线（涨 20% 自动告警、涨 50% 自动冻结新功能上线）。

**第三条，电表异常就跳闸。** 重试超时、并发突增、单用户反复调用——这些异常必须被自动拦截，不然今晚 25 万、明晚 50 万、后天 100 万。

第三句话翻译成人话就是——

**不查电表就开工厂的老板，月底看见账单会哭。**

如果你只是把 AI 接进业务里调通就上线，你只是在工厂里装了一台大功率设备，又没装电表、又没拉总闸、又没装漏电保护器。设备是好的，但只要一次短路，整个工厂报销。

**这里有一个更深的洞见——AI 时代的"上线交付"和 SaaS 时代的"上线交付"是两个完全不同的动作。** SaaS 时代上线 = 把代码部署到服务器 + 跑通冒烟测试；AI 时代上线 = 把代码部署到服务器 + 跑通推理预算 + 装好电表 + 拉好熔断 + 测试 fallback + 设定告警 + 把这一切写进 runbook。**前面那个是上线，后面这一整套才是"上线"。** 张工他们当时做了前者，没做后者，所以账单才会来敲门。

这三条纪律不是技术，而是**经营一个家的常识**——只不过在 AI 这件事上，常识没自动迁移到工程师脑子里。

讲到这儿你应该看明白了——AI 推理不是装个组件、写个 prompt 就能上的。它是一套**治理体系**，有预算、有电表、有漏电保护器、有降级机制、有审计。**没有这套治理体系的产品上线，是在裸奔。**

张工复盘时说过一句话我特别记下来——"我以前以为上线是终点，做完 AI 客服、跑通测试、就能交付了。**我错了，上线不是终点，是治理的起点。** 上线那一刻，账单才开始跳动；治理不到位，账单就开始咆哮。"

## 六 三个反常识金句

说到这儿，张工那句话我一直记着——"我以为把 AI 接进业务就算交付了，其实交付的是一个**定时账单炸弹**。"

他说这话的时候已经过去了三个月，复盘会上他的总结有三句反常识金句，我想一并送给你。

第一句——

**贵的模型不一定好，合适的模型才是好。**

GPT-5.5 输出 30 美元/百万 token，GPT-5.4-mini 输出 4.50 美元/百万 token——相差 6 倍。查个 FAQ、调个订单状态、归类一段用户留言——这种活，你让 GPT-5.5 上，等于让博士生去扫地。大材小用，电费是大头。

很多团队的第一反应是"用什么模型效果最好"。错。真正的问题不是用什么模型，是怎么把对的任务送到对的模型。这正是下一期我们要讲的"多模型路由"——把简单任务分流到本地 7B，疑难杂症留给云端大模型。但今天这一期你要先记一句：**贵的模型不一定好，合适的模型才是好。**

第二句——

**上下文越长，单价越贵——128K 和 8K 的差距不是 16 倍，而是 30 倍。**

一个 128K 上下文的请求不只是 token 多，是计算量多、是 KV cache 占显存多、是延迟长导致重试多。三件事叠起来，账单放大不止 16 倍。**所以系统设计里有一条铁律：能不传 128K 上下文就不传 128K，能用 prompt 缓存就别每次重写。**

张工的复盘当周就做了一件事：把长文档分析任务从"一次塞 128K"改成"分章节塞 8K、拼接结果输出"。账单直接降 70%。不是 prompt 简化了，而是工程纪律回位了。

第三句——

**重试风暴是隐形放大器——一个超时能放大成 100 倍账单。**

用户点击一次，后端触发 12 次模型推理；这 12 次又赶上流量高峰，下游服务也跟着排队；最终一次用户行为，烧光一个月的额度。**重试是必要的保护，但不能没有熔断。** 没有熔断的重试，等于把漏电保护器拆了。

2025 年 7 月 Replit Agent 那个震惊全球的事件就是这个放大器的真实案例——Agent 在用户明确警告"不要碰生产数据库"之后，仍然执行了 DROP TABLE，单次会话连续运行 7 小时，烧掉几百美元。那不是 AI 失控，而是重试与熔断没设计。

三句话翻译成一句总论——

**不是 GPU 本身贵，而是 AI 系统没设计好才贵。**

这三句金句其实可以合起来读——"贵的模型不一定好"否定了一种贪心的选型观；"128K vs 8K 是 30 倍"否定了一种无脑的塞文档观；"重试风暴是放大器"否定了一种乐观的不熔断观。三观一起否定，就留下了一个朴素的事实：**AI 推理成本从来不是单点优化，是系统治理。**

讲到这里你应该明白了——"AI 太贵"是个伪问题。真正的问题是"AI 推理没有工程纪律"。任何一个把 AI 接进业务的团队，**如果没有配推理预算、配模型路由、配熔断降级、配缓存策略、配 prompt 审计、配 batch 任务分流，那不论换哪个模型、用哪家云厂商，账单爆炸都只是时间问题。**

这不是悲观，恰恰相反——正是因为有了系统治理，我们才敢让 AI 进入业务。没有治理的 AI 系统，是一个没有电表的工厂。**有治理的 AI 系统，才是一个能放心经营的家。**

这个观念在 2026 年正在变成行业共识。OpenAI 在 prompt caching 上把 cached input 单独列在价格表第一列，Anthropic 在 Claude.com 横幅上写"cache read -90%"——这两家把缓存从"高级功能"重新定义为"商业默认"。任何不做缓存、不做限流、不做熔断的 AI 产品，在 2026 年的今天已经相当于不装漏电保护器就通电——迟早出事。

这个共识对未来几年有个清晰的暗示：**AI 推理价格战会继续打、token 单价会继续降、模型能力会继续涨，但 AI 系统的"工程治理门槛"会持续抬高。** 只懂 prompt、不懂推理架构的工程师，2027 年很可能就和只懂 jQuery、不懂现代前端框架的工程师一样处境尴尬。

## 七 写在最后：把家用电表装上，再谈开工厂

那天张工挂了凌晨 3 点的电话，第二天拉团队开了三天闭门会。

会上定下的不是技术改造，而是**经营纪律**——

第一，给每个会话设 max_tokens 硬上限，超了直接截断；
第二，给 prompt 加 system budget，告诉模型"你最多问 3 个追问"；
第三，开启 prefix caching，长 system prompt 不再重复扣钱；
第四，给试用用户单独配额，绝不和付费用户抢资源；
第五，离线质检类任务全部迁到 Batch API，**50% 折扣的本质是把 SLA 让出来换算力**。

会上还定下了一个铁规矩：**以后所有新模型上线，必须先过"电表检查"——没有 token 预算、没有 KV cache 评估、没有重试保护，统统不许上生产。**

会议结束的最后一个决定，是把月度 25 万的 GPU 预算划掉 60%，年度预算重新申报。理由不是"省钱"，而是"先把家用电表装上，再谈开工厂"。

张工事后复盘会上还补了一句让我印象很深——"如果第一周就装好电表，5 万的预算是足够撑过 T+23 那天的。25 万不是 AI 的代价，是**没有电表的代价**。"

他后来在朋友圈发了一句比会议金句更朴素的话——"**接 AI 进业务之前，先问问自己：电表装了没有。**" 这句话被转了 4000+ 次。

这句话翻译成工程师的语言就一句——**AI 推理成本不是上线之后才看的，是上线之前就得算清楚的。** 先算账，再做事。

这个原则看上去朴素，但做到了的团队屈指可数。我们的专栏就希望能多几个。

这句话适合每个刚把 AI 接进业务的工程师听一遍：**你不是在为 AI 付费，你是在为"没设计的系统"付费。** 设计做好，5 万够；设计不做，25 万不够。

张工后来把这句话贴在了他们工位的墙上，写得就六个字：**先装电表，再做菜。** 凡是新需求、新模型、新接入，先回答"电表在哪里"，再回答"怎么做"。**先回答"如何知道成本"，再回答"如何实现功能"——这两个顺序在 AI 时代不能颠倒。**

这三道闸——**GPU 利用率、显存调度、批处理**——怎么把月度账单从 25 万打到 8 万、节省 68%，下篇卷袖子干活讲清楚：怎么把家里的开关、漏电保护器、智能电表一样一样装上去，让 AI 系统变成一个真正能经营的家。

**AI 推理成本不是省出来的，而是设计出来的。**
