  用户浏览器(薄)                         用户浏览器(厚)
    ┌────┐                                ┌────────────────┐
    │ UI │                                │ UI             │
    └─┬──┘                                ├────────────────┤
      │                                   │ 状态机         │
      │                                   ├────────────────┤
      ↓                                   │ 工具调用       │
    ┌─────────────────┐                   ├────────────────┤
    │ 云端厚          │                   │ 业务接口       │
    │ ┌─────────────┐ │                   ├────────────────┤
    │ │ Agent Runtime│ │                   │ 交互逻辑       │
    │ │ 状态机      │ │                   └────────┬───────┘
    │ │ 工具调用    │ │                            ↓
    │ │ 业务接口    │ │                   云端(薄,只协调)
    │ │ 交互逻辑    │ │                   ┌──────────────┐
    │ └─────────────┘ │                   │ Token 网关   │
    └─────────────────┘                   │ 跨设备同步   │
                                           │ 数据存储     │
                                           └──────────────┘

**文字锚点建议**:
- 左侧上方标:"改造前 1C1G × 1000 = 30+ 万/年"
- 右侧下方标:"改造后浏览器承担 70% 算力,云端只承担协调"
- 中间用一条粗箭头连接,写"边缘推理 = 把家搬到用户设备上"

### 2.3 第一招 · 三条技术路径选型决策树(图 3)

**目的**:让读者一眼看到"WebContainer / WASM / QEMU JSLinux"各自适合什么,**且明确警示 WebGPU 假命题**

**可视化思路**(决策树 + 假命题警示):
```
                你需要在浏览器跑什么?
                          │
      ┌───────────────────┼───────────────────┐
      ↓                   ↓                   ↓
   完整 Node.js        轻量二进制        任意 x86 二进制
      │                   │                   │
      ↓                   ↓                   ↓
   WebContainer        WASM             QEMU JSLinux
   (StackBlitz)       (Rust/Go)         (Fabrice Bellard)
   毫秒启动            即时启动          点击即用
   几十 MB             几 MB             64-256 MB
   ┌─────┐             ┌─────┐           ┌─────┐
   │ Node│             │ 数据 │           │ Linux│
   │  JS  │            │ 解析 │           │  VM  │
   └─────┘             └─────┘           └─────┘

   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   ⚠️ WebGPU 跑 LLM = 假命题
   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   显存:浏览器 GPU ≤ 用户 GPU(8-24 GB)
        Qwen-72B fp16 = 140GB
        Qwen-7B fp16 = 14GB(连最小都放不下)
   KV cache:浏览器 reload 后全丢
   通信:GPU→CPU 数据传输比云端慢一个数量级
```

**文字锚点建议**:
- WebGPU 假命题用红色边框 + 醒目警告
- 三条路径用蓝/绿/橙色块区分
- 底注:"WebGPU 适合小模型推理(embeddings / 分类 / 图像),不适合跑 LLM"

### 2.4 第二招 · Provider 多路由架构图(图 4)

**目的**:讲清"agent 触 LLM 时通过 Provider 抽象层路由,不绑死任何一家"

**可视化思路**(Adapter + 路由 + 多 provider):
```
                    agent 代码(只认 OpenAI 兼容 API)
                              │
                              ↓
                   ┌─────────────────────────┐
                   │   ProviderRouter       │
                   │   (单 API key)         │
                   │                         │
                   │   路由策略:             │
                   │   - 按消息类型选模型    │
                   │   - 按成本选模型        │
                   │   - 按 SLA 选模型       │
                   │   - fallback 兜底       │
                   └─────────┬───────────────┘
                             │
            ┌────────────────┼────────────────┬──────────────┐
            ↓                ↓                ↓              ↓
    ┌──────────────┐  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
    │ GLM 4.5 Air  │  │ DeepSeek     │ │ Ollama 本地  │ │ OpenAI 兜底  │
    │ $0.13/$0.85  │  │ V4 Flash     │ │ 免费         │ │ $5/$30       │
    │ 简单 Q&A     │  │ $0.14/$0.28  │ │ 70% 流量     │ │ 关键业务     │
    └──────────────┘  └──────────────┘ └──────────────┘ └──────────────┘
            ↓                ↓                ↓              ↓
            └────────────────┴────────────────┴──────────────┘
                                    ↓
                            月度账单(大幅下降)
```

**文字锚点建议**:
- 顶部标签:"Provider 抽象层"
- 中间路由策略框突出"按成本 + 按 SLA + fallback"
- 底部四个 provider 用不同颜色块,标价格 + 适用场景
- 底注:"DeepSeek V4 Flash 比 OpenAI GPT-5.5 便宜 35-100 倍(L4)"
