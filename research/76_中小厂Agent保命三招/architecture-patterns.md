### 2.4 自建 Adapter 的最小代码骨架(供参考)

```python
# 概念代码(非生产级,只示意架构)
class ProviderRouter:
    def __init__(self):
        self.providers = {
            "fast": OpenAIProvider("glm-4.5-air"),       # 便宜日常
            "code": OpenAIProvider("deepseek-v4-flash"),  # 代码
            "long": OpenAIProvider("qwen-long"),          # 长文档
            "fallback": OpenAIProvider("gpt-5.5"),        # 兜底
        }
        self.local = OllamaProvider("qwen2.5-7b")         # 本地免费

    def route(self, message_type: str, message: str):
        # 70% 流量走本地,20% 走便宜云端,5% 走兜底
        if self.can_handle_locally(message):
            return self.local.call(message)
        try:
            return self.providers.get(message_type, "fast").call(message)
        except ProviderError:
            return self.providers["fallback"].call(message)  # 兜底
```

**说明**:**这段代码是示意架构,不是给读者的可运行 SDK**。具体实现靠读者用 LiteLLM / LangChain / OpenRouter / 自建。

## 四、给工程师的可落地清单(供参考)

### 4.1 第一招落地(边缘推理)

1. **评估**:你的 agent 状态机/工具调用/业务接口,**有多少能在浏览器侧完成**?
2. **选型**:
   - 完整 Node.js 需求 → WebContainer
   - 轻量二进制/数据解析 → WASM
   - 完整 Linux VM 需求 → QEMU JSLinux
3. **不踩坑**:**不要**指望 WebGPU 跑 LLM(显存/KV cache/通信三道墙)
4. **数据隐私**:把敏感数据放用户设备侧,云端只做协调

### 4.2 第二招落地(Provider 多路由)

1. **抽象 Adapter**:写一个 `ProviderRouter`,只暴露 OpenAI 兼容 API
2. **路由策略**:简单 Q&A / 代码 / 长文档 / 兜底 / 本地 各一条
3. **fallback**:所有云端 provider 必须配 fallback(本地 Ollama 是免费兜底)
4. **监控**:每条消息记录用了哪个 provider + 多少 token + 多少成本
5. **不种草**:OpenRouter / LangChain / LiteLLM 都行,选哪个看团队偏好

### 4.3 第三招落地(业务专注)

1. **7 维度判断**:对照 7 个判断标准,选行业
2. **砍功能**:把"AI 客服"砍到只覆盖 3-5 个细分场景
3. **重训练**:在单一行业语料上 fine-tune(或换针对性模型)
4. **验证准确率**:单一行业从 90% → 95%+ 才是真正业务专注
5. **重新定价**:从通用 AI 客服客单价 50-300 元/月/人,提升到细分行业 200-800 元/月/人
