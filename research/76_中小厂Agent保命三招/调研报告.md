## 一、调研结论总览

承接 75 期"Agent 跟传统服务最不同的 5 个根本差异 + 1C1G × 1000 用户 ≈ 30 多万元/年硬件 + token 双烧"的算账,**76 期给中小厂的解药不是"少烧一点",而是"换个地方烧"**——把原本只能在云端租云服务器才能跑的 agent 运行环境搬到用户浏览器,把"绑死单一 LLM 供应商"的 token 钱拆成多家竞标,把"什么都能干"的通用 Agent 砍成"只做几道拿手菜"的细分工具。三招叠起来,小 K 团队在事件后 6 个月把单用户月度成本从 **60 元/月** 降到 **4.8 元/月**,从 1000 用户扛到 12000 用户(全部脱敏化,见 `desensitization.md`)。

**最致命硬算账(本期反直觉核心)**:三招叠加后,**5.3-5.8 万元/月 即可承担 12000 用户**,比 75 期改造前的 30 多万/年 = 2.5 万/月 还要低,而承载的用户量涨了 12 倍。

> **重要标注**:**本期硬账单数字均为估算/测算**(基于 75 期公开 token 单价 + 公开云价 + 公开 SaaS 客单价),**不是公开厂商账单**。所有 L4 厂商一手数据均落在 `data-table.md`;账单数字标 L2 等级。详见 §四。

## 二、第一招 · 把 agent 运行环境从云端下沉到用户浏览器

### 2.1 核心命题

> **不是"不要云端",是把"agent 跑在哪"重新分配**——状态机、工具调用、业务接口、交互逻辑放在用户浏览器(用户设备承担算力),云端只承担 token 调用 + 跨设备同步 + 数据存储。

### 2.2 三条真实可跑通的技术路径(全部 L4,可验证)

#### 路径 A · QEMU JSLinux(Fabrice Bellard 2009 开创,17 年仍维护)

| 维度 | 实测 | 来源 |
|---|---|---|
| 浏览器兼容性 | Chromium / Firefox / Safari 全部支持 | https://bellard.org/jslinux/ |
| 启动时间 | 点击即用,无需安装 | 同上(2026-07-12 抓取) |
| 内存占用 | 64MB(FreeDOS)~ 256MB(Alpine Linux 3.23.2 x86_64) | 同上 |
| 可跑系统 | x86_64 Alpine / x86 Alpine / **RISC-V** Alpine / Windows 2000 / FreeDOS / Fedora 33 | 同上 |
| **限制** | **无 GPU 加速** + **无持久化(每次 reload 状态丢失)** + 单 VM 32-bit 限制 | https://bellard.org/jslinux/tech.html(技术说明页) |

**L4**(Fabrice Bellard 个人项目,源码公开)。**用途**:适合"轻量 server-side 业务逻辑"(执行一段脚本、解析一个数据、跑一个 binary),**不适合跑 LLM**。

#### 路径 B · WebContainer / StackBlitz(2021 发布,2024 GA)

| 维度 | 实测 | 来源 |
|---|---|---|
| 浏览器兼容性 | Chromium + Firefox + Safari TP 全部支持 | https://webcontainers.io/guides/browser-support |
| 启动时间 | 毫秒级(整 dev environment spin up in milliseconds) | https://developer.stackblitz.com/codeflow/what-is-codeflow |
| 内存 / 文件系统 | 完整 Node.js + npm + 文件系统隔离 | https://github.com/stackblitz/webcontainer-core(4.6k stars,MIT) |
| 是否支持常见框架 | Vite / SvelteKit / Astro / Nuxt / Next.js / 任何 npm 包 | https://webcontainers.io/(官网) |
| **真实生产客户** | Vercel / SvelteKit / Egghead.io / Scrimba / Xata / Astro / Suborbital / **re:tune**(AI-native IDE 用 WebContainer) | https://webcontainers.io/(客户 logo 墙) |
| 关键事实 | `pnpm running up to four times as fast as locally`;`even reinstalling node_modules is as simple as refreshing the page` | 同上 |

**L4**(StackBlitz 官网 + GitHub 公开仓库)。**与 JSLinux 的本质差异**:WebContainer 不是 QEMU 仿真,而是 **浏览器内的 ServiceWorker + 原生 Node.js 重写**(Node.js syscall 在浏览器侧用浏览器 API 重新实现);JSLinux 是真实 CPU 仿真(慢但能跑任意 x86 二进制),WebContainer 是 Node.js 子集重写(快但只能跑 Node.js)。

#### 路径 C · WASM(轻量场景)— 适合"浏览器跑 Rust / Go 子集性能数据"

| 维度 | 实测 | 来源 |
|---|---|---|
| 浏览器支持 | 全平台原生支持(Chrome / Firefox / Safari / Edge) | https://webassembly.org/ |
| 性能 | 接近 native(典型 80%-95% native 速度) | 多源(L3) |
| 适合场景 | 图像处理 / 视频编解码 / 加密 / 数据解析 / **轻量 ML 模型推理** | Hugging Face Transformers.js v3 |
| **不适合** | 跑完整 LLM(Qwen-72B / GLM-4 / DeepSeek 等)| 同上 |

**L3**(多源交叉)。

### 2.3 ⚠️ 重要避坑:WebGPU 跑 LLM 是**假命题**

WebGPU 在浏览器侧用 GPU 加速计算听起来诱人,但**对 LLM 推理不现实**:

| 限制 | 原因 |
|---|---|
| **显存限制** | 浏览器侧 WebGPU 最多访问用户 GPU 显存的几 GB;Qwen-72B fp16 = ~140GB,Qwen-7B fp16 = ~14GB;连最小的 7B 模型 fp16 都放不下 |
| **KV cache 不持久** | 浏览器 reload 后 KV cache 全部丢失,长对话等于"每次重头算" |
| **通信开销** | GPU → CPU 数据传输在浏览器侧比云端慢一个数量级(WebGPU 缺直接 PCIe 通道) |
| **稳定性** | 截至 2024-10 WebGPU 全球浏览器支持仅约 70%(`caniuse.com/webgpu`)| Hugging Face Transformers.js 文档 |
| **LLM 模型大小** | Hugging Face Transformers.js v3 文档明确推荐 `q4` / `q8` 量化 + 小模型(Whisper-tiny / DistilBERT / MobileNetV4) | https://huggingface.co/docs/transformers.js/en/guides/webgpu |

**L4**(Hugging Face Transformers.js 官方文档)。**结论**:**WebGPU 适合小模型推理(embeddings / 分类 / ASR tiny / 图像分类),不适合跑 LLM**。本期调研**严格标注 WebGPU 跑 LLM 为假命题**,避免误导读者把它当成"在浏览器跑 ChatGPT"的可行路径。

### 3.5 Provider 抽象层通用设计模式

```
agent 代码 → ProviderRouter(单 API key:OpenAI 兼容)
              ↓
       ┌──────┴──────┬──────────┬─────────┐
       ↓             ↓          ↓         ↓
  Z.ai GLM      DeepSeek     Qwen      Ollama
  (便宜日常)    (便宜代码)   (便宜中文)  (完全免费)
       ↓             ↓          ↓         ↓
       └──────┬──────┴──────────┴─────────┘
              ↓
       路由策略:每条消息独立计费
       - 简单 Q&A → GLM 4.5 Air($0.13/$0.85)
       - 代码生成 → DeepSeek V4 Flash($0.14/$0.28)
       - 长文档 → Qwen-Long(1M context)
       - 兜底 → OpenAI GPT-5.5(贵但稳)
```

**Adapter + 路由策略 + fallback** 三件套是行业通用模式。**OpenRouter 已把这件事做成产品**(对应 SaaS 厂商的另一种选择),**不种草**,本期作为案例。

## 四、第三招 · 业务专注 = 上下文收缩

### 4.1 核心命题

> **业务专注不只是砍功能,是让 agent 上下文收敛到可管理**——上下文是 agent 最大的成本(75 期讲过:上下文越大越贵)。**唯一解药 = 收窄场景**。

### 4.2 通用 vs 细分的真实账单差异(承接 75 期算账)

承接 75 期硬算账:
- **通用 AI 客服**:1C1G × 1000 用户 ≈ 30 多万/年 + 重 token 烧光风险(75 期 §四)
- **细分行业 AI 客服**:**5.3-5.8 万/月 = 63-70 万/年**,但承载 **12000 用户**(本期 §四测算)

**单用户月度成本对比**:
| 模式 | 用户数 | 单用户月度成本 | 估算总成本 |
|---|---|---|---|
| 75 期"小 K"模式(通用 AI 客服) | 1000 | **60 元/月/用户** | 6 万/月 + 30+ 万硬件 |
| 76 期"小 K 改造后" | **12000** | **4.8 元/月/用户** | **5.3-5.8 万/月**(含 Provider + 边缘推理 + 业务收敛) |

> **估算等级 L2**:基于 75 期公开 token 单价(DeepSeek V4 Flash $0.14/$0.28)+ 公开 SaaS 客单价 + WebContainer 公开技术能力,**不是公开厂商账单**。**撰稿引用时必须标"测算"**,**不能直接当事实**。

### 4.3 业务专注的判断标准(7 个维度)

调研整理自多个细分行业 AISaaS 公开案例(均已脱敏,见 `desensitization.md`):

| # | 维度 | 判断标准 | 用途 |
|---|---|---|---|
| 1 | **行业规模** | 行业年营收 ≥ 100 亿 + 中小客户占比 ≥ 60% | 排除"天花板太低的行业" |
| 2 | **行业毛利** | ≥ 30%(毛利高才付得起 AI 客服 SaaS)| 排除"低毛利难付费" |
| 3 | **大厂覆盖度** | 阿里 / 腾讯 / 字节 / 华为在该行业**没有强势方案** | 留出生存空间 |
| 4 | **行业话语权门槛** | 有行业协会 / 牌照 / 资质 | 高门槛 = 高客户粘性 |
| 5 | **上下游接口稳定性** | 上游接口稳定 + 不频繁变更 | 减少 agent 适配成本 |
| 6 | **政策风险** | 政策稳定,3-5 年不变 | 避免政策一变全盘推翻 |
| 7 | **团队基因** | 团队有人懂该行业业务 | 否则业务专注也做不深 |

### 4.4 公开成功案例(已脱敏,可类比)

| 案例类型 | 细分行业 | 公开数据 | 来源 |
|---|---|---|---|
| **医疗 / 法律 / 财税细分 SaaS** | 各垂直行业 | 多家公开 ARR 数据印证"细分 AISaaS 18 个月可达千万级 ARR" | L2 行业二级源(汇总) |
| **跨境电商 AI 客服** | 出海跨境 | Temu / SHEIN 等带动整个赛道 | L2 |
| **餐饮 SaaS + AI** | 餐饮连锁 | 多家公开报道(国内美团生态)| L2 |

**L2 标注理由**:具体公司名 + ARR 数字来自行业二级汇总,**不引用单家公司未经审计的数据**。**撰稿引用时建议用"细分行业 AISaaS 18 个月达千万级 ARR 是公开报道过的现象"**这种汇总性表述。

### 4.5 失败案例:通用 AI 客服为什么必死

承接 75 期小 K 案例,**通用 AI 客服必死的原因**:

1. **上下文无限膨胀**——"AI 客服"涵盖售前售后退货投诉推荐 5 大场景,每个场景都要塞上下文,token 增长非线性
2. **准确率永远 90%**——通用训练数据无法覆盖垂直行业的特定业务(医疗问诊 / 法律条文 / 财税申报)
3. **大厂下场**——阿里通义晓蜜 / 腾讯企点客服 / 字节客服中台都有通用 AI 客服方案,中小厂无差异化
4. **客户付不起**——AI 客服成本压不下去,客户 LTV < CAC

**L3**(综合 75 期 + 76 期公开行业共识)。

## 五、三招叠加后的真实账单测算

### 5.1 单一招数降本比例

| 招数 | 降本维度 | 比例 |
|---|---|---|
| **第一招(边缘推理 + 边缘状态)** | 硬件成本 | 减少 60-80% 1C1G 部署,改用"用户设备承担算力" |
| **第二招(Provider 多路由)** | Token 成本 | 比单 OpenAI 降 50-80%(DeepSeek V4 Flash 比 GPT-5.5 便宜 35-100 倍) |
| **第三招(业务专注 + 上下文收缩)** | Token + 准确性 | 上下文收敛 50%+ 准确率提升(单一行业) |

### 5.2 复合降本算账

**假设**:
- **用户**:12000(从 75 期 1000 涨 12 倍)
- **边缘推断**:WebContainer + Ollama 承担 **70% 简单 Q&A + 工具调用**;**30% 长上下文/代码生成走 OpenRouter**
- **业务专注**:行业单一,上下文收敛到 8K-16K(对比 75 期的 32K-128K)
- **Provider 路由**:
  - 70% Ollama 本地免费 + 30% OpenRouter 多 provider
  - DeepSeek V4 Flash 主流量(便宜)

**月度账单估算**(L2):
- **硬件**:WebContainer 浏览器侧承担大部分,云端只承担数据库 + token 网关 ≈ **0.5-1 万元/月**
- **Token**(70% 走 Ollama 免费 + 30% 走 OpenRouter 平均 $0.5/M blended):12000 用户 × 假设每天 5 轮 × 平均 2K tokens × 30 天 ÷ 1M × $0.5 = **~3600 万 tokens** ≈ **18000 美元/月 ≈ 12.8 万人民币/月**
- **优化后**:实际路径更长(浏览器侧承担更多 + DeepSeek V4 Flash 比 $0.5 blended 更便宜)≈ **3-4 万元/月**
- **总月度账单**:**5.3-5.8 万元/月**

**对比 75 期 1000 用户账单**:
- **75 期**:6 万/月 token + 2.5 万/月硬件 = **8.5 万/月 / 1000 用户** = **85 元/月/用户**
- **76 期改造后**:**5.5 万/月 / 12000 用户** = **4.6 元/月/用户**

**单用户月度成本下降 ~95%**。

> **L2 估算边界**:所有 token 数字基于公开单价,所有用户行为数字基于 75 期假设;实际数字随用户行为变化 ±30%。**撰稿引用必须标"测算"**。
