上一篇我们聊了心智——**大流量防护是水龙头和下水道要分开**。

你说"懂了，四件事：水龙头开多大、下水道能接多少、水表读得准不准、溢洪道放不放"。

老板说"那去装吧"。

**然后你就懵了。**

"CDN 怎么选？网关限流怎么写？Sentinel 三模式怎么配？熔断三态机是什么？排队页怎么接 Kafka？"

心智是图，工程是钉子。这一篇我们**卷袖子干活**——把"水龙头和下水道分开"拆成**三道防线 + 五道工程纪律**：

- **第一道防线**：CDN（流量蓄水池）
- **第二道防线**：网关限流（水龙头闸门）
- **第三道防线**：应用层限流/降级/熔断/排队/预热（下水道保护）

五道工程纪律：①CDN 选型 → ②网关限流 → ③Sentinel 流量控制 → ④降级熔断 → ⑤排队预热。

下面我们一步步拆给你看。

## 一、纪律一：CDN 静态挡水——把水从远处接走

大流量防护的第一道防线是 **CDN**——把水（流量）从远处接走，不让它直接打你的源站。

为啥要先做 CDN？因为**水从远处来，成本最低**——CDN 节点在用户家门口，不用请求都回你的源站。

**第一步：CDN 不是"加速工具"，是"流量蓄水池"。**

这是 CDN 工程师和老板最容易踩的认知误区。

老板以为 CDN 是"加速器"——让用户访问更快。

工程师以为 CDN 是"静态资源缓存"——只挡 JS、CSS、图片。

**CDN 的真正身份是"流量蓄水池"——它用 3200+ 节点、180 Tbps 全网带宽，替你的源站接住 80% 的流量。**

来源锚点：阿里云 CDN 官方文档明确——全球节点数 **3200+**（中国内地 2300+，海外及港澳台 900+），覆盖中国内地 31 个省级区域 + 70 多个国家和地区，**全网带宽输出能力 180 Tbps**。

这是啥概念？**180 Tbps = 180 万亿比特每秒**——你买一辈子服务器也堆不出这个带宽。

**第二步：CDN 三层选型——基础 CDN / DCDN / ESA，按预算分层。**

CDN 不是只有一种，它有三种"档位"：

| 产品 | 边缘计算能力 | 安全防护能力 | 预算档位 |
|---|---|---|---|
| **基础 CDN** | EdgeScript 边缘脚本、图片处理 | Referer 防盗链、URL 鉴权、IP 黑白名单 | < 1 万/月 |
| **DCDN（全站加速）** | EdgeRoutine 边缘程序、A/B Test、预热 | 基础 WAF、DDoS 防御、基础 Bot 防御 | 1-10 万/月 |
| **ESA（边缘安全加速）** | 边缘函数（JS 部署）、边缘存储（K-V）、边缘容器 | 原生 WAF、**Tbps 级 DDoS 防护**、AI 防护、Bot SDK | > 10 万/月 |

**FDE 跟客户谈 CDN 的标准答法：**

**"先上 DCDN，纯静态 CDN 对 SaaS API 帮助有限。DCDN 兼顾动态 API 加速 + 静态缓存兜底，边投入边出效果。"**

预算 < 1 万/月：基础 CDN，先把静态资源挡下来。
预算 1-10 万/月：**DCDN**，兼顾动态 API 加速。
预算 > 10 万/月：ESA，加 Tbps 级 DDoS 防护。

**第三步：静态资源全走 CDN，动态 API 走网关。**

为啥要这么分？因为**静态资源是死的，动态 API 是活的**。

死的——同一份 JS、一张图片、一段视频，全网用户拿到的都一样——适合 CDN 缓存。
活的——同一个 API 在不同时间、不同用户、不同状态下返回不同结果——不适合 CDN 缓存。

CDN 只挡**死的**那部分，活的必须让你的网关和应用层去扛。

**金句 1："CDN 不是加速工具，是流量蓄水池。"**

**金句 2："基础 CDN → DCDN → ESA，按预算分层。"**

**金句 3："静态资源全走 CDN，动态 API 走网关——死的让 CDN 扛，活的让系统扛。"**

CDN 这一道闸做完，源站的压力瞬间能降 80%——剩下的 20% 流量走网关限流继续挡。

## 二、纪律二：网关限流——水龙头开多大

CDN 挡住了静态资源的 80%，剩下 20% 的动态 API 请求会到你的**网关**。

网关是大流量防护的第二道防线——**它就是那个水龙头，决定让多少水进你的下水道。**

**第一步：选对限流算法——Redis 4 种算法，对号入座。**

网关限流的第一步是选限流算法。

来源锚点：Redis 官方术语表《Rate Limiting》明确给出 4 种主流算法的官方定义（2025-12-08 更新）：

| 算法 | 工作原理 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|---|
| **Fixed Window（固定窗口）** | 固定时间窗计数 | 简单 | **窗口边界突刺**（59 秒和 60 秒边界可瞬时 2x 流量） | 简单场景 |
| **Sliding Window（滑动窗口）** | 滚动时间窗精确计数 | 灵活准确 | 实现稍复杂 | **短信验证码、API 限速（首选）** |
| **Token Bucket（令牌桶）** | 桶按固定速率填 token，请求消费 token | **允许突发** | 平均速率受填 token 速率限制 | **API 网关（首选）** |
| **Leaky Bucket（漏桶）** | 请求入桶匀速漏出 | **严格削峰**、输出平滑 | **不支持突发** | **DB 保护（首选）** |

**第二步：API 网关选令牌桶，DB 选漏桶——一允许突发，一严格平滑。**

为啥这么分？

**令牌桶允许突发**——桶满的时候，瞬时可以打满一波。这对 API 网关很合适，因为 API 网关后面是应用，应用可以扛瞬间的流量高峰。

**漏桶严格平滑**——不管你怎么冲，输出都是匀速的。这对 DB 很合适，因为 DB 接不住瞬时高峰，匀速打才不会被秒杀打挂。

来源锚点：Wikipedia 令牌桶原文明确——"A conforming flow can thus contain traffic with an average rate up to the rate at which tokens are added to the bucket, and have a burstiness determined by the depth of the bucket."（令牌桶允许流量突发，突发大小由桶深度决定。）

来源锚点：Wikipedia 漏桶原文明确——"The leaky bucket as a queue is therefore applicable only to traffic shaping, and does not, in general, allow the output packet stream to be bursty."（漏桶作为队列时只用于流量整形，输出流严格无抖动。）

**金句 4："令牌桶 = 允许突发 + 平均速率可控——API 网关首选。"**

**金句 5："漏桶 = 严格匀速 + 输出平滑——DB 首选。"**

**第三步：Nginx ngx_http_limit_req_module 内部用漏桶。**

为啥 Nginx 这个细节重要？

因为 Nginx 是国内最常用的反向代理/网关之一，它的限流模块用的是漏桶算法。

来源锚点：Wikipedia 漏桶原文明确——"The leaky bucket algorithm is used in Nginx's `ngx_http_limit_req_module` module for limiting the number of concurrent requests originating from a single IP address."

这意味着——如果你用 Nginx 做网关限流，**单 IP 维度它默认就是漏桶**（不允许突发）。

**金句 6："Nginx ngx_http_limit_req_module 内部用漏桶——单 IP 维度的限流默认不允许突发。"**

**第四步：Redis 官方推荐的"INCR + EXPIRE"基础模式。**

来源锚点：Redis Glossary 明确——基础的限流模式就是 `MULTI` + `INCR` + `EXPIRE` 两行命令：

```bash
MULTI
  INCR [user-api-key]:[current minute number]
  EXPIRE [user-api-key]:[current minute number] 59
EXEC
```

**MULTI 事务保证 INCR + EXPIRE 原子性**——Redis 官方明确背书。

但要注意——**Fixed Window 有窗口边界突刺问题**：14:00:59 攻击者打 1 条，14:01:00 又打 1 条——2 秒内 2 条，但频控只看到 1 条/分钟。

要解决突刺，**必须用 ZSET 滑动窗口**——精确款，生产推荐。

## 三、纪律三：Sentinel 流量控制——三模式焊死下水道入口

网关限流是水龙头的闸门，但**闸门开了之后，进入下水道的流量还得再过一道关**——这就是应用层的限流。

应用层限流在国内最主流的工具是 **Sentinel**（阿里出品，10 年双 11 大促验证）。

**第一步：Sentinel 是 Java 微服务限流首选——Hystrix 已停止维护。**

来源锚点：Sentinel Wiki「常用限流降级组件对比」明确：

| 组件 | 隔离策略 | 实时统计 | 流量整形 | 控制台 | 多语言 | 开源状态 |
|---|---|---|---|---|---|---|
| **Sentinel（阿里）** | 信号量 | LeapArray | **预热 + 匀速排队** | **开箱即用** | Java/C++ | **活跃** |
| **Hystrix（Netflix）** | 线程池/信号量 | RxJava 滑动窗口 | 不支持 | 简单 | Java | **已停止维护** |
| **Resilience4j** | 信号量 | Ring Bit Buffer | 简单 RateLimiter | 不提供 | Java | 较活跃 |

**FDE 选限流组件的标准答法：**

- 选 **Sentinel**：Java 微服务、需要细粒度限流 + 流量整形 + 控制台——**新项目首选**。
- 选 **Hystrix**：**Netflix 已停止维护，新项目不要选**——这条金句你必须刻进脑子里。
- 选 **Resilience4j**：轻量、函数式编程、Spring Boot 集成好——次选。

**金句 7："Hystrix 已停止维护，新项目不要选。"**

**第二步：Sentinel 流量控制三模式——直接拒绝 / 冷启动 / 匀速排队。**

来源锚点：Sentinel 流量控制官方文档给出 QPS 限流时的 3 种 `controlBehavior` 模式。

**模式 1：直接拒绝（默认）。**

**QPS 超过阈值，新请求立即拒绝，抛 FlowException**。

适用场景：**对系统处理能力确切已知**的情况下（如压测已确定系统水位）。

工程取舍：**最简单、最可控**；代价是用户体验是"瞬间 429"，没有任何缓冲。

**秒杀场景首选直接拒绝——**100 万请求进来，前 1000 个放过，剩下 999000 个立即 429。

**模式 2：冷启动 / Warm Up。**

**流量激增时，通过的流量缓慢增加，在一定时间内逐渐增加到阈值上限**。

典型场景：**系统长期低水位运行，流量突然增加**（早上 9 点上班打车 App 开门红）。

Sentinel 文档原文："通过'冷启动'，让通过的流量缓慢增加，在一定时间内逐渐增加到阈值上限，给冷系统一个预热的时间，避免冷系统被压垮。"

工程取舍：**保护冷系统**（JIT 刚预热、连接池刚建好）；代价是前几秒用户体验"被限速"。

**早高峰开门红首选冷启动——**9 点之前系统是冷的，9 点流量瞬间涌入，慢慢放热。

**模式 3：匀速排队（Rate Limiter / 漏桶）。**

**严格控制请求通过的间隔时间，堆积的请求排队，超过超时时长直接拒绝**。

Sentinel 文档原文："这种方式的实现是基于 Leaky Bucket 算法"——**确认这是漏桶算法的工程实现**。

典型场景：**间隔性突发流量**（如消息队列写后端处理）。

工程取舍：**削峰填谷**；代价是用户感受到"排队等待"——可配合前端排队页。

**秒杀削峰首选匀速排队——**请求按固定间隔通过，DB 接得住的下游才不会被秒杀打挂。

**金句 8："直接拒绝 = 秒杀首选（瞬间 429）。"**

**金句 9："冷启动 = 早高峰开门红首选（慢慢放流量）。"**

**金句 10："匀速排队 = 秒杀削峰首选（基于漏桶）。"**

**三种模式不互斥，可叠加。** 典型配置：大促开闸时用**冷启动**（保护刚启动的容器）→ 稳定后用**匀速排队**（平滑下游压力）→ 触底限流时用**直接拒绝**（避免线程堆积）。

## 四、纪律四：降级 + 熔断——溢洪道 + 自动跳闸

水龙头、下水道、水表都做好了，还有最后一道防线——**降级 + 熔断**。

为啥要这道？因为**前三道闸都是预防，但预防不可能 100%**——总有漏网之鱼。

漏网之鱼来了怎么办？**主动放水（降级）+ 自动跳闸（熔断）**。

**第一步：降级——撑不住时主动放水。**

降级是**溢洪道**——水库水位快到警戒线了，主动开闸放水，保住主坝。

工程实现——**关闭非核心功能，保核心功能**：

活动当天 CPU 飙到 90% 时：
- 关闭"个性化推荐"——非核心，让路给登录。
- 关闭"评论列表"——非核心，让路给下单。
- 关闭"统计埋点上报"——非核心，让路给支付。

核心功能继续跑，非核心功能先停——这叫主动放水。

**降级有三种粒度**——自动降级（超时/失败率触发）、手动降级（运营/值班一键开关）、降级粒度（方法 → 接口 → 服务 → 全局）。

**金句 11："降级是溢洪道——撑不住时主动放水，保主坝。"**

**第二步：熔断——自动跳闸的三态机。**

熔断是**家里的电闸**——某个电器短路了，电闸自动跳开，避免烧掉整条电路。

来源锚点：Resilience4j 官方文档明确给出**有限状态机**：

> "The CircuitBreaker is implemented via a finite state machine with three normal states: **CLOSED, OPEN and HALF_OPEN**."

熔断三态：

| 状态 | 含义 | 行为 |
|---|---|---|
| **CLOSED（关闭）** | 正常状态 | 统计调用结果 |
| **OPEN（断开）** | 短路状态 | 立即拒绝所有调用，抛 CallNotPermittedException |
| **HALF_OPEN（半开）** | 探测恢复 | 放行 N 个请求探测（默认 10 个） |

状态转换——

- **CLOSED → OPEN**：失败率 ≥ 阈值（默认 50%）。
- **OPEN → HALF_OPEN**：等待 `waitDurationInOpenState`（默认 60 秒）。
- **HALF_OPEN → CLOSED**：探测请求成功。
- **HALF_OPEN → OPEN**：探测请求失败。

**熔断不是按时间熔断，是按失败率熔断**——这是它和"超时熔断"最大的区别。

**第三步：熔断阈值怎么配——支付类 10%，内容类 50%。**

来源锚点：Resilience4j 文档明确——`failureRateThreshold` 默认值是 **50%**（失败率阈值，达到即熔断）；`minimumNumberOfCalls` 默认值是 **100**（计算失败率前最少需要的调用数）。

但默认参数是双刃剑——**保护冷启动系统不被 1 次失败就熔断**，但**也意味着前 100 次失败的代价是真金白银的故障请求**。

工程取舍——**FDE 调参时，这个值要按业务流量级数重设**。

| 业务敏感度 | failureRateThreshold | 场景 |
|---|---|---|
| **极高（支付）** | **10%** | 支付失败 10% 立刻熔断，避免资损 |
| **高（订单）** | 20%-30% | 订单失败 30% 熔断 |
| **中（内容）** | 50% | 内容查询失败 50% 熔断 |
| **低（推荐）** | 70% | 推荐失败率高也能忍 |

**金句 12："熔断三态：CLOSED → OPEN → HALF_OPEN——Resilience4j 官方原话。"**

**金句 13："熔断阈值：支付类 10%，内容类 50%——按业务敏感度分级。"**

**第四步：Sentinel 熔断三种策略——慢调用比例/异常比例/异常数。**

来源锚点：Sentinel 熔断降级文档（1.8.0+）列出 3 种熔断策略：

| 策略 | 触发条件 | 适用场景 |
|---|---|---|
| **慢调用比例（SLOW_REQUEST_RATIO）** | 慢调用比例 > 阈值 | 下游 RT 飙升但未失败（慢 SQL/慢 RPC） |
| **异常比例（ERROR_RATIO）** | 异常比例 > 阈值 | 下游报错率飙升 |
| **异常数（ERROR_COUNT）** | 异常数 > 阈值 | 极小流量场景，百分比不适用 |

Sentinel 熔断状态机与 Resilience4j 一致：**CLOSED → OPEN → HALF-OPEN**。

## 五、纪律五：排队 + 缓存预热——把流量摊开

前四道纪律做好了，还有最后一道——**排队 + 缓存预热**。

为啥要这道？因为**水龙头再小，下水道容量再大，也挡不住瞬时 100 倍峰值**。

这时候需要**把流量摊开**——让 100 倍峰值变成 1 倍稳态。

**第一步：排队页——秒杀前把用户挡在门外。**

Cloudflare Waiting Room 是排队页的典型产品。

来源锚点：Cloudflare Waiting Room 官方文档（Business 及以上套餐可用）：

- **核心机制**：当请求超过设定阈值，把超出用户引导到"虚拟候车室"页面，避免源站被打爆。
- **关键能力**：显示实时预计等待时间（用户侧体验可控）；跟踪入流/出流，动态放行；通过 Cookie 记忆用户排队位置（刷新不丢队）。
- **依赖前置**：必须先配 CDN + proxied DNS——所以 Waiting Room 不是独立的限流产品，而是 **CDN 之上的"用户层排队"**。

**排队页不是 bug，排队页是用户体验可控。**

白屏 = 用户看到 500，啥都不知道。
排队页 = 用户看到"前面还有 12345 人，预计等待 8 分钟"，心里有数。

**当系统承载不了流量，与其让用户看到 500 白屏，不如让用户看到排队页。**

**金句 14："排队页不是 bug——是用户体验可控，比白屏强一万倍。"**

**第二步：缓存预热三场景——应用启动预热 + 大促预热 + 热 key 探测。**

缓存预热是**大促前的"提前把锅烧热"**——活动开始时不让系统冷启动。

三种预热场景——

| 场景 | 预热时机 | 典型做法 |
|---|---|---|
| **应用冷启动预热** | 容器启动/滚动发布时 | 启动时执行 `CacheLoader`，把热点数据加载到 Redis |
| **大促预热** | 大促开始前 N 分钟 | 定时任务把活动商品/库存/配置预加载到 Redis |
| **热 key 探测** | 运行时持续 | 通过 Redis MONITOR / redis-cli --hotkeys 识别 |

为啥要做热 key 探测？因为**有些 key 是活动当天临时爆热的**——你不可能提前知道哪个 key 会爆。

边缘缓存预热也得做——**大促前 1 小时，运营/运维手动调用 CDN 预热接口，把首页、活动页、商品详情页推送到边缘节点**——避免开闸瞬间"全国用户回源"。

**金句 15："缓存预热三场景：应用启动预热 + 大促预热 + 热 key 探测。"**

**第三步：Kafka 异步队列——秒杀请求丢队列，后台慢慢处理。**

秒杀场景是异步队列的典型战场。

为啥秒杀必须异步？因为**你的 DB 不可能在 1 秒内处理 100 万订单**——但你可以让 100 万用户都拿到"我排队了"的反馈，1 分钟后陆续处理完。

工程实现——**用户提交秒杀订单 → 入 Kafka → 立即返回"排队中，预计 30 秒出结果" → 后端 Worker 异步消费 → 实际下单 → 推送结果给用户**。

**好处**：即使后端处理慢 10 倍，前端也不会 502。
**代价**：用户看不到"立刻成功"，需要轮询/WebSocket 推送结果。

**金句 16："Kafka 异步队列——秒杀请求丢队列，后台慢慢处理。"**

**金句 17："排队页 + 缓存预热 + Kafka 异步——三件套把瞬时 100 倍峰值摊成 1 倍稳态。"**

## 六、把五道纪律串成"100 万用户白屏"的事故复盘

上一节我们讲过那个"100 万用户涌入就白屏"的故事。

这一节我们把那个故事**重新走一遍**——但这次用**三道防线 + 五道工程纪律**复盘。

**出事前：CDN 预热 + 限流配置 + 降级开关 = 三道闸。**

- **CDN 预热**：活动前 1 小时，把首页/活动页/详情页推送到边缘节点——100 万用户进来，80% 静态资源在边缘节点直接挡回。
- **限流配置**：网关令牌桶（允许突发）+ 应用层 Sentinel 匀速排队（漏桶平滑）+ 单 IP 漏桶限流（Nginx limit_req）。
- **降级开关**：配置中心一键关闭非核心功能——CPU 飙到 90% 时，个性化推荐/评论列表/统计埋点一键停，给核心让路。

**出事中：实时监控 + 熔断跳闸 = 保住核心服务。**

- **实时监控**：水表读得准——大屏秒级刷新全站 QPS/响应时间/错误率/排队人数。
- **熔断跳闸**：支付接口失败率 > 10% 立刻熔断——返回"支付排队中"，而不是 500。
- **降级生效**：CPU 90% 时运营一键降级——个性化推荐关掉，订单支付保住。

**出事后：5 分钟定位 + 30 分钟恢复 + 1 小时对账。**

- **5 分钟定位**：七步事故链模板套一遍——DNS/CDN/网关/应用/DB/缓存/全站，5 分钟定位是哪个环节出事。
- **30 分钟恢复**：根据定位结果一键恢复——降级关掉、限流放宽、缓存重建、DB 连接释放。
- **1 小时对账**：漏单/重复单/超卖单全部捞出来——支付通道 2 小时对账窗口，必须 1 小时完成。

**金句 18："出事前：CDN 预热 + 限流配置 + 降级开关 = 三道闸。"**

**金句 19："出事中：实时监控 + 熔断跳闸 = 保住核心服务。"**

**金句 20："出事后：5 分钟定位 + 30 分钟恢复 + 1 小时对账。"**

这就是完整的"大流量白屏"五道工程纪律——**CDN 选型 + 网关限流 + Sentinel 流量控制 + 降级熔断 + 排队预热**——**五道防线一条都不能少**。

**少一道，活动当天半夜的电话就会响。**
