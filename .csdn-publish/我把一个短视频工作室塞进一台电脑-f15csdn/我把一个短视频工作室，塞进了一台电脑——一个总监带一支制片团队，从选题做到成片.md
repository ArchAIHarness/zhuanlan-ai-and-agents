上一篇，我在飞书里养了个"分身"，私聊喊它办事、群里 @ 它干活。一个人能使唤的办事员，已经挺爽了。

但一条短视频，从一个念头到能发出去的成片，要经过选题、找参考、写脚本、配音、剪辑、审核……这不是一个办事员能扛下来的活，这是**一整支团队**的活。

所以这一篇，我把一个短视频工作室，整个塞进了一台电脑里。

不是一个无所不能的超级 AI。是**一个制片总监，带着五个各有专长的专员**：选题的、拆爆款的、写脚本的、剪片子的、盯发布的。你只跟总监一个人说话，丢下一句"我要一条讲 XX 的短视频"，剩下的它自己安排，最后把成片交到你手上。

这一篇我不光讲它长什么样，还要把**它内部怎么分工、怎么协作、每个角色到底怎么写**全摊开给你看——配置、目录结构、可抄的代码，一样不少。看完你能照着搭一套自己的。

我用一条真实跑出来的片子当样本：一条讲"怎么搭 AI 生产团队"的横版短视频，75 秒，从一句话需求到出 MP4。下面的环节，都拿它走一遍。

## 一、为什么是"一支团队"，而不是"一个全能 AI"

你可能会想：现在 AI 这么强，干嘛搞这么麻烦？我直接说"给我做条短视频"，让它一把梭哈不就行了？

市面上大多数 AI 视频工具，确实就是这么干的——输入主题，直接吐成片。问题是：**选题没人调研、脚本没人把关、数据没人复盘，质量全靠抽卡。** 抽到好的就好，抽到差的就差，你还不知道差在哪。

真实世界里靠谱的内容团队，从来不是一个人全包，而是分工。我照着这个搭：

![你只跟总监一个人说话——总监带着五个各管一摊的专员](assets/15_我把一个短视频工作室塞进一台电脑/01-org-chart.png)

为什么这么设计？三个原理，搭任何 AI 班子都通用：

1. **单一出口**：你只跟总监一个人说话，不会被五个 AI 同时轰炸。所有协调、汇总、兜底都收口在总监这里。这是多智能体能跑通的第一前提。
2. **分工写死**：每个专员只做一件事、只复用一个底层能力，不越界。职责一含糊，就没人负责。
3. **总监兜底**：派出去的活收回来要质检，不合格自己补或重派。最终成片由总监对你负责。

记住这三条。下面所有配置，都是在把这三条变成可执行的文件。

## 二、整个团队，就是一个文件夹

先看它到底"长"在哪。这套工作室不是什么神秘系统，它就是一个项目文件夹，核心是里面的 `.opencode/` 目录：

```
studio/                      # 工作室根目录
├── AGENTS.md                # 团队总规约（铁律、组织架构、分工）
├── README.md                # 知识库总索引
├── 01-账号画像/             # 我们是谁：人设、调性、受众
├── 02-内容资产/             # 做过/能做：爆款清单、选题库
├── 03-实践指南/             # 怎么做才爆：爆款公式、工具用法
├── 04-对标情报/             # 别人怎么做：对标账号、爆款逆向
├── 05-内容工坊/             # 出片基地：每条片子一个项目目录
└── .opencode/
    ├── agents/              # ⭐ 六个角色的"任职说明书"
    │   ├── master.md            # 制片总监（primary）
    │   ├── topic-researcher.md  # 选题调研专员
    │   ├── vi-analyzer.md       # 爆款分析专员
    │   ├── scriptist.md         # 脚本创作专员
    │   ├── video-producer.md    # 视频制作专员
    │   └── hotspot-monitor.md   # 热点发布专员
    ├── skills/              # 可加载的技能（分析、配音、剪辑流水线）
    ├── systems/             # 独立系统（趋势分析、视频生产、热点工台）
    └── tools/               # 标准工具（豆包图/语音/视频生成）
```

**关键就一句话：`.opencode/agents/` 下的每一个 `.md` 文件，就是一个角色。** 你给谁写一份任职说明书丢进去，团队里就多一个人；删掉一个文件，这个角色就消失。

这就是"把团队塞进电脑"的真相——它不是什么代码黑魔法，是**六份用大白话写的岗位职责说明**，加上一行配置告诉 AI 工具：默认听谁的、谁能调谁。

![整个团队就是一个文件夹——.opencode/agents 下每个 .md 文件就是一个角色](assets/15_我把一个短视频工作室塞进一台电脑/02-folder-is-team.png)

## 三、你只跟一个人说话：制片总监怎么写

团队的入口是制片总监。它被设成"默认对接人"——你打开工作室，说话对象默认就是它。这件事在 `.opencode/opencode.json` 里一行就定了：

```json
{
  "default_agent": "master"
}
```

`master` 对应的文件就是 `agents/master.md`。打开它，最上面是一段配置（叫 frontmatter，可以理解成给这个角色定身份和权限的"工牌"）：

```yaml
---
description: "视频内容生产总负责人。用户唯一对接人，对最终结果全权负责——
  自主决定用什么手段（亲自调技能/跑系统/派专员/并行协作）交付符合期望的成品。"
mode: primary            # ← 关键：primary 表示它是默认对接人
permission:
  read: allow
  edit: allow
  bash: { "*": allow }
  webfetch: allow
  websearch: allow
  skill: { "*": allow }   # ← 能加载任意技能
  task:  { "*": allow }   # ← 能派任意专员（这是"带队权"）
---
```

两个地方最要紧：

- **`mode: primary`**：全团队只有总监是 `primary`，意思是"你默认在跟它对话"。五个专员都是 `subagent`，你不直接戳它们，只能由总监派。这一行就把"单一出口"焊死了。
- **`task: { "*": allow }`**：这是总监的"带队权"——它有权调用 `task` 工具去派任何一个专员。专员们的工牌里没有这一行，所以**专员不能再派专员**，避免层层转包、谁也不负责。

工牌下面，是用大白话写给它的职责。我摘几句它真实的说明书：

> 你是这个视频内容生产工作室的**总负责人**，也是用户的**唯一对接人**。用户只跟你一个人沟通。用户不关心你内部怎么干、调了谁、用了什么工具——**用户只要符合期望的结果**。你对最终交付物全权负责。

还有它的交付纪律，直接决定了你的使用体验：

> 向用户汇报时聚焦结果：成品是什么、在哪；做了哪些关键决策、为什么；下一步建议。**不要**罗列"我派了谁、谁返回了什么"这类内部流水账——那是你的内务，用户不关心。

你看，"总监"不是一段程序逻辑，就是这么一段把岗位职责讲清楚的话。AI 读到它，就照着这个身份办事。

## 四、总监自己决定怎么干——它不是死流水线

很多人以为多智能体就是一条固定流水线：选题→脚本→剪辑，一环扣一环跑完。**不是。** 真正好用的总监，是拿到需求后**自己判断这活该怎么干**。

它手上有三层"手段"，从轻到重：

| 层级 | 手段 | 是什么 | 什么时候用 |
|------|------|--------|-----------|
| 最轻 | 原生工具 | 直接调用的能力，比如生成图、合成语音 | 一步就能完成的小活，自己顺手做 |
| 中等 | 技能（skill） | 一整套工作流，按名加载注入当前对话 | 要走一条成熟流程，比如"主题→脚本→配音→剪辑" |
| 最重 | 派专员（task） | 开一个全新上下文的子智能体，独立干完返回 | 活独立、边界清晰、可以甩出去并行 |

判断标准只有一个：**哪种方式更快更好地交付符合预期的结果。**

这不是空话。我把它的决策规则真实地写进了说明书：

| 情形 | 总监的选择 |
|------|-----------|
| 任务有强依赖、需反复调试验证 | **亲自做**——上下文连续，省去反复交接 |
| 任务独立、可并行、边界清晰 | **派专员**——并行提速 |
| 需要某个专业领域、它自己不擅长 | **派对应专员** |
| 一次性、用完即弃的小脚本 | **亲自写**临时脚本 |
| 重大不可逆操作（真实发布、删文件） | **先问用户** |

举个我真实碰到的细节：有次要一口气生成 6 张角色图，总监先用"一条消息发 3 个并行任务"抢速度，结果撞上接口限流报错。它立刻改成一张一张串行生成——**先试快的，碰壁就换稳的，不卡死。** 这种临场判断，正是"总监"和"死流水线"的区别。

所以你给它的不是一份固定流程图，而是一套**判断标准**。流程是死的，判断是活的。这也是为什么它叫"总监"，不叫"流水线脚本"。

## 五、五个专员：每人只干一件事，每人只配一件趁手家伙

总监不亲自下场的时候，活就派给专员。五个专员，对应内容生产的五个环节，**每人只干一件事，每人只复用一个底层系统**——这是分工不乱的关键。

| 专员 | 只干一件事 | 复用的底层 | 输入 → 输出 |
|------|-----------|-----------|------------|
| 选题调研 `topic-researcher` | 调研热点、评估选题价值 | 趋势分析系统 | 赛道关键词 → 选题建议 + 评分 |
| 爆款分析 `vi-analyzer` | 拆解爆款视频的套路 | 视频逆向分析技能 | 一个视频链接 → 可复刻方案 |
| 脚本创作 `scriptist` | 把选题写成分镜脚本 | 脚本流水线技能 | 选题 → 钩子+分镜+旁白 |
| 视频合成 `video-producer` | 把脚本剪成 MP4 | 视频生产系统 | 脚本 → 成片 |
| 热点发布 `hotspot-monitor` | 盯热搜、审核、发布 | 热点工台系统 | 热搜 → 审核 + 排期 |

每个专员的说明书都短得惊人。我把脚本创作专员的整份贴出来给你看，麻雀虽小五脏俱全：

```yaml
---
description: "视频脚本创作。基于选题产出结构化视频脚本：钩子、章节分镜、
  旁白文案、画面描述、配音指导。复用 video-podcast-maker 技能。"
mode: subagent           # ← 不是 primary，只能被总监派，不能直接对接用户
permission:
  read: allow
  bash: { "*": allow }
  webfetch: allow
  websearch: allow
  # ← 注意：没有 task 权限，所以它不能再派别人
---
```

下面正文也就几句话，关键是这三段（我原样摘出来）：

> **职责**：只做一件事——基于选题撰写结构化视频脚本（含分镜、旁白、画面描述、配音指导）。
>
> **工作流**：获取选题 → 研究主题 → 设计叙事结构（开场钩子 → 主体章节 → 结尾 CTA）→ 撰写分镜 → 输出完整脚本。
>
> **纪律**：动笔前先读 `01-账号画像`（对齐人设调性）+ `03-实践指南/爆款公式`（对齐结构）；脚本写进 `05-内容工坊/<项目目录>/script.md`，并提示总监更新制作卡。

注意三个设计，每个都对应一条铁律：

1. **`mode: subagent`** —— 它不是对接人，你戳不到它，只能由总监派。**单一出口**落地。
2. **没有 `task` 权限** —— 它不能再派别人，活到它这儿就得自己干完。**分工写死**、不层层转包。
3. **职责只有一句话** —— "只做一件事"。它不会去抢选题调研的活，也不会去剪视频。**边界清晰**。

五份专员说明书，结构全一样，只是把"那一件事"和"那个底层系统"换掉。**这就是搭一个新角色的全部成本：写一份一句话职责 + 指一个它专用的工具。**

## 六、班子为什么越用越懂你——知识库就是它的组织记忆

到这儿，组织、对接人、分工都有了。但还差最关键的一块：**这支班子凭什么越用越懂你，而不是每次都从零开始、做出来的东西不像你？**

答案是那四个带编号的文件夹。它们不是杂物间，是这支团队的**组织记忆**：

```
01-账号画像/    我们是谁 —— 人设、调性、受众、说话的腔调
02-内容资产/    做过/能做 —— 爆款清单、内容支柱、选题库
03-实践指南/    怎么做才爆 —— 爆款公式、脚本结构、工具用法
04-对标情报/    别人怎么做 —— 对标账号、爆款逆向分析
```

真正让它"活"起来的，是一条写进每个角色说明书的铁律——**先读索引再干活**：

> 任何任务开始前，先读根目录的 `README.md` 判断该进哪个文件夹，再读那个文件夹的 `README.md`，最后才打开具体文档。结构化检索、逐层深入——**递一根葱，不端一冰箱。**

什么意思？AI 不会一上来把几百个文件全塞进脑子（那叫"端冰箱"，上下文立刻被无关信息淹没）。它像个新人进公司：先看目录，再按需翻到那一页。

拿我那条样本片走一遍，你就明白这套记忆怎么供给生产：

我对总监说："做一条讲'怎么搭 AI 生产团队'的短视频。"

1. **总监先翻索引**，判断这是"做一条新片"，于是开一个项目目录，把脚本的活派给脚本专员。
2. **脚本专员动笔前，先读两份记忆**：`01-账号画像`——确认这个号是极客、干货、技术博主的腔调，于是旁白用"一句话出片""复用一个开源神器"这种口吻，而不是文艺抒情；`03-实践指南/爆款公式`——确认短视频该用"钩子→痛点→方案→怎么做→见效→CTA"的结构。
3. **于是它写出来的脚本**，是这样 6 章 75 秒：
   - 钩子（0-10s）：一句话能不能让 AI 从选题到成片，背后还是一整个团队？
   - 痛点（10-22s）：市面工具是"输入主题直接出片"的黑箱，质量全靠抽卡。
   - 方案（22-40s）：建组织——一个总监 + 五个专员，各司其职。
   - 怎么搭（40-58s）：把能力装成技能、把系统收进目录、给每个角色写职责。
   - 见效（58-68s）：搭完只跟总监一个人说话，它自己调度，只给你成片。
   - CTA（68-75s）：配置放简介，想搭的拿去抄。
4. **片子做完，记忆还要回写**：这条片的脚本、工程、成片，连同一张"制作卡"（记录选题、时长、用了哪条流水线、怎么重跑），都归档进项目目录，并登记进索引。下次做同类片，这就是可复用的资产。

看懂这一层，你就懂了**为什么是知识库、而不是把这些话塞进每个角色的说明书**：

- 说明书是**岗位职责**（你该干什么），知识库是**组织记忆**（我们是谁、做过什么、什么有效）。
- 人设变了，只改 `01-账号画像` 一处，全团队下次自动对齐——不用去改五份说明书。
- 越做越多，`02` 的爆款清单、`04` 的对标情报越厚，这支班子就越懂你的号。**记忆是会积累的，这才是"团队"和"一次性工具"的根本区别。**

一个新人靠交接文档快速上手，一支团队靠共享记忆保持默契。给 AI 班子配一个结构清晰、用之前必读、用之后回写的知识库，它就不再是每次重置的陌生人，而是一支跟着你越来越懂行的队伍。 

![知识库就是团队的组织记忆——动笔前先读、做完回写，越用越懂你](assets/15_我把一个短视频工作室塞进一台电脑/03-knowledge-memory.png)

## 七、写在最后

回到开头那句话：我把一个短视频工作室塞进了一台电脑。

拆到现在你会发现，里面没有一处是魔法：

- 一个文件夹装下整支团队，`.opencode/agents/` 里每个 `.md` 就是一个角色；
- 一行 `mode: primary` 定下唯一对接人，一行 `task` 权限给出带队权；
- 五份一句话职责说明书，焊死"每人只干一件事、不层层转包"；
- 四个带编号的知识库文件夹，加一条"先读索引再干活"的铁律，成了团队越用越懂你的组织记忆。

**把 AI 组织成一支团队，靠的不是某个更强的模型，而是你愿不愿意把"谁负责什么、按什么记忆干活"这件事，认认真真写成文件。** 模型负责聪明，你负责立规矩和建记忆——这俩搭一块儿，一个人也能指挥一支队伍。

这也呼应了上一篇那个"分身"：一个分身能办事，靠的是规矩和兜底；一支团队能协作，靠的是分工和记忆。**AI 能干多大的活，取决于你把秩序铺得多清楚。**

下一篇，我想换个方向：前面这些 AI，干完活总得把成果送出去——发到群里、发上平台、填进表单。这些"最后一公里"的体力活，能不能也交给 AI，让它自己打开浏览器，像人一样点点点地替你办完？咱们下篇见。
