## 二、33 上篇：业务心智篇

### 2.1 标题方案候选

候选（开写前可校准）：
- A.《老师甩给我一句"我要做 Java 培训",我把它喂给 AI,三天后交出了一份结构化课程大纲》
- B.《老师想 AI 帮忙做课,AI 却问了一堆"离谱"的问题——我才明白真懂业务的是人不是 AI》
- C.《把老师模糊的"我想做课"变成 AI 能跑的标准件——一个真实业务的边界梳理全过程》

**主编倾向**：B 更带"反差钩子"，符合专栏一贯风格（"AI 越能干越要明白什么由人做"）。

### 2.2 核心命题

**功能不是工程师主观设计,只是把老师原生需求提纯、规范化。**

AI 不是自己懂业务,是人 + AI 一起把"模糊念头"剥到"可用边界"。AI 在边界内做追问、做抽象、做结构化,人做决策、做价值判断、做最终拍板。FDE 在 AI 时代的核心价值不是写代码,是**让 AI 知道问什么**。

### 2.3 真实场景引子

某企业培训负责人说："我们要给社招新人做 Java 培训,但具体怎么排课我自己也说不清。"

——这就是典型的碎片化、口语化原始需求。没有学员画像、没有知识点结构、没有题目规划、没有课时分配、没有难度曲线。FDE 拿到这种需求的第一步**不是进 IDE**,不是画架构图、不是建工程、不是写测试,而是去和老师 + AI 做三方对话,完成"业务剥离"。

### 2.4 第一轮 AI 对话（贯穿全文的"主骨"）

**核心方法论锚点**：ADD 教学设计模型（Analyze 学员/目标/任务 + Design 结构/内容/练习/反馈/难度）。这是 1990 年代公认的教学设计框架(Allen Forde / Rossett / Gary Kearsley 等研究者贡献),非私有资产,公开引用。

**真实多轮对话演示（候选对话素材，待撰稿员细化）**：

- **第一轮（场景确认）**："这门课是企业培训?中学?大学?其他?"——AI 锁定"企业培训",追问策略自动切到企业培训模板(而不是中学/大学模板)
- **第二轮（学员画像）**："社招新人是已经会 Java SE?会 Spring?工作 1 年内还是 1-3 年?每天能投入多少时间?周末是否集中?"——AI 锁定学员起点 + 时间预算
- **第三轮（终点目标）**："40 小时课程结束,新人应该能独立完成一个 CRUD 模块?能跟师傅带项目?能看懂 Spring 源码?能排查线上问题?"——AI 锁定课程终点
- **第四轮（结构逻辑）**："按 Java 知识逻辑(JDBC→Spring→SpringBoot→微服务)?按任务逻辑(从零做一个完整订单系统)?按问题逻辑(新人最常见踩坑集)?"——AI 锁定结构策略
- **第五轮（章节 / 知识点 / 课时分配）**：根据前 4 轮 AI 提炼出的"骨架"自动展开为章/节/知识点三层结构,每个知识点配布鲁姆动词目标 + 前置依赖 + 预估课时 + 难度
- **第六轮（题目配置）**：对所有 contentType=exam 的知识点,追问"出什么题型?几道?什么难度?每题几分?"

**3-4 轮上限纪律**（专栏必须强调）：AI 追问不能无限追问,3-4 轮必汇总确认,否则用户体验崩。这是"长期挂载的约束",不是 AI 自己悟出来的。

### 3.4 DDD 工程范式

**三个限界上下文(候选拆分,撰稿员可校准)**：

- **course 限界上下文(核心域)**：
  - 聚合根：Course(课程) / Chapter(章) / Section(节) / KnowledgePoint(知识点)
  - 通用语言：course / chapter / section / knowledgePoint / prerequisite / bloomLevel
  - 责任：课程结构管理 + 知识点编排 + 课时分配
  - 仓储接口 vs 实现跨模块分离(沿用 27/28 篇架构原则)

- **question 限界上下文(支撑域)**：
  - 聚合根：ExamConfig(题目配置) / Question(题目)
  - 通用语言：examConfig / questionType / difficulty / score
  - 责任：题目类型管理 + 难度分级 + 分值分配

- **media 限界上下文(支撑域)**：
  - 聚合根：Material(材料) / Reference(引用关系)
  - 通用语言：material / referenceId / ragContext
  - 责任：上传材料存储 + RAG 检索 + 引用关系维护

**上下文映射**：
- course ↔ question：Customer-Supplier (Feign) — 课程定稿触发题目配置
- course ↔ media：Published Language (Kafka) — 课程定稿同步材料引用
- 共享内核：布鲁姆动词表 + 知识点 ID 规范 + prerequisites 字段

**三个限界上下文 vs 一个平台的清晰分层**(沿用 27/28 篇风格)**：**
- course 限界 = 业务域(domain 零 Spring / 业务规则)
- question 限界 = 业务域(domain 零 Spring / 业务规则)
- media 限界 = 业务域(domain 零 Spring / 业务规则)

### 3.5 SDD 双轨

**Story 端(交付阶段,FDE 工程师唯一交互入口)**：
- 业务对话脚本 —— 老师需求 → ADD 多轮追问 → JSON 输出 → 审阅修改 → Exam 配置 → 定稿
- 全程贯穿"老师只聊业务,AI 自动按 Spec 端规范落地"

**Spec 端(训练阶段,人全权沉淀)**：
- **ADD 追问规则长期挂载**：A 学员/A 目标/A 任务 + D 结构/D 内容/D 练习/D 反馈/D 难度,每轮追问必须覆盖其中至少一个维度,3-4 轮上限硬约束
- **JSON Schema 长期挂载**：courseName / description / totalHours / scenario / targetAudience / bloomTaxonomy / structure / knowledgePoints / examConfig 必填字段;enum 范围;条件必填(exam 知识点必须有 examConfig,非 exam 禁止)
- **题型规则长期挂载**：questionTypes ∈ {choice, fill, coding, essay, judge};难度三档 beginner/intermediate/advanced;分值正整数
- **难度规则长期挂载**：beginner(intermediate(advanced;艾宾浩斯遗忘曲线调整建议
- **材料上传规则长期挂载**：格式 md/mp4/wmv/ppt/xls/doc;Pre-Signed URL 直传 OSS;引用关系落 Kafka

### 3.6 TDD 自动化审计

**三层测试护栏**：

- **追问覆盖度审计**：
  - 每一轮追问必须覆盖 A 学员 / A 目标 / A 任务 / D 结构 / D 内容 / D 练习 / D 反馈 / D 难度至少一个
  - 累计 3 轮追踪,不能漏 5 个以上维度
  - 汇总确认前必校验

- **JSON 合法性审计**(对应 framework JaCoCo 90% + P0 红线原则,本方案自定义校验规则)：
  - 必填字段校验：courseName / description / totalHours / scenario / targetAudience
  - enum 范围校验：scenario ∈ {enterprise, k12, university, other}; contentType ∈ {text, exam, lab}
  - 数值范围校验：totalHours > 0; estimatedHours > 0; score > 0
  - 引用完整性校验：prerequisites 中的 id 必须存在 structure 里(非孤儿引用)

- **修改一致性审计**：
  - 用户修改 JSON 后,关联的 prerequisites / examConfig / 学习目标必须重新校验
  - 删除章节时,该章节下所有 prerequisites 引用必须清除或重新指派

**自动化审计的工程价值**：把"AI 写糊"挡在 PR 合并前,焊死的闸,跟 framework JaCoCo 90% 是同一种秩序。

### 3.7 工程化取舍(4 选 1 主线决策)

| 决策项 | 选项 A | 选项 B | 推荐 | 理由 |
|------|------|------|------|------|
| 对话状态 | Redis(TTL ≤ 24h) | DB(长期存档) | **A** | 多轮追问是单次会话行为,过期不必要 |
| 材料上传 | Pre-Signed URL 直传 OSS | 经过服务代理 | **A** | 减轻服务端 IO 压力 |
| JSON 解析 | Schema 库(jsonschema) | 自研正则 | **A** | Schema 库更通用,生态成熟 |
| 题目生成 | 同步 Worker | 异步消息队列 | **B** | 大题量场景下异步更稳 |

撰稿时可展开 2-3 个最具说服力的取舍,其它一笔带过。

- **一门真实课程实例** —— "Java 社招新人 40 小时速成"完整 JSON 骨架(复用 33 的产物)

### 3.9 关键概念(工程师核心收获)

- DDD 战略设计(三限界上下文划分)+ 战术设计(聚合根 / 实体 / 值对象)
- 上下文映射(客户-供应商 / 发布-订阅 / 共享内核 / 开放主机服务 / 防腐层)
- SDD 双端架构(Story / Spec)
- TDD 三层测试护栏(追问覆盖度 / JSON 合法性 / 修改一致性)
- 工程化取舍(对话状态 / 材料上传 / JSON 解析 / 题目生成)
- 布鲁姆动词表(共享内核)
- 价值复利(一次创建,多次复用)

### 3.10 核心金句

- "业务模型有了,工程模型是把业务边界焊死、选择放开。"
- "追问规则不是 AI 学出来的,是 Spec 端长期挂载的——AI 忘掉一次,下次还得听你的。"
- "AI 不懂 ADD,但 AI 能把 ADD 跑得比人更稳。"
- "JSON 字段错一个,整个课程结构就崩——TDD 是焊死的闸,不是装饰。"
- "工程化的真正价值不是复杂度,是复杂度可被消化——一次创建,多次复用。"
- "未来真正会用 AI 做产品的人,不是会写架构的人,是会立规范的人。"

### 3.11 核心比喻(沿用 27/28 厨房三件套 + 33 标准菜谱)

**"标准菜谱厨房"系列**：

### 3.12 预计结构(7 节)

## 八、风险与边界

### 8.1 内容真实性风险

**风险**：33+ 强调"真实业务",但本业务素材来自 FF 单元,跨单元引用可能涉及"暴露私有方法论"。
**缓解**：专栏文章里只引用公开方法论(ADD 模型 + JSON Schema),不复述 FF 私有 skill 内部细节,业务素材只用于"案例演示",不写实现源码。

### 8.2 真实对话素材风险

**风险**：33 上篇需要"复现真实人-AI 多轮对话",但本次创作时未采集到 FF 单元的真实对话素材。
**缓解**：撰稿时用"基于公开 ADD 方法论的对话流还原",明确标注"对话为方法论复现,非真实业务对话截屏",保证专栏读者不被误导。后续若补充真实对话,可迭代更新。

### 8.3 双篇一组工作量风险

**风险**：双篇一组是 2 倍工作量,可能拖慢节奏。
**缓解**：并行调度调研员 + 撰稿员 + 插画员(并非真并行,而是有序派单),严格按 §七执行清单推进,主编全程验收。

### 8.4 平台发布用户介入风险

**风险**：按用户决策"发布我自己安排",发布动作需要用户手动执行;但草稿准备阶段 AI 必须严格按 question 门禁向用户确认。
**缓解**：每个平台发布前单独 question 一次,**不合并征询**,严格守 §7.0 门禁不可绕过。
