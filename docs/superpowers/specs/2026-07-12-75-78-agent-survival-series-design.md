# 75-78 期「AI 时代中小厂 Agent 求生」设计 Spec

> **文档定位**:75-78 期(4 篇 · 2 组双篇)的完整设计文档,供 调研 → 撰稿 → 配图 → 审核 → 发布 全链路 Agent 按章执行。
>
> **核心命题**:AI 时代,大厂做 Agent 不是为了赚钱,而中小厂学大厂 Agent 范式 = 死字不知道怎么写。本文给中小厂一条"技术层 + 战略层"双层突围路径。
>
> **贯穿比喻**:**自助餐 vs 点菜制**(整套 4 篇主线贯穿)
> - 大厂 = 豪华自助餐(赔本赚吆喝 / 流量入口 / 云 + 模型双绑定)
> - 中小厂 = 招牌小炒(只做几道、做到极致、皮实抗造)

## 二、4 篇完整规划

### 第一组(75-76)· 中小厂 Agent 隐性硬性成本(技术层)

#### 75 期(上·业务心智篇)

**文件名**:`75_中小厂做AI客服一晚上烧了八千——Agent跟传统服务最不同的不是贵是这五个根本差异.md`

**副标题**:Agent 跟传统服务最不同的不是贵,是这五个根本差异

**核心命题**:中小厂做 AI 客服一晚上烧了八千——Agent 跟传统服务最不同的不是贵,是这五个根本差异。**Agent 是有状态 / 秒级响应 / 上下文是 LLM 的不是 app 的 / 存储是 agent 自身(叠加态)/ 上下文越大开销越大**——五件事一起,决定了"每多一个用户,每个用户也都更贵"。

**贯穿比喻**:自助餐(传统 SaaS / 分布式服务)vs 点菜制(Agent / AI 服务)—— 但更深一层是**自动售货机 vs 私厨**:
- 传统服务 = 自动售货机(无状态 / ms 响应 / 上下文无关 / 拿了就走)
- Agent = 私厨(有状态 / 秒响应 / 上下文决定一切 / 给你定制)

**开场场景**:某 AI 客服创业公司(脱敏)"小 K"上线第 47 天,凌晨 4 点被账单预警叫醒——某用户用脚本刷了 67 个账号 × 单次 2000 token × 8000 次 = 单晚 8.2 万元。CTO 早上醒来一看:月度预算 5 万,一晚烧超 60%。事后复盘会上 CTO 说了一句话,让所有人沉默——"**我以为我做的是传统 SaaS,其实我做的是私厨,每一道菜都是定制,每一份食材都要真付钱**。"

**连贯叙事伏笔(关键)**:小 K 这个公司 4 篇 75-78 都出现——形成"首次失败 → 改造活下来 → 二次失败 → 真正破局"的失败/救赎连贯叙事。75 这篇中后期**必须**预埋一处暗示:小 K 学了三招(76 讲的三招)改造后活了一时,但又开始学大厂,这就是 77 的开场。允许的反向引用句式如"**这场复盘会上小 K 决定了三件事,先做再做做看。**你看 6 个月后,他会做了,但只活了一时"——为 77 留出"二度失败"叙事接口。**不剧透**具体大厂名字、不写"他要学字节/阿里/腾讯",只暗示"他活了一时但又开始学某一套看似光鲜的新玩法,下一回再说"。篇幅控制:§六 三个反常识金句之上,**只用一句留口**。

**核心论点(Agent vs 传统服务的 5 个根本差异)**:

| 维度 | 传统服务 | Agent 应用 | 中小厂的代价 |
|---|---|---|---|
| **状态** | 无状态 / 冷热持久化就够 | **有叠加态(状态机 + 历史 + 工具结果 + LLM 上下文)** | 每一会话都要持续烧 |
| **响应时间** | ms 级 | **s 级** | 同一个 GPU,服务的用户数少 1-2 个数量级 |
| **上下文归属** | app 收口 | **LLM 收口(app 管不了)** | 上下文会涨,涨完一次烧完一次 |
| **存储** | DB / Redis / 对象存储单一栈 | **agent 自身(状态机 + 上下文 + 工具结果 + 历史记忆)** | 多栈叠加,每一栈都烧钱 |
| **规模关系** | 用户越多边际成本越摊薄 | **上下文越大开销越大(模型上下文越大,agent 开销越大)** | 用 128K 模型 = 每对话烧的钱是 8K 的 16-30 倍 |

**7 节结构**:
1. §一 反差开场——AI 客服一晚上 8.2 万元 + CTO 那句"我做的是私厨不是 SaaS"(反面案例)
2. §二 Agent ≠ 传统服务(核心命题:为什么传统服务经验不能套到 Agent)
3. §三 把"AI 太贵"剥成 5 个根本差异(状态 / 响应时间 / 上下文归属 / 存储 / 规模关系)
4. §四 一个核心算账——**1C1G × 1000 用户 = 30 多万硬件/年 + 不算 token**(中小厂没硬件没资本,**用户不愿意为硬件付费**,活都活不下去,别想规模化)
5. §五 上下文陷阱——为什么"上下文越大越好"是个假命题(8K / 32K / 128K 的真实账单对比)
6. §六 三个反常识金句
7. §七 金句 + "我们下篇见。下篇卷袖子干活讲三招——**agent 运行环境下沉浏览器 / token 与 agent 解耦 / 业务专注**,把私厨做出自助餐的便宜。"

#### 76 期(下·工程落地篇)

**文件名**:`76_中小厂Agent的保命三招——agent运行环境下沉浏览器加token与agent解耦加业务专注别让点菜制把你的厨子烧光.md`

**副标题**:中小厂 Agent 保命三招

**核心命题**:中小厂 Agent 的保命三招——**agent 运行环境下沉浏览器(在用户设备端跑 agent,而非云端服务器)** + **token 从 agent 解耦(agent 触 LLM 时不限一家 provider)** + 业务专注。**别让点菜制把你的厨子烧光**。每一招都解决一种隐性硬性成本,三招叠起来才让中小厂 Agent 真正算得过账。**真正可跑通的技术路径**是 **QEMU JSLinux / WebContainer / WASM(轻量场景专用)**——浏览器内跑真实 Linux VM 或 Node.js 容器。**WebGPU 不要想(假命题),浏览器内跑大 LLM 模型也不现实**——本篇布道的核心是 **"agent 运行环境下沉 + token 与 agent 解耦"** 这个双轴骨架,不是某种单一技术。

**贯穿比喻**:点菜制下的"便宜的点法"——硬件下沉(自己带调料上桌) + Provider 解耦(不绑一家供应商) + 业务专注(只做几道拿手菜)

**开场场景**:承接 75 期"小 K"事件后 6 个月,小 K 团队做的三个动作——把 agent 整套运行环境(状态机 + 工具调用 + 业务接口)从"云端服务器"搬到"用户浏览器"(走 QEMU JSLinux / WebContainer 路线)、token 这一层彻底从 agent 拆开走 OpenRouter / 阿里云百炼 / 智谱 GLM / MiniMax 多家 provider 路由、砍掉通用功能只做"AI 客服 for 跨境电商"细分。结果:月度账单从不可预测(5万→25万)变成可预测(月度 5.3-5.8 万),用户从 1000 涨到 12000,单用户成本从 60 元降到 4.8 元。同时另一组隐性的好处——agent 在用户浏览器里跑,用户的敏感数据根本不上云,合规风险也降了一档(预告 78 期再展开)。

**核心论点**:中小厂 Agent 不是做大厂同样的事,是用**分布式 + 解耦 + 收缩**避开每用户线性成本:
- **第一招 agent 运行环境下沉浏览器**:把 agent 的"运行环境"(状态机 + 工具调用 + 业务接口 + 交互逻辑)从云端服务器搬到用户浏览器——QEMU JSLinux / WebContainer / WASM(轻量场景专用)是真正可跑通的技术路径;**WebGPU 不要想(假命题),浏览器内跑大 LLM 模型也不现实**;**布道的核心是"agent 运行环境与 LLM 解耦下沉"这个骨架**,不是某一种技术
- **第二招 token 与 agent 解耦(Provider 多路由)**:agent 出 browser 调 token 时,不绑任何一家 provider——OpenRouter / 阿里云百炼 / 智谱 GLM / MiniMax 多家供应,通过统一的 Provider 抽象层路由(便宜选便宜 / 失败兜底 / 按业务分级);**这一招是真正解决 75 期"上下文越大开销越大"的根本招**——模型上下文大小可控 + 多家比价 = 单会话 token 成本可控
- **第三招 业务专注 = 上下文收缩**:75 期提的"Agent 是叠加态 / 上下文越大开销越大",唯一的解药就是**业务专注**——砍掉通用功能,只做细分场景的"几道拿手菜"。**业务专注不只是"砍功能",是"让 agent 的上下文收敛到可管理"**——用户变多但每用户上下文长度可控 = 边际才可能摊薄。**75 期的"叠加态难题"在 76 期的"业务专注"这里得到收口**

**7 节结构**:
1. §一 反差开场——6 个月后小 K 的真实账单:从不可预测到可预测
2. §二 第一招 边缘推理(用户设备端跑小模型):浏览器端跑小模型,把"实时 GPU"搬到用户设备
3. §三 第二招 Provider 解耦:多模型路由 + fallback + 价格策略
4. §四 第三招 业务专注:砍掉通用,只做"AI 客服 for 跨境电商"细分
5. §五 三招叠起来:从单用户 60 元到单用户 4.8 元的真实账本
6. §六 反常识金句 + 三招的失败模式(每招都会踩什么坑)
7. §七 金句收束 + 不写"下篇见"(双数实践篇)

**金句池(精选 10+)**:
- "中小厂 Agent 不是要把点菜制做出自助餐,是把点菜制做出外卖的便宜。"
- "边缘推理把 GPU 从'公司的机房'搬到'用户的浏览器',边际成本被切到几乎为零。"
- "Provider 解耦不是技术问题,是不被任何一家云厂绑架的战略问题。"
- "业务专注不是砍功能,是把每道菜做到用户非要找你不可。"

**开场场景**:承接 77 期小 K 关张,另一家中型 SaaS 创业公司"深 K"反向操作——不学大厂,选了一个细分行业(出海跨境电商的 AI 客服 for 中小卖家),用大厂模型(阿里云百炼 + 智谱 GLM + OpenRouter)但严格限制每用户 token 用量,把业务做到能写进合同能验收能收钱。18 个月后 ARR 突破 1200 万,单用户 LTV 8 万元,从未公开融资,纯利润 35%。

**核心论点**:中小厂 Agent 的两招战略骨架:
- **垂直深耕**:选一个细分行业走产业链上下游——医疗 / 法律 / 教育 / 金融 / 出海 / 农业 / 物流 / 制造 / 等等。**不要做"通用 AI 客服",要做"AI 客服 for 出海跨境电商中小卖家"**。每加一层垂直,大厂就越难跟。
- **业务皮实**:业务要够扎实,能写进合同、能验收、能收钱。**不是 PPT 项目,不是 demo 数据,是真能落地的业务**。客户能复用、能复购、能推荐。

1. §一 反差开场——深 K 18 个月 ARR 1200 万,纯利润 35%,从不融资(正向案例)
2. §二 第一招 垂直深耕——为什么"通用 AI 客服"必死,"AI 客服 for X"才有戏
3. §三 怎么选垂直——七个判断标准:行业规模 / 行业毛利 / 大厂覆盖度 / 行业话语权门槛 / 上下游接口稳定性 / 政策风险 / 团队基因
4. §四 第二招 业务皮实——能写进合同、能验收、能收钱的三件硬指标
5. §五 业务皮实的反义词——demo 项目 / 烧钱换流量 / 模式创新没现金流
