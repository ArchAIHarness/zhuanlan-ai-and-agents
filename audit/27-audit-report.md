## 三、亮点清单（撰稿员做得好的地方）

1. **核心比喻"作曲家与乐团"贯穿始终且讲到位**：在 §一第 11 行就抛出"给它一份乐谱，它能演奏交响乐"，§三到 §五逐件乐器对应（DDD 划旋律线、TDD 钉音、SDD 是乐谱），§六扩展为"音符是硬规范，表情记号是软秩序"，§七第 318-320 行收束成完整的金句。比喻全部来自日常经验（谱架、谱子、指挥），且每处都跟落地解释，没停留在"就像…"就结束——完全符合 AGENTS.md §5.4 "比喻是推理工具，不是装饰"。

2. **用户补充的"软秩序"洞察单独成节讲透**：§六全节（70 行）都在讲"硬规范 + 留白设计 = 完整的秩序"，从"边界焊死 / 选择放开"金句（§六第 241 行）到 gateway 6 个 SPI（§六第 249-256 行）到 `@ConditionalOnMissingBean` 替换示例（§六第 258-272 行）到 framework vs gateway 对照（§六第 278-281 行），最后第 285 行金句"秩序不是锁死 AI 的手，是给 AI 画好赛道——边界内随便跑，边界外一步都越不了"完整闭环。这是初版规划漏掉、用户两轮校准补的关键洞察，撰稿员没浮于表面。

3. **真实工程现场细节拿捏精准**：§四第 148 行主动暴露"framework 仓的 `pom.xml` 实际配置 JaCoCo 门槛是 LINE 70% / BRANCH 30%，但各层 `AGENTS.md` 文档写的是 90%"——这是 AGENTS.md 调研员在 §2.3 / §12.4 提示的"工程现实妥协"论据，撰稿员用"有 70% 在那里卡着，AI 就不敢全跳过测试"反向利用，把"文档与代码打架"从瑕疵变成 AI 时代的秩序哲学论据。§五第 211 行指出"这条 `tenant-center` 守得很死，`user-center` 还没全跟上"——也是调研报告 §5.4 提示的"同体系不同仓执行力度有差异"真实证据。工程师的诚实感拉满。

## 五、建议优化清单（非必须）

### 优化 1：§七首段"让我们回到…"略嫌课堂收束语气
- **位置**：§七第 297 行
- **现状**：`让我们回到开头那个反直觉的判断——**AI 编码能力越强，秩序越贵**。`
- **建议**：AGENTS.md §5.5 明确禁用"今天我们来讲 / 大家好 / 让我们"等课堂演讲式开头。本节虽不是开头而是收尾，但"让我们回到"仍是同源课堂语气。建议改为 `回到开头那个反直觉的判断——**AI 编码能力越强，秩序越贵**。` 或 `还记得开头那个反直觉的判断吗——**AI 编码能力越强，秩序越贵**。`，节奏更朋友聊天感。
- **影响**：仅风格颗粒度，不影响核心论点。

### 优化 2：§四 JaCoCo 70%/30% 锚点补一行号
- **位置**：§四第 148 行
- **现状**：`我看到一组真实现场的数字——framework 仓的 `pom.xml` 实际配置 JaCoCo 门槛是 LINE 70% / BRANCH 30%`
- **建议**：调研报告 §2.3 给出的代码位置是 `framework/pom.xml` LINE 139-141（JACOCO `<limits>` 块）。建议加一句路径锚点：`——来自 `framework/pom.xml` 第 139-141 行的 JaCoCo 配置块`。这样工程师读者可直接跳验证。
- **影响**：增强可信度与可验证性。

### 优化 3：§四 code-review.md 锚点补行号
- **位置**：§四第 174 行
- **现状**：`这套机制有个生动的实证——`framework/code-review.md` 里记着 AI 自动审查的结果：3 个 P2 建议、0 个 P0、0 个 P1。`
- **建议**：调研报告 §2.4 已锁定 `framework/code-review.md` 报告元数据（393 行、3 个 P2、0 P0/P1）。建议加 `§十一 问题汇总` 或 `第 393 行的报告元数据` 作为精确指针，方便工程师验证。
- **影响**：增强可信度。

### 优化 4：§三 ASCII 依赖图改表格或更清晰
- **位置**：§三第 119-122 行
- **现状**：用 ASCII 字符画的六模块依赖方向图，行内排版较密：
  ```
  common（零框架） → domain（领域） → application（用例编排） 
     → infrastructure（仓储 / Feign / Kafka） → interfaces（Controller / 切面） → bootstrap（启动）
  ```
- **建议**：可改为表格或编号列表，让依赖方向更易读。如：
  ```
  依赖方向（只允许单向）：
  1. bootstrap → interfaces
  2. interfaces → application
  3. application → domain
  4. domain → common
  5. infrastructure → domain
  6. common 零框架依赖（不允许 import 任何 Spring/JPA/Feign）
  ```

- **影响**：可读性提升，配图清单的"图 3"位置（图 1 之后）正好覆盖这一节，可考虑把这张依赖图合并到配图里，正文改用简洁引导。

### 优化 5：§六第 287 行"关键密钥"用词稍显生造

- **现状**：`这个洞察你得记牢——它是这一篇的"关键密钥"，也是 28 篇要在 gateway + auth + user + tenant 真实链路上验证的核心命题。`
- **建议**："关键密钥"是生造比喻，与全篇"乐谱/音符/作曲"主线不太一致。建议改为 `这个洞察你得记牢——它是这一篇的"关键命题"，也是 28 篇要在 gateway + auth + user + tenant 真实链路上验证的核心断言。` 或 `这个洞察你得记牢——它是这一篇的"关键乐句"，也是 28 篇……`（保留乐谱主线）。

### 优化 6：§六"gateway 的 6 个 SPI" 与 §三 DDD 边界的衔接可再点一下

- **影响**：章节衔接。

### 优化 7：§七"DDD/TDD/SDD"收束金句可前置到 §六
- **位置**：§七第 318-320 行 vs §六
- **现状**：完整金句 `DDD 划旋律线，TDD 把每个音都钉在调上，SDD 是乐谱本身——音符是硬规范，表情记号是软秩序` 在 §七才完整呈现；§六只讲到"framework 告诉你不能做什么，gateway 告诉你可以怎么做"。
