## 1. 一句话结论

**能扎实支撑。** 35-36 的"缝合式交付"立意在公开事实层面站得住：四块积木里有三块（编辑器、AI 后端、容器/编排）都是成熟第三方开源工具，其"可被扩展、靠注册不改内核"的机制**全部有官方一手文档背书**（VS Code 扩展 API、OpenCode plugin/agents/skills/tools、Vite dev server proxy）。

**最大的料在方向 A + B**：36 下篇两块新料——"编辑器交互插件如何嵌 Web UI"和"能力插件如何注册进 AI 后端"——都能用官方文档讲清"这层只是薄胶水/靠注册扩展、不碰内核"，且**恰好呼应"缝合"本质**：VS Code 官方自己就把"在编辑器里显示 Web 内容"设计成 Webview（受控 iframe）和 Integrated/Simple Browser（打开外部 URL）两种现成机制；OpenCode 官方就把 agents/skills/tools/plugin 设计成"放进约定目录/写进 config 就自动加载"的注册式扩展。**换句话说：这套产品的"可缝合性"不是 FDE 发明的，是这些开源工具的作者原生设计好的——FDE 的价值正是识别并利用了这种设计。** 建议把这点作为 35-36 的"暗线"。

**方向 C（方法论印证）**相对弱但够用：Palantir 官方博客对 FDE 角色的定义（"配置现成平台、不重造轮子、组合架构"）几乎是本立意的完美一手佐证；但"glue code / composition over creation"的经典权威条目（Wikipedia、Martin Fowler）本次抓取失败，属证据缺口，见第 7 节。

## 2. 方向 A 成果：VS Code / code-server 扩展嵌 Web UI 的通用公开机制

> 服务 36 下篇第三块新料「编辑器交互插件（脸/交互）」。以下均为 VS Code 官方公开机制，可写。

### A1. 扩展的激活机制（activation events）——"按需唤醒，不常驻"
- **事实**：VS Code 扩展在 `package.json` 的 `activationEvents` 里声明"何时被激活"，如 `onCommand`（执行某命令时）、`onStartupFinished`（启动后延迟激活）、`onView`、`onUri` 等。扩展必须导出 `activate()` 函数，仅在触发时被调用一次；官方明确建议"只在用户真正需要时才激活"，`*`（启动即激活）是下策。
- **来源**：VS Code 官方《Activation Events》 https://code.visualstudio.com/api/references/activation-events
- **成熟度**：L4
- **对撰稿的意义**：印证"交互插件是一层薄胶水"——平时不占资源，用户需要 AI 对话时才被唤醒、拉起后端与 UI。

### A2. 编辑器里显示 Web 内容的两种官方通用方式（36 下篇最关键的结构解释）

**方式一：Webview API（编辑器内的受控 iframe）**
- **事实**：Webview 是"扩展完全控制的、VS Code 内的一个 iframe"，可渲染几乎任意 HTML，用消息传递（`postMessage` / `onDidReceiveMessage`）与扩展通信。官方明确定位："应谨慎使用、仅当原生 API 不足时才用；Webview 资源开销大、运行在独立上下文"。默认禁用脚本，需显式 `enableScripts: true`，并强调 CSP、只走 https、清洗输入等安全实践。
- **来源**：VS Code 官方《Webview API》 https://code.visualstudio.com/api/extension-guides/webview
- **成熟度**：L4

**方式二：Integrated / Simple Browser（在编辑器里打开一个外部 URL）**
- **事实**：VS Code 内置"集成浏览器"，可在编辑器标签页里直接打开 `http://localhost` 等 URL 预览 Web 应用；点击 VS Code 里任意 `localhost` 链接即可唤起，也可用命令 `Browser: Open Integrated Browser`。官方文档《Integrated browser》定位就是"预览本地跑起来的 Web 应用、测试认证流程"。（同类的 `Simple Browser: Show` 命令是既有的等价机制。）
- **来源**：VS Code 官方《Integrated browser》 https://code.visualstudio.com/docs/debugtest/integrated-browser
- **成熟度**：L4
- **两者区别（判断，L3）**：Webview = 扩展自己"画"UI、深度可控但重、要处理安全；Simple/Integrated Browser = 扩展只把"某个本地端口上跑着的现成 UI"打开给用户看，几乎零定制、最省事。**本案例的编辑器交互插件走后一类思路（拉起本地 UI 服务后用浏览器打开），这正是"缝合而非重造"的教科书示范：不自己重画界面，只把现成 UI 端口接进来。**

### A3. 扩展启动子进程 / 本地服务——通用做法
- **事实（判断层）**：VS Code 扩展运行在 Node.js 的"扩展宿主（Extension Host）"进程里，具备 Node 能力，因此可用标准 Node 方式（如 `child_process`）拉起子进程/本地服务。官方《Extension Host》《Extension Anatomy》说明扩展是 Node 模块、有 `activate()` 生命周期入口。官方没有"教你怎么 spawn 一个 serve 进程"的专门页面，这属于 Node 通用能力 + 扩展宿主环境的组合，故标 L2。
- **来源**：VS Code 官方《Extension Host》 https://code.visualstudio.com/api/advanced-topics/extension-host ；《Extension Anatomy》 https://code.visualstudio.com/api/get-started/extension-anatomy
- **成熟度**：L2
- **对撰稿的意义**：印证"插件激活后在本地启动 AI 后端 serve 进程 + 启动前端 UI dev server"这一动作机制上完全成立——扩展就是个 Node 程序，拉进程是它本来就能做的事。

### A4. 前端开发服务器的 API 代理（dev server proxy）——是什么、为什么
- **事实**：前端开发服务器（如 Vite）提供 `server.proxy` 配置，把匹配某前缀的请求转发到另一个后端地址（如 `/api` → `http://localhost:xxxx`），支持路径重写、`changeOrigin`，也支持 WebSocket（`ws: true`）代理。
- **来源**：Vite 官方《Server Options · server.proxy》 https://vite.dev/config/server-options
- **成熟度**：L4
- **为什么要代理（判断，L3）**：前端 UI 和 AI 后端是两个独立进程/端口，浏览器同源策略会挡跨端口请求；dev server proxy 让前端"以为"API 就在自己这个端口下，请求被透明转发到本地 AI 后端。这就是"前端通过代理把 API 请求转发到本地 AI 后端"的机制解释。
- **脱敏提示**：Vite、端口号是"背景理解"用。按主编定调（36 只讲结构与作用、不讲技术栈名词），**正文不出现"Vite / 5173 端口"这类具体名词**，只说"前端界面通过一层代理，把请求转给本地的 AI 后端"。

## 3. 方向 B 成果：OpenCode 插件 / agents / skills / tools 的公开注册机制

> 服务 36 下篇第四块新料「场景化能力插件集合（手艺/业务能力）」。均为 OpenCode 官方公开机制，可写。专栏 13 篇已核实"本地插件放 `.opencode/plugins/` 自动加载、插件=导出函数返回 hooks"，本节补充 agents/skills/tools 这一层。

### B1. plugin 字段与插件加载——"放进目录 or 写进 config，自动加载"
- **事实**：OpenCode 加载插件两条路：①本地文件——JS/TS 放进 `.opencode/plugins/`（项目级）或 `~/.config/opencode/plugins/`（全局），启动时自动加载；②npm 包——在 `opencode.json` 的 `plugin` 数组写包名。插件是导出插件函数的模块，函数接收上下文（project/client/`$`/directory/worktree）并返回 hooks 对象。加载顺序：全局 config → 项目 config → 全局插件目录 → 项目插件目录，所有 hooks 按序执行。
- **来源**：OpenCode 官方《Plugins》 https://opencode.ai/docs/plugins/ ；《Config · Plugins》 https://opencode.ai/docs/config/
- **成熟度**：L4
- **与 13 篇的关系（避重风险）**：13 篇已讲"本地插件自动加载 + 插件=函数返回 hooks"。**36 不要重复讲这个基础机制**，一句带过即可，重点放到 B2/B3/B4。

### B2. agents（角色定义 .md）——"写个 Markdown 文件就多一个 AI 角色"
- **事实**：agent 两种定义方式：①`opencode.json` 的 `agent` 字段写 JSON；②放 Markdown 到 `~/.config/opencode/agents/` 或 `.opencode/agents/`，**文件名即 agent 名**（`review.md` → `review`）。Markdown 用 YAML frontmatter 声明 `description`（必填）、`mode`（primary/subagent/all）、`model`、`temperature`、`permission` 等，正文即该角色 system prompt。分 primary（主对话）和 subagent（被主 agent 调用或 `@` 提及调用）两类。
- **来源**：OpenCode 官方《Agents》 https://opencode.ai/docs/agents/
- **成熟度**：L4

### B3. skills（SKILL.md 技能包）——"约定目录 + frontmatter，按需加载"
- **事实**：技能是"一文件夹放一个 `SKILL.md`"。搜索位置含 `.opencode/skills/<name>/SKILL.md`、`~/.config/opencode/skills/<name>/SKILL.md`，也兼容 `.claude/skills/`、`.agents/skills/`。`SKILL.md` 必须以 YAML frontmatter 开头，识别字段仅 `name`、`description`（均必填）、`license`、`compatibility`、`metadata`。技能**按需加载**：OpenCode 在 `skill` 工具描述里列出可用技能（名+描述），agent 需要时才 `skill({ name })` 加载全文。`permission.skill` 控制哪些技能可访问（allow/deny/ask，支持通配符）。
- **来源**：OpenCode 官方《Agent Skills》 https://opencode.ai/docs/skills/
- **成熟度**：L4

### B4. tools（自定义工具脚本）——"文件名即工具名，任何语言都能包"
- **事实**：自定义工具放 `.opencode/tools/` 或 `~/.config/opencode/tools/` 的 TS/JS 文件，**文件名即工具名**（`database.ts` → `database`）；一个文件多导出则生成 `<文件名>_<导出名>` 多工具。用 `tool()` helper 声明 `description`、`args`（Zod schema）、`execute`。**工具定义是 TS/JS，但 execute 里可调用任何语言脚本**（官方示例用 `Bun.$` 跑 Python）。与内置工具重名时自定义优先。
- **来源**：OpenCode 官方《Custom Tools》 https://opencode.ai/docs/custom-tools/
- **成熟度**：L4

### B5. 「靠注册扩展、不改核心」的解耦机制——公开依据（本方向核心结论）
- **事实 + 判断（L3~L4）**：B1–B4 合起来看，OpenCode 官方有一套**统一的"约定目录 / config 声明 → 自动发现 → 加载"注册模型**：agents/skills/tools/plugins 各有约定目录（`.opencode/agents|skills|tools|plugins/`）或 config 字段，放进去/写进去就被自动加载，**无需改 OpenCode 本体源码**。config 还多层合并（remote → global → project → `.opencode/`），项目级覆盖全局默认。这就是"场景化能力集合如何组织成 agents/skills/tools 注册进 AI 后端、完全解耦不碰内核"的公开机制依据。
- **来源**：综合 https://opencode.ai/docs/config/ （Locations/Precedence/Plugins）+ B1–B4 各页
- **成熟度**：L4（各机制官方明载）/「完全解耦」定性判断标 L3
- **对撰稿的意义**：36 下篇第四块骨架——**能力插件集合 = 一组角色定义（agents）+ 技能包（skills）+ 工具脚本（tools），靠 AI 后端原生注册约定挂载，不动网关、不动基础镜像。** 完美对应"缝合"：新增业务能力靠"往约定位置放文件"，而非改任何现成件内核。

## 4. 方向 C 成果：缝合式 / 组装式交付的方法论印证

> 服务 35 上篇立意。诚实说明：本方向找到一个强一手源（Palantir FDE 官方博客），另两个经典条目抓取失败，属证据缺口。

### C1. FDE 官方定义——"配置现成平台、不重造轮子、组合架构"（强佐证）
- **事实（一手引用）**：Palantir 官方博客对 Forward Deployed Software Engineer（FDSE）的定义与自述，几乎逐句命中本立意：
  - "FDSE 是嵌入客户现场、**配置 Palantir 既有软件平台**来解决客户最棘手问题的软件工程师"；
  - "与咨询顾问不同，我们能**大部分开箱即用地把各部件拼起来（pull most of the pieces together out-of-the-box），不必为每个客户重新发明轮子（reinvent the wheel）、不必花数年拼补丁式方案**"；
  - "我们聚焦于**组合出正确的功能架构（composing the right architecture of features）**"；
  - "平台**建立在数十个开源工具和微服务之上**"，FDE 常"审阅、理解、测试从未碰过的既有系统与代码"。
- **来源**：Palantir Blog《A Day in the Life of a Palantir Forward Deployed Software Engineer》2020-11-02 https://blog.palantir.com/a-day-in-the-life-of-a-palantir-forward-deployed-software-engineer-45ef2de257b1
- **成熟度**：L3（企业官方博客、一手观点；非中立学术定义）
- **对撰稿的意义**：35 上篇讲 FDE"交付价值不是代码"时，**这是最硬的可引用一手佐证**——"不重造轮子、把现成件拼起来、组合架构"就是"缝合式交付"的行业原话。注意：这是 Palantir 语境的 FDSE，引用时说清是"业界（Palantir）对 FDE 角色的公开定义"，勿暗示与本案例组织有关。

### C2. "组合/集成优于自建"（composition / glue code / build-vs-buy）——通用理念
- **判断（L2，概念行业公认但本次未取到权威原文）**：软件工程长期有"能买/能组装就不自建"（build vs buy）、"组合优于创造"、"胶水代码/粘合层"（glue code）等通用理念，核心是价值在**选型、集成与取舍**，而非重复造底层能力。与本立意同构。
- **来源现状**：本次抓取 Wikipedia《Glue code》（返回 403）与 Martin Fowler bliki 词条（404，URL 有误）**均失败**，暂无可直接引用的权威 URL。**标为证据缺口**（见第 7 节）。
- **成熟度**：L2（概念行业公认）/ 但**可查证 URL 缺失**，正式引用需补源。

### C3. VS Code for the Web 的"扩展生态即缝合面"旁证
- **事实**：VS Code 官方明确其可扩展性建立在"丰富的扩展生态，支持几乎所有后端、语言与服务"，vscode.dev 就是靠扩展接入各种远端与服务。侧面印证"编辑器本身是现成底座，业务价值靠在扩展点上缝接"。
- **来源**：VS Code 官方《VS Code for the Web》 https://code.visualstudio.com/docs/remote/vscode-web
- **成熟度**：L4（作"编辑器天生为缝合而设计"的旁证）

## 6. 给撰稿的建议

### 6.1 哪些放 35 上篇（业务心智），哪些放 36 下篇（工程落地）
- **35 上篇（业务心智）**：
  - 用 C1（Palantir FDE 官方定义）立"FDE 交付价值不是代码"的行业锚点。
  - 用"装机/攒机"比喻讲清四块积木的**职责与咬合关系**（脱敏代号），强调"没造一个零件、只做缝合决策"。
  - C2 的"组合优于自建"理念作为价值观收口（若补到可引用源）。
  - **不碰任何机制细节**——机制留给 36。
- **36 下篇（工程落地）**：
  - 第三块新料（编辑器交互插件）用 A1/A2/A3/A4：激活即唤醒→拉起本地后端与 UI→前端经代理接后端→用集成浏览器打开现成 UI。**核心叙事：这层是"薄胶水"，官方本就提供 Simple/Integrated Browser 这种"打开现成 UI"的机制，不用自己重画界面。**
  - 第四块新料（能力插件集合）用 B2/B3/B4/B5：能力 = agents（角色）+ skills（技能包）+ tools（工具脚本），靠约定目录/config 自动注册，不改内核。

### 6.2 第一次出现要给直觉解释的概念
- **Webview vs Simple/Integrated Browser**：直觉——"前者是插件自己在编辑器里画一个网页面板；后者是插件把'已经在本地某端口跑着的现成网页'直接打开给你看"。本案例走后者=更省事的缝合。
- **dev server proxy（代理）**：直觉——"前端和后端是两个门牌号，浏览器不让跨门牌号串门；代理就是个'前台转接'，让前端以为后端就在自己楼里"。
- **agents/skills/tools 三件套**：直觉——"角色（agents）= 给 AI 换个身份和口吻；技能包（skills）= 给 AI 一本按需翻阅的操作手册；工具（tools）= 给 AI 一个能真正动手干活的按钮"。
- **注册式扩展**：直觉——"不改主程序，只把新东西放进它约定好的'收件箱'文件夹，它自己就发现并用上了"。

### 6.3 与 17 / 18 / 13 篇的重复风险（必须规避）
- **17 篇（运行时底座）**：本组只一句话带过，**不要重讲** /opt 固定、seed-if-absent、环境变量覆盖等细节。
- **18 篇（调度网关）**：本组只一句话带过，**不要重讲** K8s 创建/删除、子域名路由、WebSocket 透传、Redis TTL 三规矩。
- **13 篇（插件是聊出来的 / OpenCode 插件机制）**：13 篇已核实"本地插件放 `.opencode/plugins/` 自动加载、插件=导出函数返回 hooks"。**36 的 B1 只做一句衔接，不要重复展开插件基础机制**；新增价值全在 B2/B3/B4（agents/skills/tools 如何组织与注册）——这是 13 篇没讲过的层。

### 6.4 一条建议作为暗线（可选但推荐）
把"这套产品的**可缝合性**不是 FDE 发明的，是 VS Code / OpenCode 这些开源工具作者**原生设计好的扩展点**"作为 35-36 的暗线：FDE 的本事不是造扩展点，而是**看懂哪里有现成扩展点、并决定往里缝什么**。这让"价值在取舍不在代码"更有说服力。
