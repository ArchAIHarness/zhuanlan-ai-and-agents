## §一、纪律一：频率限制——Redis 滑动窗口

频率限制是三道闸里**最容易被轻视**的一道。

为啥轻视？因为老板以为"我加了 Redis 频控，1 分钟 1 条够了吧？"

上一篇我们说过——**1 分钟 1 条/手机号够不够？攻击者手里有 100 万个手机号，他开 10 万线程照样打你 100 万条。**

频控是必要的，但它只防"同一手机号被打爆"，**不防"被打爆的总额"**。

那怎么防总额？

答案分两层——**单手机号频控 + 单租户/单 IP 总频控**。

**第一层：单手机号频控——阿里云给的硬天花板。**

这是**阿里云官方硬限速**，任何客户场景都绕不过去：

| 维度 | 阈值 |
|---|---|
| 验证码 — 同签名同手机号 | **1 条/分钟** |
| 验证码 — 同签名同手机号 | **5 条/小时** |
| 验证码 — 同签名同手机号 | **10 条/天** |
| 验证码 — 同手机号（全签名合计） | **40 条/天（真实硬上限）** |

**40 条/天**——这是你无论如何也绕不过去的真硬上限。**哪怕你换签名、换模板**，同一个手机号一天最多收到 40 条验证码。

**触发拦截后 24 小时自动解除**——攻击者真把你打爆了，不用慌，过了 24 小时自然恢复（钱已经扣了）。

**时间窗是 UTC+8 整点切窗**——不是滚动窗口，是整点切。14:00:59 发的和 14:01:01 发的跨两个整点，互不冲突。

**短信提交即计入流控**——SDK 返回 OK 但运营商没下发，**也按"已发"算**。所以**SDK 超时不能瞎重试**，重试一次就多扣一条额度。

**第二层：单租户/单 IP 总频控——防"100 万手机号打爆总额"。**

这一层阿里云不替你做，**必须你自己写**。

工程实现——**Redis 滑动窗口**。

最简单的实现：`INCR + EXPIRE`，两行代码搞定。

```python
def send_sms_fixed_window(phone):
    key = f"sms:rate:{phone}:{int(time.time() // 60)}"
    with redis.pipeline() as pipe:
        pipe.multi()
        pipe.incr(key)
        pipe.expire(key, 59)  # 59 而非 60,容忍时钟漂移
        count, _ = pipe.execute()
    if count > MAX_PER_MINUTE:  # 例如 1(验证码类型)
        raise RateLimitExceeded(f"per-minute cap: {MAX_PER_MINUTE}")
    return call_aliyun_sms_sdk(phone, code)
```

**金句 1："1 条/分钟、5 条/小时、10 条/天"是阿里云硬上限。**

**金句 2："40 条/天"是同手机号全签名合计硬天花板。**

但 `INCR + EXPIRE` 有个**致命问题**——**临界突刺**。

举个例子：14:00:59 攻击者打 1 条（命中 14:00 这个窗口），14:01:00 攻击者又打 1 条（命中 14:01 这个新窗口），14:01:59 再打 1 条（还是 14:01 窗口）。**2 秒内打了 3 条，但频控只看到 1 条/分钟**。

要解决临界突刺，**必须用 ZSET 滑动窗口**——

```python
def send_sms_sliding_window(phone, window_seconds=60, max_count=1):
    now = time.time()
    key = f"sms:window:{phone}"
    with redis.pipeline() as pipe:
        pipe.multi()
        pipe.zremrangebyscore(key, 0, now - window_seconds)  # 清窗口外
        pipe.zcard(key)                                       # 看窗口内有多少
        pipe.zadd(key, {f"{now}-{uuid4()}": now})             # 加一条
        pipe.expire(key, window_seconds + 10)                 # 防内存泄漏
        _, count, _, _ = pipe.execute()
    if count >= max_count:
        raise RateLimitExceeded(f"sliding {window_seconds}s cap: {max_count}")
    return call_aliyun_sms_sdk(phone, code)
```

**金句 3："INCR + EXPIRE 两行搞定；ZSET 滑动窗口精确款——生产推荐用这个。"**

**金句 4："UTC+8 整点切窗，触发拦截 24 小时自动解除——是阿里云原话。"**

频率限制这一道闸，**用滑动窗口**就对了。

## §二、纪律二：验证码强度——6 位数字 + 5 分钟过期

验证码是短信防刷的"门面"——绝大多数老板以为"加了验证码就安全了"。

上一篇我们驳过这个观点：**4 位数字验证码 = 10000 种组合 = 单线程 30 分钟跑完。**

这一篇我们把"6 位数字 + 5 分钟过期"**拆到底层逻辑**。

**第一层：位数——为什么是 6 位？**

| 验证码类型 | 字符空间 | 单次尝试命中概率 |
|---|---|---|
| 4 位纯数字 | 10⁴ = 1 万 | 1/10000 = 0.01% |
| **6 位纯数字** | **10⁶ = 100 万** | **1/1000000 = 0.0001%** |
| 8 位纯数字 | 10⁸ = 1 亿 | 1/1 亿 |

**4 位数字 30 分钟跑完，6 位数字 5 分钟跑不完**——

如果攻击者 1 秒试 100 次：4 位验证码，**100 秒跑完**。6 位验证码，**10000 秒 ≈ 2.78 小时跑完**——超过 5 分钟过期时间，验证码已经失效。

**金句 5："位数少 1 位，理论上被破概率降为 1/10；但用户在手机上输入 8 位比 6 位更容易错。"**

这是**安全 vs 易用的经典取舍**。

**第二层：过期时间——为什么是 5 分钟？**

**金句 6："5 分钟不是拍脑袋定的，是 NIST SP 800-63B 行业标准。"**

美国 NIST 在《Digital Identity Guidelines — Authentication and Lifecycle Management》（SP 800-63B）里把 SMS OTP 定位为"带外设备验证器"——必须有合理过期时间。

行业事实标准是 **5 分钟过期**——阿里云、腾讯云、容联等主流厂商的默认模板都是这个数。

**第三层：单次有效——为什么必须删掉？**

验证码不是密码——密码能复用，验证码**只能用一次**。

工程实现——`SET key value EX 300 NX` 一行命令完成"写入 + 5 分钟过期"。校验通过**立刻删**——`DEL key` 必须紧随校验通过之后。**校验成功后立刻 DEL，绝不允许验证码复用。**

**第四层：错误次数限制——为什么错误 5 次必须锁？**

如果验证码位数是 6 位、过期是 5 分钟，攻击者 1 秒试 100 次，**理论上还是可能试出 1 个对的**。

所以必须加**错误次数限制**——**错误 5 次锁 10 分钟**。

工程实现——Redis 里多记一个 `err_count` 字段。每次校验失败 +1，到 5 就锁这个手机号 10 分钟。

**金句 7："错误次数限制必带，没有这个攻击者可以用同一个手机号一直试错。"**

**完整状态机**（用 Redis 描述）：

```
待生成 → 已下发（存 Redis: code + expire_at + err_count）
已下发 → 校验通过 → 已使用（DEL key）
已下发 → 错误计数 + 1（err_count < 5）→ 已下发
已下发 → 错误计数 == 5 → 已锁定（10 分钟）
已下发 → 当前时间 > expire_at → 已过期（自动 TTL 清理）
```

**金句 8："存 Redis 而非 MySQL——5 分钟过期 = 一行 EX 命令；MySQL 要写定时器。"**

把"6 位 + 5 分钟 + 单次 + 错误锁"四件事焊死，验证码这一道闸就**真正防得住**了。

## §三、纪律三：图形验证码前置——证明你是人

验证码这一道闸解决了"6 位 + 5 分钟"——但**它是第二道**。

第一道闸是什么？**图形验证码前置**——**在你拿到 6 位短信验证码之前，先证明你是人**。

为啥要这道闸？

因为**短信验证码是 2FA（双因素认证）**——你必须证明"拥有这个手机号"，而不仅仅是"知道密码"。

但在你证明"拥有这个手机号"之前，你得先证明"你是个人类在操作"——这叫**1FA（单因素认证）**。

**金句 9："短信验证码是 2FA，图形验证码是 1FA，1FA 都没过别想 2FA。"**

为啥这么分？**因为绝大多数攻击者是脚本机器人**——它们只会机械地 POST 请求，不会算数。

让它们先做一道"3 + 5 等于几"的题，9 成 9 直接卡死。

剩下 1 成用 OCR 识别或滑动轨迹伪造——这些进阶攻击，**就交给图形验证码厂商的 AI 反作弊引擎**。

**主流图形验证码方案**：极验 GT4（滑动拼图 + 行为轨迹分析）、网易易盾（滑动 + 点选 + 行为评分）、腾讯天御（腾讯系业务广泛使用）——**这三家随便选一个都行，别自己造**。

为啥？**自己造 = 你得跟全世界的 OCR 工程师 + 滑动轨迹伪造工程师对抗**——这是 AI 与 AI 的战争，**别让自己去当靶子**。

工程实现——图形验证码是**一次性 token**，**验证通过后服务端才放行下一步**：

```python
def send_sms_with_captcha(phone, captcha_token):
    # 1. 校验图形验证码 token
    if not captcha_service.verify(captcha_token):
        raise CaptchaInvalid("图形验证码未通过")

    # 2. 通过后再走频控 + 验证码生成
    return send_sms_sliding_window(phone, code)
```

**金句 10："1FA 都没过别想 2FA——这是顺序焊死，不能颠倒。"**

**金句 11："图形验证码别自己造——你打不过全世界的 OCR 工程师。"**

这道闸看似简单，**但少了它，你后面所有的闸都白费**——脚本机器人会绕过所有"代码层"的防御，因为它们根本不需要破解代码，**它们只需要打 URL**。

## §四、纪律四：签名实名制——打电话前登记身份证

前面三道闸——频率限制、验证码强度、图形验证码前置——都是**给攻击者设的关卡**。

这一道闸——**签名实名制**——是**给合法用户设的规矩**。

啥叫签名实名制？

工信部 + 运营商联合要求：**所有短信签名必须完成实名制报备，否则将导致发送失败。**

说人话——

**金句 12："打电话之前必须登记身份证，不登记运营商不发。"**

为啥要这道闸？

因为任何短信出去，运营商那头都要追溯——"这是谁发的、用了什么签名、走的是哪个模板"。

**签名不规范 = 短信发不出去。**

2026 年的最新规范（阿里云官方原文）——以下签名来源**已不再支持**运营商实名报备：

- ❌ 已上线 **APP 名**
- ❌ **公众号或小程序名**
- ❌ **电商平台店铺名**
- ❌ 已备案网站名

✅ **唯一合规的签名来源**：**企事业单位全称或简称** 或 **已注册商标名**。

也就是说——你的签名必须是**"XX 公司"**或**"XX 商标"**——**中性词（"客服通知""客户您好""温馨提示"）审核直接打回**。

**金句 13："中性签名 = 短信发不出去——工信部强制要求。"**

**金句 14："签名不规范 = 100% 发送失败，不是降级，是完全发不出去。"**

这里有个 **FDE 第一年最容易踩的坑**：客户说"下周要上线注册功能"，你心想"接个 SDK 就行了"。结果卡在签名报备上一周走不通。

**新业务预留至少 10 个工作日**——阿里云官方原话："新业务上线请务必至少提前 10 个工作日完成所有报备流程，避免影响业务计划。" 实际观测——运营商审核平均要 **5-7 个工作日**，实际多在 **7-10 个工作日**。

**金句 15："新业务上线预留 2 周做签名报备——阿里云原话 10 个工作日，实际观测 7-10。"**

**金句 16："已不再支持已上线 APP、公众号/小程序、电商平台店铺名作为签名——2026 年新规。"**

这一道闸的工程实现——

```python
def send_sms_with_signature(phone, code, sign_name, template_code):
    # 1. 校验签名是否在阿里云控制台报备
    if not sign_name in approved_signs:
        raise SignatureNotApproved(f"签名 {sign_name} 未报备")

    # 2. 校验模板是否合规
    if not template_code in approved_templates:
        raise TemplateNotApproved(f"模板 {template_code} 未审核")

    # 3. 模板必须含"验证码/注册码/校验码/动态码"之一
    # 4. 必须含"使用平台/用途/失效时间"之一
    # (这俩阿里云控制台审核时已经卡了)

    return call_aliyun_sms_sdk(phone, sign_name, template_code, code)
```

**短信防刷不光是"防攻击者"，也是"防你自己不规范"——签名报备不到位，你自己的短信都发不出去。**

## §五、纪律五：审计追溯——出事能查

最后一道闸——**审计追溯**。

前三道闸都是"事前预防"，这一道闸是"事后复盘"。

**金句 17："事前的事故预防靠前三道闸，事后的事故复盘靠审计。"**

为啥要这道闸？因为**再严的防护也有漏网之鱼**——你前四道闸做得再好，攻击者也可能找到新的角度打你。

一旦被打，**5 分钟内能不能定位"哪个手机号、哪个 IP、哪个签名、哪个模板、哪个租户"**——这就是审计的职责。

**每条短信必须留痕的 7 个字段**（一个不能少）：`sms_id`（一次请求服务端唯一编号）/ `tenant_code`（发起方租户）/ `phone`（目标手机号，hash 后存储）/ `client_ip`（调用方源 IP）/ `sign_name`（短信签名）/ `template_code`（短信模板 ID）/ `send_time`（调用 SDK 的服务端时间）。

**进阶可记字段**（8 项）：`sdk_request_id` / `sdk_response_code` / `biz_biz_id` / `rate_limit_window` / `captcha_passed` / `region_carrier` / `cost_cents` / `error_count_before`。

工程实现——**RAM 子账号 + ActionTrail + 对象存储 + 日志服务**四件套：**RAM 多因素认证 + 最小权限**（业务账号只能调 `SendSms` + `QuerySendDetails`，**不能改签名/模板**）；**ActionTrail 全公司强开**（记录 OpenAPI 调用日志 + 控制台操作）；**OSS 异地备份**（ActionTrail 日志归档）；**SLS 日志服务**（实时告警 + 长期查询）。

**金句 18："每条短信必须留痕：手机号/模板/发送时间/费用/状态码——7 个字段一个不能少。"**

**金句 19："ActionTrail 90% 的中小 SaaS 公司都漏掉——出事时是唯一能证明'我们没犯错'的证据。"**

**5G 消息可撤回（5 分钟内）——2G 时代不能。**

这一条是审计追溯的**最后一道兜底**。

如果 5G 通道发送的短信，运营商支持 5 分钟内撤回——这是 2026 年的新能力。

如果是 2G 通道发送的短信，**泼水难收**——发了就发了，没法撤回。

所以——**短信通道选 5G 是真香的**——出事能补救。

**金句 20："5G 消息可撤回 5 分钟内，2G 不能——出事能不能补救，看你用哪代。"**

## §六、把五道纪律串成"半夜 100 万条短信"的事故复盘

上一篇我们讲过那个"凌晨三点告警响"的故事。

这一篇我们把那个故事**重新走一遍**——但这次用**五道工程纪律**复盘。

**出事前：频控 + 验证码 + 图形验证码 + 签名实名 + 审计 = 五道防线。**

- 频控：Redis 滑动窗口，单手机号/单租户/单 IP 三层卡死
- 验证码：6 位数字 + 5 分钟过期 + 单次有效 + 错误 5 次锁
- 图形验证码前置：极验/网易易盾/腾讯天御，证明你是人
- 签名实名制：报备审核 10 个工作日，规范签名走完流程
- 审计追溯：每条短信 7 字段必记 + ActionTrail + RAM 最小权限

**出事中：5 分钟内定位 + 5G 撤回 + 报警。** ActionTrail 查 5 分钟内的请求记录——**哪个 IP 段打的、用的是哪个签名、走的哪个模板**；5G 消息 5 分钟内撤回——**损失可控**；钉钉值班群 + 短信值班 + 客户成功——**三通道报警**；RAM 子账号权限排查——**是不是业务账号被滥用**；基线 × N 告警规则——**单分钟全平台短信提交量超阈值立刻告警**。

**出事后：复盘 + 整改 + 同步客户。** 写事故报告（根因 + 损失 + 整改措施）；同步客户（精确损失金额 + 已采取措施 + 后续承诺）；整改（哪道闸有缺口，加哪道闸）；5G 通道升级（把 2G 通道全量迁移到 5G）；给客户做余额补偿或短信额度赠送。

**金句 21："出事前：五道闸焊死。出事后：审计 + 撤回 + 报警 + 复盘。"**

**金句 22："出事的从来不是'没加验证码'，是'加了但只加了一道'。"**

这就是完整的"短信防刷五道工程纪律"——**频控、验证码、图形验证码、签名、审计**——**五道防线一条都不能少**。

少一道，半夜三点的电话就会响。

回过头看上一篇和这一篇——
