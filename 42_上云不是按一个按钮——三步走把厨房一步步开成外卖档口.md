# 上云不是按一个按钮——三步走把厨房一步步开成外卖档口

上一篇我们聊了心智——"上云是把厨房开成外卖档口"。

这一篇我们卷袖子干活——"怎么开"。

我把"上云"这件事拆成三步，每一步都讲清楚：做什么、怎么做、踩过什么坑。

你看完这三步，可以拿着这张图去跟老板汇报、跟团队拆任务、跟客户聊方案。

但有一件事我必须先讲在前面——

**上云不是按一个按钮。**

客户问得最多的就是"我们啥时候能点那个按钮"，好像上云就是按一个开关的事。

不是。

**上云是三件不同的事，每件事解决不同的问题，每件事都有自己的坑。** 把这三件事分清楚，你跟老板、客户、团队讲上云的时候，才不会被人糊弄。

为啥我必须先讲这句话？因为太多团队栽在"以为上云是一件事"上——他们以为搞完 K8s 就算上云了，结果数据库还在自建机房、对象存储还在用 NAS、监控还是 5 年前搭的 Nagios。

上云第一步，他们压根没迈出去。

下面我一步步拆给你听。

## §一、第一步：装铁盒子——把应用装进标准化盒子里

三步里的第一步，叫**装铁盒子**。

啥意思？把你的应用，连同它依赖的运行环境，打包塞进一个标准化的盒子里。这个盒子长啥样、装多满、怎么搬，都按一个统一的标准来。

为啥要先做这一步？因为你以前装一个应用，是装在一台服务器上——这台服务器有自己的操作系统、自己的配置、自己的脾气。换一台机器，同样的应用可能要重新装一遍。

你以前一个应用跑 5 台机器，每次升级都得登 5 台机器、传 5 次文件、起 5 次进程——出错概率直接 ×5。

**装铁盒子就是把"装一次"变成"装一次，到处跑"。**

我打个生活比方——

> 你以前是给每个厨师发一份"菜谱复印件"——A 厨师拿到的是一版，B 厨师拿到的是另一版，字还可能抄错。装铁盒子就是给所有厨师发同一份"标准印刷菜谱"——你只印一次，每个厨师拿到一模一样的一份，谁也改不了、抄不错。

这一步在云上有个正式名字，叫**容器化**。云上有个标准组织叫 OCI（Open Container Initiative），2015 年由 Docker、CoreOS 一起贡献给 Linux Foundation，专门定义"铁盒子"应该长啥样、怎么打包、怎么分发。

为啥这事这么重要？因为**没有铁盒子，后面的调度员、无人仓都没法玩**。你连盒子都没有，调度员管啥？无人仓装啥？

这一步有三件事你要心里有数。

**第一件，铁盒子有尺寸讲究。**

你装一个 JDK + Tomcat + Web 应用的中型 Java 盒子，大小常常 800MB-1.2GB——比一部高清电影还大。

为啥这么大？因为底包不讲究。官方推荐的做法是用 Alpine 镜像做底包，大小 < 6MB；外加多阶段构建，能把最终镜像砍掉 60-80%。

但很多人图省事直接用 Ubuntu 做底包，800MB 一大坨全部塞进盒子里。结果一个铁盒子启动要拉 1GB 的镜像，光拉镜像就 30 秒，业务还没起来用户先走了。

**铁盒子的尺寸，决定了它能跑多快。**

**第二件，铁盒子启动有快慢。**

JVM 应用启动普遍 5-30 秒；用 GraalVM 编译的 Native Image 可以降到几百毫秒。

为啥这事重要？因为你以后上"调度员"的时候，调度员要快速拉新盒子接客。盒子启动慢 30 秒，客人已经走了；盒子启动快 100ms，客人还没看到白屏，菜已经上桌了。

**铁盒子的冷启动速度，决定了它能不能跟得上客人的节奏。**

**第三件，反常识金句——容器化不是微服务化。**

这一句你必须刻进脑子里。

你把一个 Java 单体应用装进铁盒子，它还是单体。它没有变小、变快、变多。它只是换了个地方住的胖子。

Martin Fowler 在 2015 年专门写过一篇《MonolithFirst》——他论证过，几乎所有成功的微服务改造都起步于一个"已经太大的"单体。所以先用容器把单体原地搬上云，是工程上更稳的起点。

先容器化，再考虑要不要拆。

**容器化解决的是"标准化打包"的问题，不是"重新拆分业务"的问题。** 别把它俩混为一谈。

这一步做完，你能拿到什么？

你的应用从一个"挑机器的娇气鬼"，变成了一个"扔哪都能跑的铁盒子"。换台机器？装一遍？配一遍？都不用了。盒子还是那个盒子。

这是上云的第一步，也是后面两步的**地基**。

## §二、第二步：搬食材——把外部依赖搬到云上

第一步把应用装进铁盒子，第二步把"食材"搬到云上冷库。

啥叫"食材"？就是你的应用**之外**那三件东西——

- 数据库（你的核心账本）
- 缓存（你的临时便签）
- 文件（你的档案柜）

这三件东西在传统机房时代，是你"自己家厨房的食材"——自己买、自己洗、自己切、自己放冰箱、自己怕坏。

上了云，你不用自己养冰箱了——云上有现成的冷库让你租。

为啥要先做这一步？因为**应用只是烧菜，食材才是家底**。你应用写得再漂亮，账本丢了、便签没了、档案没了，业务照样崩。

这一步在云上叫**外部化**——把应用外部的"基础设施三件套"迁到云上托管服务。**应用本身仍然跑在云上虚拟机上**（容器或裸进程都行），但下面的数据库、缓存、文件都不再自己运维。

这一节的反常识金句最多，我给你逐条拆。

**反常识 1：数据库不是你想迁就迁的——DB 不该上 K8s。**

很多人以为"上云就是上 K8s，应用跑在 K8s 里，数据库也跑在 K8s 里，多整齐"。

不是。

DB 是"账本"，K8s 是"调度员"。调度员擅长调度一群无状态的工人，**不擅长照顾一本珍贵的账本**。

K8s 的设计哲学里，调度员假设每个盒子都是"死了可以随便换"的。DB 不是——DB 一旦出问题，整本账本都没了，调度员没能力替你兜底。

行业实践里几乎所有生产环境都是把 DB 放在云厂商的托管数据库服务里（云数据库 RDS 系列），不是放在 K8s 里。

**DB 要的是稳定，不是弹性。**

**反常识 2：迁到云上数据库，记得问三件事——RPO、SLA、跨可用区。**

你从自建 MySQL 迁到云上 RDS，不是改一行连接串那么简单。你得问三件事——

- **RPO**（Recovery Point Objective）：你能接受丢多少数据？最严格可以做到 RPO=0（同步复制，集群版），代价是贵一些；RPO=5 分钟意味着极端情况下丢 5 分钟账。
- **SLA**（Service Level Agreement）：服务可用性承诺到多少？云厂商的高可用版 RDS 通常 SLA 高达 99.997%——一年只允许宕机 15 分钟。
- **跨可用区**：DB 是单可用区还是多可用区部署？多可用区意味着同城多活，RPO=0 但读写延迟会增加几毫秒；单可用区便宜但有可用性短板。

这三件事问完，再谈钱。

**反常识 3：对象存储不是文件系统——这是无数人栽过的坑。**

云上的文件存储（OSS / COS / OBS / S3 这类）叫做**对象存储**。它的承诺是 99.9999999999%（12 个 9）的数据持久性——你存一万亿年才会丢一次。

但它不是文件系统。

文件系统有目录层次、有文件锁、有原子重命名。对象存储**没有目录层次、没有文件锁、没有原子重命名**。

你要是把对象存储当文件系统用——用 ossfs / s3fs 挂载当本地盘——并发写、随机写、文件锁全都失真，出问题查不到根因。

对象存储的正确用法是：**只存一次性写入、读多写少、最后不再修改的东西**——图片、视频、日志、备份、归档文件。

**对象存储不是文件系统，是超大规模的水泥仓库——只能堆，不能锁。**

举个例子你就懂——你想往水泥仓库里存 10 万张用户头像，可以；你想改其中一张的某个像素，不行；你想同时锁住两张图防止别人改，不行；你想列个"按用户 ID 排序的目录"，不行。

水泥仓库只能堆，不能锁——对象存储就这样用。

**反常识 4：Redis 不是 DB——它俩的承诺不一样。**

自建 Redis 多用 AOF 或 RDB 持久化。云上托管的 Redis（云缓存）常默认关闭持久化或限制持久化频次。

为啥？因为 Redis 设计上是缓存，不是 DB。

你要是把 Redis 当 DB 用——以为它能像 MySQL 那样"丢了能从日志恢复"——你就踩坑了。云上的 Redis 是**默认接受"丢一点没事"**的。

如果你要 DB 级别的持久化，云厂商也提供持久化版 Redis，但价格贵得多、延迟高一些。

**Redis 是便签纸，DB 是账本。便签纸丢了你可以重抄，账本丢了你就完了。**

这一节反常识金句够多吧？讲完这四句反常识，你对"搬食材"这件事的理解，就已经超过了 90% 的初次上云团队。

## §三、第三步：让铁盒子自己会搬家——弹性扩缩

第二步把食材搬好了，第三步让你的"铁盒子"自己会搬家。

啥叫"自己会搬家"？就是客人多时自动多派几个厨师，客人少时让厨师回家。

这一步在云上叫**弹性扩缩**。云上有两条路可以走——

- **路 A：上 K8s**——一群铁盒子的总指挥，自动帮你调度盒子
- **路 B：上 Serverless**——连盒子都省了，你只管扔代码

两条路都能解决"自己会搬家"这个问题，但它们的脾气不一样。我先讲共同点，再讲反常识。

**共同点 1：扩容速度都不是即时的。**

K8s 单节点从创建到就绪约 30 秒；扩容 1000 个节点约 40 秒（不同云厂商有差异）。

Serverless 冷启动普遍 100ms-数秒；Java/Python 函数几百毫秒到数秒，Native Image 编译后可到 100ms。

注意——**没有任何一种云上扩容是即时的**。客人爆增的那一瞬间，你的扩容请求已经在排队了；等新的盒子就绪，客人可能已经走了。

这就是为啥"预热"和"压测"这么重要——你不能等流量真来了才想到扩容。

**共同点 2：HPA 配错比不配更糟。**

K8s 有一个机制叫 HPA（Horizontal Pod Autoscaling）——按 CPU/内存/自定义指标横向扩缩 Pod 数量。

很多人配 HPA 就一句"CPU 超过 80% 就扩容"，结果——

- 配得太紧：流量一抖就扩容，频繁抖动反而把集群拖垮；
- 配得太松：等 CPU 真的 80% 了，业务已经卡 5 分钟了。

K8s 官方建议是基于业务指标（QPS/队列深度）而非单一 CPU。你得根据自己的业务特点调阈值，没有万能配法。

**HPA 不是装上就灵，是要根据业务节奏慢慢调。**

**反常识 1：Serverless 不省钱——稳态负载反而更贵。**

这是云原生最大的反常识之一。

Serverless 的卖点是"按调用计费、零闲置成本"。但这个卖点只在**两种场景下成立**——

- **闲时负载**：平时没流量，每天零星几个调用，云上几乎不收钱；
- **突发流量**：平时几千 QPS，突然冲到几十万 QPS，云上扛得住你不收额外钱。

但**常驻高并发、稳态流量**呢？

Serverless 按 vCPU-秒 + 调用次数计费。你 7×24 小时跑着一个常驻服务，每秒都在调函数，云厂商按秒收你 vCPU 的钱——**比你自己买机器贵得多**。

更扎心的是"预置并发"——你为了避免冷启动，得提前预留 N 个并发实例。这 N 个实例**按小时收费，不管有没有调用**。这就是另一笔保底钱。

**Serverless 不是万能解，是"脉冲流量 + 闲时任务"的解。稳态业务上 Serverless，是给云厂商送钱。**

**反常识 2：K8s 不解决 DB 问题。**

我提前在第二步讲过这一句。这里再强调一次——

客户问"我们上 K8s 是不是就齐了"？

不是。

K8s 是调度员，不是数据库管理员。它能帮你管一群无状态的铁盒子，**但管不住你那本珍贵的账本**。

DB 还是得放在云数据库服务里。K8s 只管应用层。

**K8s 管盒子，云数据库管账本——这俩是分开的。**

**反常识 3：HPA 不是银弹，预热才是。**

我见过太多团队配了 HPA，然后真来一波流量，扩不出足够的盒子——因为镜像太大、冷启动太慢、节点池不够。

为啥？因为你没预热。

预热有三招——

- 镜像预热（提前把大镜像拉到节点上）
- 节点预热（提前留一批低负载节点）
- 函数预热（Serverless 的预置并发）

这三招都是提前花钱、提前准备。真出流量的时候才扩容，叫"裸奔"。

**弹性不是出事时的临阵磨枪，是平时就磨好了刀。**

## §四、Docker/K8s/Serverless 三者怎么选——不是非此即彼

三步走完，你的应用已经在云上跑起来了。但你会遇到一个问题——

**我到底是 Docker 就够了、还是要上 K8s、还是直接上 Serverless？**

这三个不是非此即彼。它们是三种档口模式。

我打个比方——

- **Docker = 自己开小档口**：标准化做菜流程，一个盒子就是一道菜，自己看着管；
- **K8s = 美食广场连锁**：很多档口共一个总指挥（调度员），自动排班、自动补人；
- **Serverless = 无柜台便利店**：连档口都没有，客人扫码下单，云帮你调出这道菜。

我举个生活场景——

> 你想周末在家烧一桌菜请朋友。方案 A：自己骑车去菜场买菜、洗切炒煮全程亲力亲为（Docker，你啥都自己管）。方案 B：雇个厨师上门做，你只管点菜（K8s，有人帮你调度一群厨师，但厨师、灶具还是要你安排）。方案 C：直接叫外卖，每道菜云厨房单独烧、按份付钱（Serverless，你啥都别管，云帮你调度一切）。

三个方案都能让你吃上饭，但你花的精力、付的钱、得到的灵活性完全不一样。

三种模式各有各的脾气。

| 模式 | 适合谁 | 不适合谁 | 一句话 |
|---|---|---|---|
| Docker | 业务稳定、流量可预测、团队有运维能力 | 流量脉冲、追求极致省心 | 自己动手，丰衣足食 |
| K8s | 微服务多、流量波动大、要精细扩缩 | 业务简单、团队小、没人懂 K8s | 总指挥帮你管一群盒子 |
| Serverless | 事件触发、流量脉冲、闲时多忙时少 | 常驻高并发、需要长连接、状态多 | 连盒子都省，云帮你跑 |

**关键判断点不是"哪个最牛"，是"哪个最贴你的业务"**。

**反常识 1：你完全可以不上 K8s。**

很多客户听说 K8s 就害怕——"我们团队没人懂 K8s，是不是不能上云？"

不是。

你可以**只走第一步 + 第二步**——Docker 容器化 + 外部化。跑 5-10 年也很常见。

你也可以**直接上 Serverless**——用阿里云的 FC、腾讯云的 SCF、AWS Lambda、Google Cloud Run，连 K8s 都不碰。

你还可以**用云厂商的托管 K8s**——阿里云的 ACK Auto Mode、华为云的 CCE Autopilot、腾讯云的 TKE 超级节点。这些产品把 K8s 的复杂度藏在后面，你只需要写 YAML，不用懂调度器原理。

**不上 K8s 也能上云，上 K8s 也不代表你上云了。**

**反常识 2：客户问"上 K8s 就完事"——这是个误区。**

客户第二种误区——"我们上 K8s 是不是就万事大吉？"

不是。K8s 只解决**应用层**的编排。DB、Redis、对象存储——都不归 K8s 管。

你 K8s 上得再漂亮，DB 没迁好、Redis 没接好、对象存储没用好，你的业务照样崩。

**K8s 是调度员，不是数据库管理员。**

这一节讲完，你应该明白一件事——

**没有最好的档口模式，只有最贴你业务的档口模式。**

## §五、上云这条路的 9-18 个月时间线

三步走讲完了。你心里可能在盘算——"我这三步走完要多久？"

我说句实话——**9 到 18 个月**。

这是给"50 个微服务、5TB 数据、100 万注册用户"的中型 SaaS 公司摸出来的一个大致时间线。⚠️ 这是行业通用经验，不是个别项目的具体数字。

为啥这么久？三步就是三件事，每一件都有自己的里程碑。我把每一步的时间线摆给你看。

**阶段 1：装铁盒子（容器化）—— 2-4 个月。**

| 里程碑 | 工作 | 大致周期 |
|---|---|---|
| M1 | Dockerfile 标准化、定基础镜像、写 docker compose | 2-4 周 |
| M2 | CI/CD 接入镜像仓库、灰度首批镜像跑在云上 | 2-3 周 |
| M3 | 50 个服务全部上镜像、流量分批切换 | 4-8 周 |
| M4 | 监控、日志、告警全部接入云上 | 2 周 |

为啥这一步要这么久？因为你 50 个服务，每一个都得写 Dockerfile、测一遍、灰度一遍、回滚方案备一遍。

**容器化不是写 Dockerfile 那么简单，是把整个 CI/CD 流水线重写一遍。**

**阶段 2：搬食材（外部化）—— 3-6 个月。**

| 里程碑 | 工作 | 大致周期 |
|---|---|---|
| M1 | 非关系型先迁：对象存储、Redis 缓存 | 4-6 周 |
| M2 | DB 拆表迁移：先从非核心库开始 | 4-8 周 |
| M3 | 核心库 DB 迁移：双写 + 对账 + 切流 | 6-12 周 |
| M4 | 数据校验、对账、归档与冷热分层 | 2-4 周 |

为啥这一步更久？因为数据库迁移是所有上云动作里**最危险**的一步。

你以为迁库就是停写几小时、搬数据、切流——其实中间藏着无数个雷——

- 存量数据有几 T，停写几小时用户已经炸锅；
- 增量数据双写期间任何一个 bug 都可能让两边的数据不一致；
- 切流时机错了，回滚要走一遍全流程；
- 业务方突然扔过来一个"这个接口得用旧库"，你得临时兼容。

**迁库最贵的成本不是云数据库的账单，是"半在云上半在机房"的双写期人力。**

**阶段 3：让铁盒子自己会搬家（弹性化）—— 3-6 个月（可选）。**

| 里程碑 | 工作 | 大致周期 |
|---|---|---|
| M1 | 选型：上 K8s 还是 Serverless | 2-4 周 |
| M2 | Helm Chart / Operator 编写、CI/CD 接 K8s | 4-6 周 |
| M3 | 灰度 1-5% 流量到新集群、对比监控 | 4-6 周 |
| M4 | 全量切流、老 VM 退订 | 2-4 周 |

这一步是可选的——你不一定要做，做了也不一定三步全做。你可以只让一部分非关键服务先上 K8s，看效果再决定要不要全切。

**关键反常识金句——迁云不是上 K8s。**

这是这一节最扎心的一句。

很多团队以为"上云 = 上 K8s"，把 9-18 个月的时间全部砸在阶段 3。结果阶段 1、2 没做扎实，应用还是 VM 跑的、DB 还在自己机房、K8s 上只跑了几个非关键服务。

你问他"你上云了吗？"他说"上了，上了 K8s"。你再问"你数据库迁了吗？"他说"还没"。

这不是上云，这是上了一半的云。

**真正的上云，是三步全走。K8s 只是阶段 3，不是上云的全部。**

三个阶段加起来，9-18 个月是行业通用经验。具体到你的项目，可能 6 个月，也可能 24 个月。但你心里得有这个数——**不要把上云当闪击战，要当持久战**。

---

讲完时间线，我必须再补一件事——**上云路上最容易踩的三个坑**。

这一节是给你打预防针。客户听你讲完时间线，多半会心里发虚，问你"那我们中途会不会翻车？"

我的回答是——**会。** 而且是特定的三种翻车姿势。

**坑 1：阶段 1 没做扎实就跳到阶段 3。**

很多团队容器化还没做完，就被"K8s 真香"勾走了，开始全员突击 K8s。结果呢——

- 应用 Dockerfile 没写全，半数服务还在裸跑；
- CI/CD 还是 5 年前那套，K8s 上的镜像谁来推、谁来拉、谁来回滚，没人说清；
- 监控告警还在用自建的 Zabbix，K8s 上的 Pod 挂了都没人知道。

K8s 是个调度员，但你得先把"被调度的那群盒子"准备好。盒子都还没装好，调度员来管啥？

**没装好铁盒子，调度员接不住你。**

**坑 2：阶段 2 双写期拖太久，"半在云上半在机房"。**

数据库迁移最容易翻车的一件事——**双写期太长**。

理想的双写期是 4-12 周。但很多团队"双写期"一开就收不回，业务连续性导致他们迟迟不敢切流量、不敢回滚、不敢清理旧库。

结果你打开他们的架构图一看——

- 用户写：新机房 + 云上双写；
- 用户读：还是新机房；
- 对账：跑个脚本一周一次；
- 切流计划：写了三版没人敢签字。

这种"半上云"状态是最危险的——你在付两份钱、养两套环境、跑两份脚本。出问题时没人知道去哪儿查。

**双写期一长，你就同时被两个机房拖累。** 切流要趁早，拖久了切不动。

我打个比方——这就像搬家时"旧房子还没退、新房子已经住进去"的状态，你付两份房租、装两份家具、记两套地址，任何一份快递都可能被送到错的地方。

**坑 3：阶段 3 直接跳过选型，先上 Serverless 后切 K8s。**

也有团队反着来——先听说了 Serverless 真香，跳过 K8s，直接上 Serverless。

然后呢？

- 长连接被卡了——Serverless 没法常驻连接；
- 本地状态全没了——每次冷启动都重新初始化；
- 冷启动延迟用户感知到了——投诉了一波；
- 按调用付费的账单超了——老板懵了。

Serverless 不是 K8s 的替代品，它是另一种档口模式。你得先想清楚自己的业务到底稳态还是脉冲，再选。

**先想清楚再选档口，别被"哪个新"忽悠着跳。**

三个坑讲完，你应该明白一件事——**9-18 个月不是死线，是节奏**。节奏对了，6 个月也能跑完；节奏错了，24 个月还在原地打转。

**节奏对的关键，是三步顺序走、一坑一坑填、不跳步。**

## §六、写在最后

回到开头客户那句话——"我们要不要上云"。

上一篇我们讲了心智：上云不是搬家，是把厨房开成外卖档口。

这一篇我们卷袖子干了活：三步走——先装铁盒子，再搬食材，最后让铁盒子自己会搬家。

你要是把我这两篇串起来看，你会发现一件事——

**真正的上云，是分三步：先把菜装进铁盒子，再把食材搬去云上冷库，最后让铁盒子自己会搬家。**

三步走完，你才是真的把厨房开成了外卖档口。

但有三件事你必须刻进脑子里——

第一，**上云不是按一个按钮**。它是一场 9-18 个月的持久战，不是一次闪击战。

第二，**每一步都有自己的坑**。容器化不是微服务化、DB 不该上 K8s、对象存储不是文件系统、Serverless 不省钱、K8s 不解决 DB 问题——每一个反常识金句都是真金白银换来的。

第三，**没有最好的档口模式，只有最贴你业务的档口模式**。Docker / K8s / Serverless 三选一不是看谁更牛，是看你的业务是稳态、是脉冲、还是事件触发。

最后，我把上一节那句最扎心的话再敲一遍——

**上云这件事，99% 的公司都走错了第一步——他们以为"上云"是按一个按钮，结果按完发现，自己家厨房的灶具根本塞不进云上的铁盒子。**

真正的上云，是分三步：先把菜装进铁盒子，再把食材搬去云上冷库，最后让铁盒子自己会搬家。

三步走完，你才是真的把厨房开成了外卖档口——**菜还是那个菜，灶还是那个灶，但烧菜的人不一样了，吃菜的人也不一样了。**

如果你只能带走一句话，我希望是这句——

**上云这件事，不是按一个按钮，是开一家新厨房。按钮按错了可以退，厨房开错了要拆三个月。**

所以慢慢来，别抢这三步。
