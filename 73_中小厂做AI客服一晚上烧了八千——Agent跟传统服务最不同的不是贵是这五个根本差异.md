# 中小厂做 AI 客服一晚上烧了八千——Agent 跟传统服务最不同的不是贵,是这五个根本差异

> Agent 跟传统服务最不同的不是贵,是这五个根本差异。

你有没有这种感觉——

AI Agent 概念火了大半年,朋友圈里全是"我用 Agent 替我干了 XX 个小时",GitHub 上 Agent 框架一个接一个冒出来,身边朋友聊起来都说"未来 Agent 会取代 SaaS"。

听起来挺激动人心。

可你自己一上手,做完一个 demo,接进真实业务,然后账单飘过来——

你盯着那行数字,愣了 5 秒。

不是"贵"这一个字能形容。

是你发现自己过去十年做分布式服务、做 SaaS、做 API 网关那一整套算账方式,放在 Agent 身上**完全对不上**。

这篇文章想跟你聊的就是这件事——Agent 跟传统服务最不同的,不是"贵",是**这 5 个根本差异**。每一个差异都对应一组真实的账单代价。你以为你在做 SaaS,其实你做的是**私厨**。你以为你在卖自助餐,其实每一桌都得现做。

下面这个故事,你会一直读到。

**一、上线第 47 天凌晨 4 点,某 AI 客服创业公司被叫醒。**

有一家 AI 客服创业公司,我管它叫"小 K"。

他们干的事情不复杂——给中小卖家做 AI 客服,主打"7×24 小时不睡觉、不会发脾气、随叫随到"。上线前 CTO 算账:接 1000 个付费客户,客单价 99 元/月,一年收入 120 万,扣掉服务器和 token,怎么算都赚钱。

上线 47 天,凌晨 4 点,账单预警把他从床上震起来。

那晚发生了什么?

某个用户在平台上搞了一件事——脚本开 67 个账号,每个账号开多轮对话,每轮对话让 AI 把"为了更好回答你需要我提供哪些信息"这种诱导式提示重复几十遍,加 tool call 把每个会话撑到接近 8K context,会话一次结束——下一轮又开。

后台累计:**67 个账号 × 单会话平均 2000 token × 500 次调用 = 6700 万 tokens**。

按 OpenAI GPT-5.5 标准价(input $5/M, output $30/M,input 和 output 大致 1:1 平均 $17.5/M),6700 万 tokens 折合 **$1,173 ≈ 8000 元人民币**——一晚上烧光。

OpenAI 的账单在凌晨 4 点跳到 **8000 元人民币**。

第二天 CTO 醒来一看,整月预算 5 万,**一晚烧掉月度预算的 16%**。

> 这个数字拆开看:一晚被刷 8000 元(按 GPT-5.5 $17.5/M 平均单价算约 $1172),听着不算大,真正吓人的是**这种失控如果月度发生 1-2 次**,年度失控成本就是 **10-19 万元**——占年度收入 120 万的 8-16%,直接把毛利吃掉一半。

他后来在复盘会上说了一句话,让所有人沉默——

**"我以为我做的是传统 SaaS,其实我做的是私厨。每一道菜都是定制,每一份食材都要真付钱。"**

这个场景跟前面聊过的"被刷了一夜 token 欠了几十万"那期有点像,但本质上不是一回事——那一篇讲的是**怎么挡**,这一篇讲的是**为什么传统那套算账方式在 Agent 身上根本对不上**。

那 8000 元最后怎么处理的?

复盘会上,CFO 把财务模型跑了一遍——按当晚那种调用强度,假设不是被一个用户集中刷,而是 1000 个付费客户**正常按 30% 月活率用**:
- 单用户日均 50 次调用 × 单会话平均 2000 token × 30 天 = **9000 万 tokens / 月**
- 按 GPT-5.5 平均 $17.5/M → **$15,750 ≈ 11 万元 / 月**
- 年度累计 **130 万元 token 成本**,几乎吃掉全部 120 万年收入

当时公司账上只有 12 个月的跑道,月度毛利几乎为零,**任何失控事件都会让账本直接归零**。

换句话说——失控一次(单晚 8000)不可怕,可怕的是**默认状态下月度账单天然就到 11 万,留给你试错的空间是零**。

CTO 那一刻才意识到,**他算的那套账从根上就错了**。

这中间还有一个细节——小 K 团队一开始**根本不知道是哪个用户刷的**。

为什么?因为他们没接任何 token 用量监控。Agent 跑在云上,前端只看到对话流,后端只看 GPU 利用率,**真正的钱花在哪个用户身上,没人知道**。

等到账单飘过来,他们才发现这一晚上的调用量是过去一个月的 3 倍——但已经晚了。

**他算的不是账,是估值。**

很多中小厂做 Agent 之前,拿的财务模型是 LTV/CAC、毛利率、回收周期——这些数字全部基于"用户越多、ARPU 越高、边际成本越低"这条 SaaS 基本假设。

但 Agent 时代,**不是用户越多越便宜,而是用户越多越贵**。

你拿 LTV/CAC 算 Agent 的财务,**每一行数字都是反的**。

你写不出这种财务模型,投资人也写不出,大家都在**假装这套公式还成立**。

所以小 K 的真正问题不是那 8000 元,是**他根本没办法告诉董事会,这一年下来到底会烧多少钱**。

不可预测,本身就是最大的成本。

你想想,如果失控事件每个月发生一两次,**月度账单就不仅仅是 11 万了,会再叠加失控成本 1.6-3.2 万,变成 13-14 万**——离月度预算 5 万,已经远一倍。

这是后话,先把它放下,回到核心问题——

为什么 Agent 跟传统服务的算账方式对不上?

不是你的算账公式写错了。是你**算账的对象就不一样**。

**对象不一样,公式一定不一样。**

**二、Agent 不是换了皮的传统服务,它是另一种东西。**

我们这行有个老毛病——看到新概念,第一反应是把它装进自己熟悉的老框框里。

Agent 火起来的时候,第一波讨论全是"这就是个新分布式服务""这是 LLM 加了一层 API 网关""这是带状态的服务"。听起来都对,听起来也都没用。

为什么?

**因为 Agent 不是一个"加了点东西的传统服务",而是另一种根本范式。**

把 LLM 接进你的 API 网关,**不是让传统服务升级了,而是让传统服务的算账方式全部失效了**。你以为是在升级,其实是在换赛道;你以为是做加法,其实是在做除法——**不是用旧模型算新成本,而是用新模型算旧成本**。

因为你拿这套老框框去算成本、去排扩容、做监控、做告警,**每一个动作都会出大错**。

传统服务是**自动售货机**——你投币、我掉货、拿了就走,中间不需要记忆,每次交易相互独立,机器死了不影响其他机器,加一台机器就多服务一份用户。

Agent 是**私厨**——一个厨师专门服务一桌客人,从点单到选材到烹饪到摆盘全程盯着,得记住你今天不吃辣、上次说喜欢七分熟、对花生过敏,下次你再来他得全记得——他停一秒钟,后面 30 分钟的活就重做。

![自动售货机(传统服务)vs 私厨(Agent)对比](assets/73_中小厂做AI客服一晚上烧了八千——Agent跟传统服务最不同的不是贵是这五个根本差异/02-vending-machine-vs-private-chef.png)

**自动售货机不需要记住你,私厨必须记住你。这条差别,几乎所有 Agent 成本问题都从这里冒出来。**

私厨什么意思?展开讲三点。

这里有个反直觉的判断——很多人觉得 Agent 跟传统服务的差别是"AI 加进来了",**不是 AI 加进来了,是 AI 把整套服务范式换了**。

传统服务是"我卖一个功能,你用就行",功能跟功能之间是相互独立的,数据库里加一列加一行,加完了不增加运行成本。

Agent 不是——Agent 是"我陪一个用户走一段",陪的过程里所有状态都在叠加,每多陪一秒就多一秒的成本。**不是加一个 AI 模块,是把整个服务模型从"卖功能"换成"陪人"。**

第一,**它必须为"你这一桌"一直在线**——你不是投完币就走,你是坐下来等菜上桌,厨师在你身上挂着心思,这一桌结束前他都"在线"。**Agent 也是**,一个会话挂起后,你不知道用户什么时候回来,但会话状态不能丢、上下文不能释放、工具的中间结果不能清。

第二,**它每一道都得现做**——自助餐是把菜提前做好放保温台上,谁来了都能拿。私厨不一样,你点什么他做什么,你这次点的和上次点的不是一回事。**Agent 也是**,同一个用户连续来 3 次,3 次对话的内容可能完全不同,因为他的提问、上下文、工具调用序列都不同——你不能用"标准化输出"省时间。

第三,**它的食材全是你掏钱**——自助餐的成本结构是"我提前按估算批量做,做多了浪费一点、做少了少卖一点",亏赚都摊在全天所有客人头上。私厨的成本结构是"你这一桌用了多少食材,我就收多少食材的钱",加一份牛排多收一份钱,加一只虾多收一只虾。**Agent 也是**,多一轮对话多烧一次 token,多一次 tool call 多烧一次 API 费,多一段上下文多烧一次缓存——**每一笔都是单独计费的**。

这三条叠在一起——私厨永远比自助餐**单价贵**。

传统服务是**自助餐逻辑**:你提前做一批,人来就拿,人走就关门,中午的高峰和晚上的低谷之间还能歇一歇。

Agent 是**私厨逻辑**:你得 24 小时待命,每一桌都从点单开始,做完上一桌不知道下一桌什么时候来。

自助餐还有一个隐性好处——**客人越多,单个食材成本越低**。你做 100 份鱼香肉丝和做 1000 份,食材采购价不一样、灶台能耗不一样,边际成本往下走。

私厨不是——你给 100 桌做 100 份鱼香肉丝,每份都要单独切、单独炒、单独装盘,食材采购没有"批量折扣",因为每个客户的口味不一样,**你的采购永远是 N 份小单,不是 1 份大单**。

放到 Agent 上,**同样的剧本**——1000 个 Agent 实例 vs 1 个 Agent 服务 1000 个用户,**前者烧的是 N 倍硬件 + N 倍 token,后者连会话状态都串了**。

这意味着 Agent 没办法走"先聚合再分发"的传统 SaaS 路线。**不是 SaaS 模式的问题,是 Agent 模式天然有 1 个用户 1 份硬件的硬开销。**

**三、把"AI 太贵"剥成 5 个根本差异。**

"AI 太贵"——这是所有中小厂做 Agent 的人说的第一句话。

但"贵"是个结果。**真正要回答的是:贵在哪?为什么贵?每一项贵到什么程度?**

我把它剥成 5 个根本差异,每一个都对应一组可观测的账单代价。

**差异一:状态——Agent 是叠加态,不是确定态。**

传统服务要么无状态(请求来、请求走,中间啥也不留),要么冷热持久化够用(把会话塞进 Redis、过期清掉)。

Agent 不行。Agent 的"状态"是**叠加态**——同一时刻它身上挂着四五样东西:

- 状态机的当前节点
- 这一轮的对话历史
- 工具调用的最新结果
- LLM 自己 hold 的上下文窗口
- 长期记忆里跟你相关的部分

这五样东西**同时存在**,而且会一直叠加,**直到这桌"饭"吃完**。

![5 个根本差异对照表(状态 / 响应 / 上下文 / 存储 / 规模)](assets/73_中小厂做AI客服一晚上烧了八千——Agent跟传统服务最不同的不是贵是这五个根本差异/03-five-differences.png)

Anthropic 把这件事计费的方式特别直白——官方公开价格里有一项叫"Managed Agents",**按活动会话按小时单独计费,0.08 美元/小时**。

0.08 美元看起来不多。

但你换算一下:1000 个活跃会话,跑 24 小时,**单天就是 1920 美元**,一个月约 5.7 万美元,**40 万人民币**。

这不是 token,不是模型调用,**就是"你的会话在我这挂着,我按时间收钱"**。

传统服务从来不按"会话在线时长"收钱——它只按"请求次数"或者"实例数"收钱。Agent 多了一根时间轴,你不知道用户什么时候会回来,你就**关不掉这个计费**。

更扎心的是——**用户的"在线"不等于"活跃"**。

传统服务的"在线"是指这台机器活着,它不费什么资源,你把它放着不调用,代价只是几瓦电。

Agent 的"在线"是指这个会话状态还活着,代价是上下文窗口还在被 hold、工具 schema 还在被加载、系统提示还在被缓存——你把它放着不调用,**钱也在跑**。

> **Agent 是叠加态,不是确定态——同一时刻它有多个可能状态,传统服务从来不这么累。**

**差异二:响应时间——ms 对 s,GPU 服务用户数少 1-2 个数量级。**

传统 REST API 的 p99 latency 是 50-200 毫秒,数据库查询、缓存读、消息投递,基本都在这个区间。

Agent 一次响应端到端 1-10 秒——含 LLM 推理 + 工具调用 + 多轮思考 + 流式输出。

这意味着什么?

意味着**同一块 GPU,服务用户数可能要少 1-2 个数量级**。

Character.AI 公开过他们的算式:**1 亿日活 × 每用户每天 1 小时 × 服务成本 0.01 美元/小时 = 年服务成本 3.65 亿美元**。

折算下来:**3.65 美元/日活/年**。

这是行业头部、一线优化过的单位成本。

Character.AI 用的还是自家推理栈 + AMD MI300X 大规模替换 + 50% 成本下降之后的数字。如果你是中小厂,直接买云上 GPU 跑,**这个数字还得往上加**。

传统服务是"我跑一次毫秒级,所以一块 GPU 一秒能服务几十个用户"。Agent 是"我跑一次几秒到十几秒,所以一块 GPU 一秒只能服务零点几个用户"。

**同一块 GPU,服务的并发用户数相差 50-100 倍**。

更隐蔽的是——这块 GPU 不能共享。

传统服务的"忙"和"闲"是可以在用户之间平滑过渡的——A 请求处理完立刻接 B,机器一直在线就行。

Agent 的"忙"和"闲"是 GPU 被一个会话独占——A 在等 LLM 推理的时候,B 没法用这块 GPU,**B 必须等**或者**B 占用另一块 GPU**。

你买的不是一台"高效跑服务的服务器",你买的是"一个会话独占的一块 GPU"。

> **传统服务 ms 响应,Agent s 响应——同一个 GPU,Agent 服务的用户数可能是传统服务的 1/100。**

**差异三:上下文归属——app 收不了口,LLM 收口,涨完烧完。**

传统服务里,请求进来、上下文关闭,是你的应用层说了算。你能决定"这次请求不存任何上下文,下次请求重新开始"。

Agent 里,**上下文是 LLM 收口的,你写代码管不了**。

每一次轮对话,你都得把历史、工具结果、系统提示塞回去,塞进 LLM 的窗口。LLM 是窗口多大就是多大,你只能干瞪眼看着钱烧。

更狠的是——主流厂商已经在价格上"按上下文长度分段收费"了。OpenAI GPT-5.5 公开规定:**输入超过 27.2 万 tokens 后,input 价格 ×2,output 价格 ×1.5**。

整段会话,不分你超出多少,**全段提价**。

你的上下文越长,单价越高;上下文越长,下一轮还得再塞回去;下一轮塞得越多,超阈值的概率越大——一个正反馈循环,越聊越贵。

你做 SaaS 的时候,数据库里加一列、加个索引就完了,加完了所有人共用。

你做 Agent 的时候,每一轮的上下文都要钱,**钱数还跟你"今天跟用户聊了多少"有关**——聊得越深,陪得越久,烧得越多。

**传统服务加列免费,Agent 加上下文烧钱。**

更扎心的是——这钱你看不见。

传统服务你买了一台数据库,数据库的账单是固定的、可预测的,月底一算,**多少钱就多少钱**。

Agent 的上下文账单是浮动的、不可预测的——今天这个用户聊了 1K,明天聊了 5K,后天聊了 8K,你月底算,**账单的方差大到你不敢相信**。

**不是钱变多了,是钱变得不可预测了。**

更讨厌的是——这钱你控制不了。

传统服务你优化 SQL、加索引、改查询条件,**优化效果能写进代码 review,能写进 SLA**。

Agent 上下文优化——你截短、加摘要、用 vector 召回,**效果取决于 LLM 怎么用这些上下文**。你截短了,L1 时还很准,L2 时漏信息;你用 vector 召回,召回了错的,模型还一本正经胡说。你调优 3 天,账单从 100 降到 80,**省的那 20 是毛毛雨**。

这里有个更深的问题——**你优化的是 LLM 的输入,但 LLM 的输入不是你的业务边界**。

你做 SaaS,数据库里加一列,这是你的业务边界,优化效果你能写进代码。

你做 Agent,上下文是 LLM 的输入,但 LLM 的输入由"用户怎么问 + 工具怎么回 + 之前聊过什么"决定,这三件事你都不能完全控制。**不是你的边界,也不是你能优化的边界——你优化的是"LLM 愿不愿意省着点用",这不是工程问题,是哲学问题。**

> **上下文不是 app 收口,是 LLM 收口——你写代码管不了上下文大小,是 LLM 在 hold,你只能干瞪眼看着钱烧。**

**差异四:存储——不是单一栈,是四层叠加。**

传统服务的存储很简单——DB / Redis / 对象存储,三选一,或者三选三。

Agent 应用至少要叠 4 层存储:

- 向量数据库(Qdrant / Milvus):RAG 检索用
- 关系数据库(Postgres / MySQL):用户数据、业务数据
- 缓存(Redis):会话状态、工具调用中间结果
- 对象存储(OSS / S3):附件、长期记忆快照

每一层都要钱。每一层都要监控。每一层都要备份、容灾、扩容。

Anthropic 公开的官方价格表里,"Managed Agents"是把"会话存储 + 工具执行 + 上下文管理"打包成一整个计费项——**意味着厂商默认这四层存储就是 Agent 的标准成本结构**,不是临时拼出来的。

你做 SaaS 的时候,接一个 Postgres 就够,接一个 Redis 就更快。

你做 Agent 的时候,**四层存储一起上,每一层都有它自己的扩容逻辑、计费曲线、故障模式**。

更狠的是——这四层之间还有**数据一致性**问题。

向量数据库新增了一条,Postgres 那边用户表也得改;Redis 改了,持久化到对象存储又是个异步流程。**四层叠在一起,一致性 bug 翻倍**,你能把"为什么这个 agent 答错了"从"调一个接口"变成"追四个存储系统"。

运维团队得多配 1-2 个人,这一项一年又是 50-80 万。

**单栈免费,多栈烧钱。**

> **Agent 的存储是四层叠加的,不是单栈——每一栈都烧钱,加在一起就是指数级的运维复杂度。**

**差异五:规模关系反转——用户越多,边际越贵。**

传统服务是"用户越多,边际成本越摊薄"——这是 SaaS 的命根子。一个用户用、十万用户用,基础设施成本不是十万倍,是十倍甚至更少。

Agent 不是。

Agent 是"**上下文越大,开销越大**"——而且这个"大"不是线性增长,是**叠加 attention 计算 + KV cache 重建 + prefill 与 decode 成本分离**多个非线性系数的乘积。

我用 OpenAI GPT-5.5 公开价格做了个测算(完整数据附在调研报告里),假设单会话平均 token 数随上下文窗口增长:

- 8K 上下文,单会话平均 2K tokens,账单基准 1x
- 32K 上下文,单会话平均 8K tokens,账单基准约 4x
- 128K 上下文,单会话平均 32K tokens,账单基准约 16x
- 200K 上下文,单会话平均 50K tokens,账单基准约 **25-30x**

**128K 的会话,单会话账单是 8K 的 16-30 倍**。

![上下文 8K / 32K / 128K / 200K 真实账单倍数曲线](assets/73_中小厂做AI客服一晚上烧了八千——Agent跟传统服务最不同的不是贵是这五个根本差异/04-context-length-curve.png)

这意味着——你为了"用户聊得更爽"把上下文从 8K 升到 128K,体验提升了,但账单也涨了 16-30 倍。

你做 SaaS 的时候,边际成本越来越低,你可以拿"免费增值 + 增值订阅"那一套玩法——免费用户来再多,边际成本几乎为零。

你做 Agent 的时候,**每个用户的成本曲线不是下降,是上升**。你做"免费增值",免费用户用得越久你亏得越多。

**传统服务是规模效应——用户越多越便宜。**

**Agent 是规模反效应——用户用得越深,你烧得越狠。**

这是 Agent 时代最反常识的一条——不是"规模带来效率",而是"规模带来烧钱"。

你拿 SaaS 的算账方式算 Agent,**每一行数字都是反的**。

更深一层——**规模反效应的本质是"上下文"这个变量接入了成本函数**。

传统服务的成本函数只看两个变量:用户数和调用量。**用户数和调用量都是线性的,可以被摊薄**。

Agent 的成本函数至少有三个变量:用户数、调用量、上下文长度。**上下文长度不是线性的,是随用户使用深度爆炸的**。你接入了一个非线性变量到成本函数里,你的整套成本模型就废了。

为什么 SaaS 那套算账方式算 Agent 不对?就是因为传统算账假设了**所有变量都是线性的**。Agent 把"上下文长度"这个非线性变量拉进来,你的线性模型全部失真。**不是 Agent 太贵,是你的模型太简单。**

> **传统服务用户越多越摊薄,Agent 上下文越大越烧钱——AI 时代的"规模效应"不存在,只存在"规模反效应"。**

把 5 个差异串起来——

**叠加态、秒级响应、LLM 收口、多栈叠加、规模反转。**

不是"贵"一个词能解释的。**是 5 件事一起,在每一个会话上同时发力。**

**四、最致命硬算账:1C1G × 1000 用户 = 30 多万硬件/年。**

差异讲完了,该算总账了。

我做了一个真实的、保守的、面向中小厂的算账——硬件先,token 后。

**4.1 硬件算账。阿里云/腾讯云价格以官方页为准;数据来源 DigitalOcean 官方页 + 腾讯云主站公开价 + AWS Lightsail/EC2 公开价)

**最便宜的"1C1G + 经济型"路线,1000 个实例一年 ≈ 30-50 万元**。

这只是硬件,**不算 token**。

![100 / 1000 / 10000 用户真实账单对比(摊薄 vs 线性)](assets/73_中小厂做AI客服一晚上烧了八千——Agent跟传统服务最不同的不是贵是这五个根本差异/05-100-vs-1000-vs-10000-bills.png)

如果你不满足 1C1G,要换成 4C8G(更现实)——年费 **360 万**。8C16G **600-840 万**。带 GPU 的实例 **1800-3000 万**。

这是纯硬件,还没加 token、还没加带宽、还没加数据库、还没加监控、还没加对象存储、还没加运维人天。

**4.2 Token 算账。**

硬件之外,token 是 Agent 的另一条命。

按 OpenAI GPT-5.5 标准价(input $5/M,output $30/M):

假设 1000 用户 × 每人每天 10 轮对话 × 半年累计,单轮 4000 tokens,Input:Output = 2:1(输入占大头,因为客服场景每轮要把"商品信息 + 用户问题 + 历史对话"全部塞进上下文)。

1000 用户 × 10 轮 × 4000 tokens × 180 天 = **72 亿 tokens/年**。

按 input $5/M × 2 + output $30/M × 1,平均单价 **(5×2 + 30×1)/3 ≈ $13.3/M**。

72 亿 tokens × $13.3/M = **$96,000 ≈ 67 万人民币 / 年**(单是 OpenAI GPT-5.5 标准价,如果换 Sonnet 5、Opus,这个数字还得往上翻)。

如果换国产模型,这个数字能砍到一半——但你换来的是响应更慢、效果更差、客服回答客户问题容易出岔子。**账单省下来的,被客服投诉吃掉**。

**4.3 总账。**

硬件 + token = **80-130 万元/年**。

加上数据库、Redis、对象存储、监控、带宽、人天,**120-150 万元/年** 是个保守底线。

**4.4 跟中国 SaaS 用户付费意愿比一比。**

中国 SaaS 用户的付费意愿典型区间:

- 微中小企业(20-100 人):**月费 50-300 元 / 用户**
- 中型企业(100-500 人):月费 200-800 元 / 用户

1000 个微中小企业客户 × 100 元/月 = **年收入 120 万**。

——**跟你的硬件+token 总账基本打平**。

如果算上研发、销售、客户成功,**净利率直接打穿**。

这里有个反常识的判断——中小厂做 Agent 失败,**不是产品失败,是商业模型失败**。

传统 SaaS 的商业模型是"我卖功能,你按月付费,功能越多收得越多"。这条模型的前提是——**每个新用户的边际成本接近零**。

Agent 不是——每个新用户的边际成本是 100-150 元/月(1C1G 硬件摊薄 + token)。**不是边际成本接近零,是边际成本是 100 元**。

你的用户付你 99 元/月,你的边际成本 100 元,**每个新用户你都在亏 1 元**。

1000 个用户你一年亏 12000 元,**10000 个用户你一年亏 12 万元**——不是你做大了,是你做得越大亏得越多。

**不是用户越多越赚钱,是用户越多越赔钱。**

> **中小厂做 Agent,用户不愿意为你的硬件付费,也不愿意为你的 token 付费——他们愿意为"功能"付 50-300 元/月,没人愿意为"你的账单"付钱。**

这就是为什么我开头说——你不是在烧 GPU,你是在烧**"用户不愿意掏钱的那部分成本"**。

你拿什么去接?**你拿融资去接,接完融不到下一轮,就关张**。

**4.5 跟大厂比,中小厂还差在哪一层。**

大厂做 Agent,**他们有云、有模型、有补贴、有规模摊薄**——他们烧的那点 GPU 成本,放在他们云收入、模型收入、广告收入里,**可以忽略不计**。

中小厂没有这些。

你以为大厂发 Agent 是给你打样,其实大厂发 Agent 是给你看他们的水管多便宜。

> **中小厂做 Agent 不是输给大厂的功能,是输给大厂的"赔得起"——他们有云有补贴,你没有。**

这是结构性问题,不是努力不努力的问题。

**4.6 一句最扎心的话。**

我看到很多中小厂的创始人读完这些数字会问:**"那我不接那么多付费客户,接 100 个、200 个呢?"**

好问题。100 个付费客户 × 100 元/月 × 12 个月 = **12 万元年收入**。

你这 12 万元够不够支付一个 Agent 实例一个月的硬件 + token?

——**不够。**

——一个 Agent 实例一年的成本就超过 12 万元,你的 100 个客户**连养活一个会话的钱都不够**。

除非你能把单用户 ARPU 做到 500 元/月以上——那是中型 SaaS 的客单价,不是微中小 SaaS 的。

**中小厂做 Agent,不是技术问题,是商业模型问题。你想清楚怎么收钱了吗?**

再补一刀——你接了 100 个付费客户,还有 1000 个免费用户在用。

免费用户按"7×24 小时免费试用"跑,**每个免费用户都在烧你 360-1500 元/年的硬件成本**。

1000 个免费用户 × 1000 元/年 = **100 万元/年**——这笔钱你一分钱都没收进来。

**不是免费用户越多越好,是每个免费用户都在烧你的融资款。**

**五、上下文陷阱——为什么"上下文越大越好"是假命题。**

好,上面 5 个差异讲完了,你大概明白了 Agent 为什么贵。

但还有一类问题没讲——**你以为是优化,其实是陷阱**。

最典型的就是"上下文越大越好"。

**5.1 真实账单对比。**

我用 OpenAI GPT-5.5 公开价格做了一个详细的对比(数据来自 OpenAI 官方 API 文档):

| 上下文长度 | 单会话平均 tokens | 1M input 单价 | 1M output 单价 | 跟 8K 对比 |
|---|---|---|---|---|
| 8K | ~2K | $5 | $30 | **1x** |
| 32K | ~8K | $5 | $30 | ~4x |
| 128K | ~32K | $5(>272K 时 2x) | $30 | **~16x** |
| 200K | ~50K | $5 或 $8(>272K ×2) | $30 或 $36(×1.2) | **~25-30x** |

注意——**不只是 input token 涨**,output 也涨,长上下文的 attention 计算也涨,KV cache 重建也涨,**多个非线性系数叠加**。

你为了"用户聊得更爽"把上下文从 8K 升到 128K,**体验提升了,账单涨了 16-30 倍**。

这里有一个反直觉的判断——很多团队把"上下文从 8K 升到 128K"当作升级,当作产品力提升。**不是升级,是陷阱。**

128K 不是"产品力",**不是 LLM 的能力,是 LLM 的成本曲线**。你升级的不是体验,是烧钱速度。

更阴险的是——你升上去容易,降下来难。

用户用惯了 128K 的体验,你突然降回 32K,他会立刻感知到"AI 变笨了",客户投诉量瞬间上来。**不是产品功能,是产品承诺**——你承诺过的体验,撤不回来。

**5.2 中小厂的认知误区。**

很多中小厂的 CTO 一上来就选 128K 模型,理由是"我们要给用户最好的体验"。

这是**最贵的选择**。

128K 听起来美好——理论上能塞下整本《三国演义》,能记住用户过去 100 轮对话。

但实际上:

- 用户真实用到的对话深度,90% 在 4K 以内
- 长上下文的"边际体验提升"远低于"边际账单提升"
- 你为了 10% 的体验,多付 16-30 倍的钱

> **128K 模型听起来美好,实际每会话烧的钱是 8K 的 16-30 倍——你没有那个本钱去拼"长上下文体验"。**

**5.3 该怎么选。**

选上下文大小的标准应该是——**用户真实使用深度 × 单价曲线**。

- 内部知识库问答,用户问完就走,8K 足够
- 客服场景,用户追问 3-5 轮,32K 是合理的上限
- 长文档分析、代码助手、复杂推理,才考虑 128K+
- 200K+ 几乎都是"演示用",真实业务别碰

**不是模型越大越好,是账单跟你业务匹配度越高越好。**

**5.4 上下文管理不是优化,是生存问题。**

很多团队把上下文管理当作"性能优化"——能优化就优化,优化不了就先这样。

**错。**

上下文管理对中小厂是**生存问题**——你不管住上下文,Token 替你管住嘴,账单替你关公司。

具体怎么做?有三条硬纪律。

第一,**截断要硬**——超过阈值的部分强制清掉,不管有用没用,不能"我再试试看 LLM 能不能用上"。

第二,**摘要要稳**——长对话定期做摘要,摘要本身也要算进 token 成本,但比直接 hold 整段对话便宜。

第三,**工具结果要缓存**——同一个工具的同一个查询不要每次都重跑,缓存起来复用,既省 token 又省 API 钱。

**这三件事不是建议,是活下去的最低底线。**

**六、三个反常识金句 + 一个伏笔。**

聊到这里,有几个反常识的金句我想单独拎出来,让你忘掉也能带走。

**6.1 三个反常识金句。**

**第一句:你不是在付 AI 的钱,你是在付"GPU 没关机的钱 + Token 没停流的钱 + 上下文没截断的钱"。**

传统服务你下线服务,账单停了。Agent 不一定——会话还挂着、token 还在跑、上下文还在烧。**Agent 的"关机"成本远高于"开机"成本**。

**第二句:不算清每用户成本 + 每上下文长度成本就上线 Agent,跟不查每道菜成本 + 每份食材成本就开饭馆一样死得快。**

传统 SaaS 上线前你算的是"获客成本 / LTV / 转化率",Agent 上线前你必须算"每用户每年烧多少钱 + 每千 token 烧多少钱"。**算账口径都不对,后面所有数字都是错的**。

**第三句:Agent 给用户体验的是"多对一贴心",给中小厂账单的是"每单都重付"——这是同一件事的两面。**

用户感受到的"贴心",就是账单上的"贵"。你不能既想要贴心,又想要便宜。**至少在目前的 Agent 范式下,这两件事是矛盾的**。

**6.2 一个伏笔(为下一期铺垫)。**

你还记得小 K 吗?

那家被叫醒的 AI 客服创业公司。

复盘会上他们想明白了——原来 Agent 跟传统服务根本不是一回事,原来自己过去十年的 SaaS 经验套不上,原来"AI 太贵"背后是 5 个差异一起发力。

**这场复盘会上小 K 决定了三件事,先动手再说。**

你看 6 个月后,他会做了——账单从"11 万起步随时可能被失控事件叠加到 13-14 万"变成"4.5 万可预测",用户从 1000 涨到 5000,单用户成本从 110 元降到 9 元。

**他活了一时**。

但故事没完——小 K 后来又开始学某一套看起来很光鲜的新玩法,**这一回先不展开**。

下一回再说。

你可能会好奇——他学的到底是什么?看着像大厂那套?看着像什么新框架?

我先卖个关子。**不是不告诉你,是说早了下一期开头就没意思了。**

但有件事可以提前剧透——他学的这套玩法,看着光鲜,做起来才知道,**里面藏着大厂才赔得起的另一个陷阱**。

至于具体是什么陷阱,小 K 怎么又翻了一次车,下一期我们再拆。

**七、金句收束 + 我们下篇见。**

写到这,该收了。

回到开头那个问题——Agent 跟传统服务最不同的,是什么?

不是贵。**是 5 个根本差异**——状态是叠加态、响应是 s 级、上下文是 LLM 收口的、存储是四栈叠加的、规模是反效应的。

这 5 个差异叠在一起,决定了"每多一个用户,每个用户也都更贵"。

把 5 个差异翻译成账单:

- 1000 用户,1C1G,一年硬件 **30-50 万**
- 加上 token,**80-130 万**
- 加上数据库、Redis、对象存储、监控、人天,**120-150 万**
- 中国 SaaS 用户付费意愿,一年 120 万

**账单打平收入,你拿融资接,接不上下轮就关张**。

传统 SaaS 那套算账方式,在 Agent 身上**完全对不上**。

你以为你在做私厨——其实你以为你在做自助餐。

自助餐亏钱有人埋,私厨亏钱你自己扛。

> **我们下篇见。**

下篇卷袖子干活,讲三招——**agent 运行环境下沉浏览器 / token 与 agent 解耦 / 业务专注**,把私厨做出自助餐的便宜。

每一招都不是"省钱",是"换一种烧法"。每一招都解决本篇里说到的某一根隐性硬性成本。三招叠起来,小 K 才能从"11 万随时可能被失控事件叠加到 13-14 万"变成"4.5 万可预测"。

你看——自助餐和点菜制这事,不是非得二选一。**你可以做点菜制,但用自助餐的成本结构**。

这就是下一篇要讲的事。
