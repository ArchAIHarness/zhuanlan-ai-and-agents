# AI 推理成本不是按 token 计费那么简单——GPU 利用率 + 显存调度 + 批处理三道闸把电费降到 1/3

> 敢点火是勇气，能收住才是本事

![第1节配图](assets/66_AI推理成本不是按token计费那么简单——GPU利用率加显存调度加批处理三道闸把电费降到三分之一/01-cost-trend.png)

上一篇我们讲到，那家 AI 客服创业公司的 CTO 张工被凌晨 3 点的电话叫醒，月度 GPU 账单从 5 万烧到 25 万。这一篇接着讲——他拉团队开了三天闭门会，三道闸焊死之后，月度账单从 25 万降到了 8 万。

不是少了五道工序，也不是多买了更强的模型。三件事：把 GPU 利用率从 20% 提到 70%、把显存当旅馆一样分时复用、把零碎请求合并成整托货。算下来，节省 68%。

这一篇不是想讲"还有更便宜的云厂商"，也不是想讲"换个量化精度"。**这是把 AI 推理当成一个家庭电力系统来重新设计——谁供电、谁降峰、谁存电、谁削峰，每一步都对应着一个工程动作。** 设计好了，电费自然降；设计没做，再谈单价也是白谈。

经营一个家，听上去简单，做起来全是活儿。电表要装、漏电保护器要布、节能灯要换、分时家电要配、智能插座要排时间表、阶梯电价要算账。**AI 推理跟这一模一样——只不过把"家用"换成了"机房"，把"电表"换成了"GPU 集群"。** 道理从来都是那个道理：先设计，再使用；先跑通每一条回路，再谈功能。

## 一 反差开场——同一套业务从 25 万到 8 万的故事

![第2节配图](assets/66_AI推理成本不是按token计费那么简单——GPU利用率加显存调度加批处理三道闸把电费降到三分之一/02-gpu-utilization.png)

同一套客服业务，同一个开源模型，连 prompt 都没怎么改。从 25 万降到 8 万，怎么做到的？

那次三天闭门会，张工团队没换模型、没买新显卡、没把团队人数翻倍。他们做的，是把"GPU 这台电表"重新接了一遍——从根上做了一次电力调度。

**第一个数字：GPU 利用率从 23% 提到 71%。**

复盘当天的监控截图他们留了一份：原本 8 张 H100 跑在一个推理集群上，平均 5.7 张闲置。闲置 = 没干活 = 钱白烧。闭门会第一天，他们把所有推理改成连续批处理（continuous batching），把零碎请求拼成整托货发出去。**结果 8 张卡还是 8 张卡，但每张卡的"工作时间"从 23% 涨到 71%。** 这不是省电，是把同样的电用出 3 倍。

**第二个数字：KV cache 显存占用从 41 GB 砍到 9 GB。**

客服场景有个特点——"system prompt + 用户历史对话"长得很像。两万个用户问"查一下我的订单"，system prompt 大段相似，旧版本每一个请求都重新开一份 KV cache，相当于每个房间都重新粉刷一遍。闭门会第二天，他们接上 vLLM 的 PagedAttention，让 KV cache 像旅馆房间一样分时复用：谁住过、留下痕迹，下次来直接入住，不再重新装修。**显存节省近 80%，单卡并发从 16 涨到 64。**

**第三个数字：高峰期的平均延迟从 4 秒降到 0.8 秒。**

这个数字看上去是性能题，其实是电费题——延迟高了客户端会自动重试，重试又撞高峰，账单就被"放大器"翻上去。闭门会第三天，他们上线动态 padding + 异构请求合并 + 异步队列削峰，把零碎活塞进电梯、把不紧急的活推到夜里。**延迟降下来的那一刻，重试风暴也跟着停了。**

三个数字叠起来——25 万降到 8 万，省了 17 万、节省 68%。钱没少花，事也没少做，只是把每一笔 GPU 时长都用在刀刃上。

讲到这里你应该看明白了：**三道闸不是把 AI 训得更快，而是把 GPU 的每一秒钟都用在刀刃上。** 这也是这一篇的核心命题——把 AI 推理从"月光族"模式切到"精打细算的电力调度"模式。

你听过的比喻是"AI 推理就是电费"，但光知道电费不够，还得知道怎么调电。这一篇我们就拆开这三道闸，看看每道闸后面的工程动作长什么样。

张工团队那三天闭门会并不是一开始就达成共识。第一晚团队就吵翻了——一半人说要换模型（"qwen3.7-max 比我们现在的开源 7B 强，价格也差不多"），另一半人说要换云厂商（"AWS 在打折"）。最后 CTO 把所有提案按在桌上："不要去争论单价，单价再低也救不回 23% 的利用率。先把利用率抬上去，再聊别的。"这句话定了调子，第二晚才进入正题。

闭门会第二天结束的时候，张工给团队画了张"电力系统全图"——左边是 GPU 利用率（电表盘）、右边是显存调度（储电箱）、中间是批处理（错峰闸）。**这张图不是技术架构图，而是电费治理图。** 他在图的正中央写了一句话：**"AI 推理不是把模型调强，而是把系统的电调顺。"** 这句话后来被团队贴在会议室墙上，一直没撕下来。

## 二 第一道闸 GPU 利用率从 20% 提到 70%

先说第一道闸。这道闸是整个优化的"地基"——如果利用率只有 20%，后面两道闸做得再花哨也是白搭。

GPU 利用率上不去的原因，跟家用中央空调为什么"省电模式没用"一个道理——空调开到 24 度就停室外机，但你的房间还没凉下来；GPU 也是，请求来了才计算，请求没来就停。**问题是停的间隙也算钱。**

业内的"行业甜点"是把利用率推到 60% 以上。想推到 70%，需要三个工程动作配合——

**动作一：连续批处理（continuous batching）。**

传统的批处理是"攒够一波一起发"，攒的过程 GPU 就在空转。连续批处理反着来：**请求到了就进队、立刻发出去；前一条还在算、后一条已经塞进来；做完哪条先退哪条。** 业内有个数字：Continuous Batching 比 Static Batching 吞吐量高 23 倍。这个数字听上去吓人，背后道理不复杂——整托运货比散装发货效率高 23 倍，GPU 也在拼这件事。

Anyscale 在 2023 年公开过这组数据，2026 年再用同样思路，H100 单卡扛 5000 tok/s 输出不是问题。**这是为什么你的推理服务上线第一天该先问"利用率多少"，不是先问"模型强不强"。** 模型再强，利用率不到 30%，账单照样爆炸。

**动作二：PagedAttention。**

这是 vLLM 框架的招牌创新，2024 年拿下了 SOSP 业界顶会的最佳论文。**它解决的是 KV cache 的"内存碎片"问题。**

把显存想象成饭店的包间：传统做法每来一桌客人都开一间新包间，包间没满就换桌台，结果 50% 的包间都半空着。PagedAttention 的做法是把每个包间切成"小格子"（page），谁用哪一格就用哪一格，散桌也能拼出来一张完整餐台。**显存利用率从 50% 提到 90%，等于同样的一张卡，多装一倍的客人。**

vLLM 2024-2025 的版本迭代里，这套思路已经标准化——prefix caching、disaggregated prefill/decode、动态批大小，全是基于 PagedAttention 这套分页思路延伸出来的。HuggingFace 的 TGI 在 2026 年 3 月已经归档（maintenance mode），现在社区默认是 vLLM。**TGI 退场了，vLLM 一家独大——这不是技术决定，是社区生态决定。**

**动作三：动态批大小。**

传统批处理是固定大小的——batch size 写 32，就是 32。请求不够 32 就空转；请求超过 32 就排队。**动态批大小让系统自己调节——高峰期往上顶，低谷期往下压，保证 GPU 始终在干活状态。**

动态批大小听起来简单，做起来不容易：你得让模型本身支持变长输入、让推理服务支持异步排队、让监控系统实时反馈利用率。**这三件事都做完了，利用率才上得去。**

讲到这里你可能要问：利用率从 20% 提到 70%，是不是直接优化 70% 就完事？不是。利用率是个"靠前"的指标，**真正的账单节省要看"按 token 折算后的单位成本"**——同样一张卡，70% 利用率下的单 token 成本只有 20% 利用率下的 1/3.5。GPU 利用率 20% 不是省钱，是给云厂商捐款。这句话值得你贴在墙上。

这里要补一句很多团队绕进去的常识：**利用率不是越高越好，而是越稳越好。** 利用率 90% 看起来很美，但意味着高峰期已经在排队、GPU 已经没有缓冲；遇到一次突增流量就是全线 503。所以工程师追求的"理想利用率"通常是 60%~75%，剩下的留给"应对突发"。**不是把 GPU 用尽，而是把 GPU 用稳。**

最后还要提一句：利用率到了 70% 之后还想再提，不是工程动作能解决的，是工作负载（workload）的事。**找不到合适的工作负载来填满 GPU，再优化也是空转。** 这一层留给业务方去解决，工程师先确保技术可以扛。

值得专门提一句的是 vLLM 之外的另一个选择——TGI 和 TensorRT-LLM。2026 年 7 月时点的格局是：vLLM 拿走了"开源社区 + 云厂商自建"这块主流市场，TensorRT-LLM 锁定了"极致性能 + NVIDIA Blackwell"这条线，TGI 在 2026 年 3 月已经归档、停在 maintenance mode、HF 官方推荐大家迁移到 vLLM 或 SGLang。Llama 4 Maverick 在 B200 上跑到 40000+ tok/s，TensorRT-LLM 是业内最优解。**不是 vLLM 一家独大，而是 vLLM 拿开源、TensorRT-LLM 拿极致。** 你做的是通用对话 / RAG / Agent 业务场景，vLLM 够了；你做的是超大模型 + 极限吞吐 + NVIDIA 专属集群，TensorRT-LLM 比 vLLM 快一档。这一句跟选型决策树那边呼应起来，免得漏了。

## 三 第二道闸 显存调度

![第3节配图](assets/66_AI推理成本不是按token计费那么简单——GPU利用率加显存调度加批处理三道闸把电费降到三分之一/03-kv-cache-schedule.png)

第一道闸把 GPU 干活的时间填满了，第二道闸要解决的是——内存怎么用。

显存对 LLM 推理来说比计算还金贵。同样一张 H100 80GB 的卡，跑 7B 模型还是 70B 模型、用 8K 上下文还是 128K 上下文，决定能不能塞下。**显存塞满了，要么加卡、要么排队，要么降精度——三个选项都不便宜。**

显存调度的核心思路是把"一次性大开销"切成"分时复用"。三个工程动作：

**动作一：KV cache 共享。**

每一条推理请求都要存一份"键值缓存"——相当于模型读完上文后留一份"我刚才读过这个"的备忘。两万个用户的请求，system prompt 大段重复，传统做法是每个用户单独存一份，10 GB 显存白白浪费。

KV cache 共享的做法是：识别 system prompt 的公共前缀，只存一份，不同用户引用同一个前缀。**显存占用从 10 GB 砍到 2 GB，单卡并发翻 5 倍。**

Anthropic 2024 年发布的 Prompt Caching 政策就是这个思路的云端版本——cache write 多花 25%，cache read 节省 90%。业内有团队做过测算：同一段 system prompt 被复用 5 次，总成本只有重写一次的 1.65 倍；复用 10 次，只有 1.31 倍。**缓存不只是技术，是商业模式。**

**动作二：Prefix 缓存。**

KV cache 共享是"system prompt 这部分共享"，Prefix 缓存是把"任何长上下文的前缀"都拿来缓存。比如让 AI 读完整本《红楼梦》再回答问题——"读完"这件事可以缓存，下次问别的也不用再读一遍。**前缀缓存的命中率，是 LLM 长上下文场景的命脉。**

vLLM 的 prefix caching 已经做到 industry-leading——命中率 80% 以上是常态，长的 system prompt 场景（客服、企业内部知识库、Code Review 助手）命中率能到 95%。

**动作三：多 LoRA 共享底座。**

LoRA（Low-Rank Adaptation）是给大模型做"轻量定制"的技术——不用重训模型，给模型挂一个 100 MB 上下的小适配器就能学会新技能。

传统的多 LoRA 部署是每个 LoRA 都占一份显存，5 个 LoRA 就是 500 MB 显存专门放它们——这部分显存没法用来算实际推理。**共享底座的做法是把不同 LoRA 的"底座"合并，让多个 LoRA 共用一份基础权重，只在最后一层做差异化。**

显存省 30%~50%，而且切换 LoRA 的延迟从秒级降到毫秒级。**多 LoRA 共享底座是"一个模型服务多个客户"场景的杀手锏。** 想象一下：你给 20 个大客户做定制客服助手，每个助手有自己的"风格"LoRA，但都共享同一个 70B 的底座——显存只占 1 份底座 + 20 个 LoRA 的 100 MB，20 个客户互不干扰。

显存调度的最后一道细节，是把"显存不够了"的应急预案事先写好。**显存是会爆的——某个长上下文请求突然涌进来、某个模型忘了释放旧 KV cache、某个 LoRA 加载完了忘了一直占着。** 真正的工程纪律不是"显存别爆"，而是"显存爆了也不能让用户感觉"。具体做法是显存监控（每 30 秒打点）+ 显存溢出（自动驱逐最旧 KV cache，LRU 策略）+ 显存回落到安全线（释放冷 KV cache，预留 20% buffer）。**显存不是无限资源，是要定期打扫的储藏室。** 不打扫的储藏室迟早塞满，塞满就是 503。

这三件事叠起来，显存调度的本质就一句话：**显存不是买来就用，是分时复用、命中优先、共享底座。** 每省下 1 GB 显存，就等于多了 1 GB 并发容量；多 1 GB 并发，单 token 成本就降一点。

这是第二道闸的精要。

## 四 第三道闸 批处理

第三道闸解决的是——零碎请求怎么组织起来。

如果说第一道闸是"让 GPU 别闲着"，第二道闸是"让显存别白堆"，第三道闸就是"让请求别零散"。**批处理的本质是优化"等待"——等的过程 GPU 闲着、KV cache 占着、KV cache 还在重新算。**

三个工程动作，让零碎活变得有秩序：

**动作一：动态 padding。**

LLM 处理请求时，输入长度对齐到 batch 内最长那条。传统做法是把所有请求一律塞到固定长度（32K），短的浪费、长的是上限。**动态 padding 只对齐"这一个 batch 内的实际最长"，短的不补、长的不删，浪费降到接近零。**

大模型处理器的 padding 是个细节，但汇总起来差距很大——同样 10000 次请求，动态 padding 比固定 padding 节省 15%~25% 的计算开销。这一动作对短请求占比高的场景（比如客服、分类）尤其有效。

**动作二：异构请求合并。**

客服场景里有的请求是"短文本分类"（300 token 进、5 token 出），有的是"长对话生成"（2000 token 进、500 token 出）。传统做法分两个队列分别批处理，长队列等得久、短队列时快时慢。

**异构请求合并是把"长短请求拼成一批"，让短请求填长请求的空闲阶段。** LLM 的 decode 阶段是自回归——一个 token 一个 token 吐，长请求吐到一半还空着 GPU 算力，短请求可以插进来跑。

vLLM 2024 年的版本开始支持异构批处理，到 2026 年已经是社区默认。这套思路在生产环境实测下来，**单卡利用率再涨 20%~30%。**

**动作三：异步队列削峰。**

实时请求（用户问一句话、AI 回一句话）和离线请求（客服质检、日志摘要、对话归档）混在一个集群里，高峰期一定会打架。

**异步队列削峰是把实时和离线分开——实时队列专攻延迟，离线队列专攻吞吐，离线任务在高峰期自动降优先级。**

具体做法是：所有请求进 Kafka / RabbitMQ 之类的消息队列，实时请求走"快车道"（优先调度、超时熔断），离线请求走"慢车道"（高峰让步、低谷独占）。**高峰期实时延迟不抖，离线任务也别闲着——夜里 12 点到早上 6 点，离线队列把当天的对话全跑一遍。**

讲到这里你应该看明白了：三道闸不是三件孤立的事，而是一个"层层叠加"的体系。第一道闸让 GPU 别闲着、第二道闸让显存别白堆、第三道闸让请求别零散。**三道闸焊死，账单才有降下来的可能。**

把三道闸比作电力调度：第一道闸像给每台空调装定时器（让 GPU 工作时段填满）、第二道闸像给每户装储能（让显存复用率最高）、第三道闸像给小区错峰用电（让高峰低谷拉平）。**三层都对齐了，电费自然降。**

讲到这里你应该看明白了，三道闸的"焊死"不是堆配置堆出来的，而是把每一层都按"电"的逻辑重做一遍——电表装好了能看见用了多少、电储能做好了能复用、错峰闸做好了能拉平。**AI 推理不是把硬件能力榨干，而是把电用得有节奏。** 节奏感是工程师设计出来的，不是云厂商白送的。

## 五 五类优化菜谱

![第4节配图](assets/66_AI推理成本不是按token计费那么简单——GPU利用率加显存调度加批处理三道闸把电费降到三分之一/04-five-optimizations.png)

三道闸是地基，五类优化是菜谱。地基做好，再谈菜谱——顺序不能反。

五类优化分别是：量化、模型蒸馏、提示词缓存、模型路由、降级 fallback。每个优化都不是免费的，都有要付的代价。

**菜谱一：量化（INT8 / FP8 / INT4 / Q4_K_M）。**

量化是把模型权重的"数字精度"砍下来——从 FP16 砍到 INT8，磁盘大小减半、显存减半，速度提升 20%~40%。

但量化不是越狠越好。业内有"行业甜点" Q4_K_M——Llama-3.1-8B 量化到 Q4_K_M 之后，磁盘 4.92 GB（相对 BF16 16.1 GB 减少到 31%），HumanEval 和 MMLU 精度损失 < 2%。**这就是 llama.cpp 社区的 6 年共识——Q4_K_M 是性价比最高的选择。**

更激进的选择是 Q2_K（磁盘 20%）、IQ1_M（磁盘 14%），但精度损失 5%~15%，业务能不能接受就要看场景。**GPT-5.5 输出 30 美元/百万 token，你量化一个本地 7B 模型，免费——前提是精度损失在你接受范围内。**

量化的代价是 **"延迟换精度"**——Q4 量化比 FP16 慢 10%~20%（因为权重要反复读出来），但精度换来了 4 倍显存节省。**用更慢的单次推理、换取更高的并发密度，从单 token 成本看反而便宜。**

量化不是越狠越好——INT4 把延迟降到 1/3 但准确率掉 5%，业务能不能接受？这句话适合每个量化团队写进决策表里。

**菜谱二：模型蒸馏（Distillation）。**

蒸馏是让一个大模型（教师模型）教小模型（学生模型）——把 70B 模型的知识蒸馏到 8B，账单降一个数量级。

但蒸馏不是万能的。**Llama 70B 蒸馏到 8B 适合客服（FAQ 类、多样性低），不适合法律分析（事实严谨、长链推理）。** 教师模型很强，不代表学生模型能在所有场景上赶上。

业内有团队花 5 万美元蒸馏一个 8B 模型，结果上线发现它在金融场景里跟原模型差 8 个百分点——这 5 万蒸馏费打了水漂。**蒸馏之前先验证：在你的业务场景里，小模型是否能跑出"可接受"的效果。** 没验证就上，是拿生产当赌注。

蒸馏要走的 4 步流程：① 准备高质量训练数据（业务真实场景 5 万-10 万条问答）；② 用教师模型生成"软标签"（不只是答案对错，是答案的概率分布）；③ 在学生模型上 fine-tune，保留住教师模型的核心推理路径；④ 在业务场景上做 A/B 测试，验证效果能不能上 95% 阈值。**这 4 步每一步都不能省，省一步就是给生产埋定时炸弹。**

**菜谱三：提示词缓存（Prompt Caching）。**

Anthropic 在 2024 年 8 月首发 Prompt Caching，2026 年 7 月所有主流厂商（OpenAI / Anthropic / 阿里云百炼）都已支持。**Cache write 多花 25%，Cache read 节省 90%。**

OpenAI 直接把 cached input 列在价格表第一列，Anthropic 在官网横幅上写"-90%"。**这不是技术决策，是商业决策——头部厂商把"缓存"从高级功能重新定义为商业默认。**

但提示词缓存命中率不到 30%，别上线——浪费工程时间。命中率低于 30% 意味着大部分请求"基本没命中"，你投入的工程时间换来的是电费账单上多出一行"cache write +25%"，反而比不缓存更贵。**30% 是行业经验线，低于这个数字就别硬上。**

要拉到 70% 的命中率，通常的做法是：**把 system prompt 设计成"稳定前缀 + 可变后缀"**——把工具说明、角色设定、输出格式这些不常变的内容放在前面，让 cache 自动命中；每次请求里变化的部分（用户问题、对话历史）放在后面，让 cache 不去尝试命中这种"不稳定的尾巴"。

**菜谱四：模型路由（Model Routing）。**

简单问题用本地 7B、复杂问题用云端大模型——这种"分级诊疗"是 2026 年的核心策略。

Anthropic Haiku 4.5 输入 1 美元/MTok、输出 5 美元/MTok；OpenAI gpt-5.4-mini 输入 0.75 美元/MTok、输出 4.5 美元/MTok；OpenAI gpt-5.4-nano 输入 0.20 美元/MTok、输出 1.25 美元/MTok。**这些"轻量级"模型足以解决 70% 的客服类、FAQ 类、分类类问题，剩下的 30% 才需要 GPT-5.5 / Opus 4.8 这种旗舰。**

路由不是排大小的序，是按"任务复杂度"分流——一个 7B 的小模型对 FAQ 类回答更稳定，70B 的大模型反而容易"过度回答"。**贵的模型不一定好，合适的模型才是好。** 这句金句你在上一篇也看过了。

模型路由不是技术问题，是把对的病送到对的医院。这一句值得你单独记一下，往后我们会专门拆开讲——等下一个章节展开细谈分级诊疗怎么把每条请求分到对的模型上。

**菜谱五：降级 fallback。**

复杂问题超时自动切回小模型、模型返回不稳定时切到备用模型、整条服务挂掉时切到本地兜底——这些都是"降级 fallback"。

降级 fallback 不是兜底，是分级诊疗的最后一道墙。**主路径挂掉之前，用户体验先保留——这才是降级的真正价值。**

举个例子：AI 客服在高峰期偶尔超时，自动降级到本地 7B + FAQ 知识库——虽然回答质量略降，但用户没被"系统正忙"的提示挡在外面；接下来的 1 分钟里，工程师有窗口排查问题，等恢复后再切回主路径。**这 1 分钟是 fallback 给工程师争来的，不是用户等出来的。**

5 类菜谱不是选一个上，是组合上。但组合上之前，先把三道闸焊死——地基不牢，菜谱做再花也站不住。

## 六 三个反常识金句

三道闸 + 五类菜谱焊好之后，账单当然会降。但有三条反常识金句，张工那场闭门会上是最后才悟出来的——

**第一个反常识：量化不是越狠越好。**

INT8 不一定够用，Q4_K_M 才是真甜点。**业务能不能接受那 5% 的精度损失，决定了你能压到哪一档。** 量化不是贪便宜，是做减法的平衡。

**第二个反常识：蒸馏不适合所有场景。**

Llama 70B 蒸馏到 8B 适合客服，不适合法律分析。**蒸馏前先在业务里跑实测，不要拿生产当赌注。** 5 万蒸馏费 + 8 个百分点的精度差距，等于把公司的法律声誉赌上了。

**第三个反常识：缓存命中率不到 30%，别上线。**

命中率不到三成，cache write 那多出来的 25% 就把 cache read 那省的 90% 反过来盖掉了。**这就跟你开了 5 个水龙头但只接到 1.5 个一样，水费比不开还贵。**

这三句话，张工团队写在了工位墙的另一边——左边是"先装电表"，右边是"别贪便宜"。**AI 推理的工程纪律，就这么两句。**

## 七 五类模型选型决策树

![第5节配图](assets/66_AI推理成本不是按token计费那么简单——GPU利用率加显存调度加批处理三道闸把电费降到三分之一/05-final-quote.png)

三道闸是工程动作，五类菜谱是优化手段，真正决定账单的是"任务送到哪个模型"。这一节我们讲五类模型选型场景。

**场景一：试探期 / MVP。**

日均 < 1M token。**首选 OpenAI gpt-5.4-mini 或 Anthropic Haiku 4.5——0 运维、按量付费，账单随业务增长。** 不要自建 GPU，自建门槛高、运维贵，MVP 阶段投资不值。

**场景二：成长期 / 日均 10M-100M token。**

**首选 API + Prompt Caching——Anthropic cache write +25% / cache read -90%，OpenAI cached input -90%，把"重复利用"做成一等公民。** 这一阶段核心动作不是优化模型，是优化缓存命中率。命中率每涨 10%，账单降 9%。

**场景三：规模化 / 日均 100M+ token。**

**首选自建 vLLM 集群（FP8 量化、Q4_K_M 兜底）。** 前提是 GPU 利用率能稳定 60% 以上——低于这个线，自建反而比 API 贵 2-5 倍。**本地部署不是省钱省出来的，是利用率高到能让电费摊薄。**

**场景四：数据合规 / 私有部署。**

金融、政务、医疗等强监管场景。**首选自建 vLLM + Q4_K_M 量化 + 国产芯片（华为昇腾 / 阿里含光 / 字节跳动火山 ML 平台）。** 这条路最贵也最稳，账单不会爆，因为根本没出口，模型全在你家服务器上。

**场景五：边缘 / 离线 / 端侧。**

智能音箱、车载、IoT 设备。**首选 Ollama / llama.cpp + GGUF Q4_K_M。** 0 基础设施、几百 MB 模型就能跑，账单为零。

决策树的核心不是"哪个模型最强"，而是"哪个场景用哪个模型最适合"。**贵的模型不一定好，合适的模型才是好。** 这句话你在上一篇已经看过了，这一篇我们要再加一句——**模型的贵不贵不只看单价，看三件事：利用率、缓存命中、命中率，三件事都做对了，单价贵的模型也可能最便宜。**

还有一个隐藏维度：模型组合。**真正的工程实践里，不是"用一个模型干所有活"，而是"用两三个模型分工"——小模型处理 70% 的简单任务、中模型处理 25% 的中等任务、大模型处理 5% 的疑难杂症。** 这种组合比"全用大模型"或"全用小模型"都便宜，效果还稳。

选型决策树的最后一道，是"成本上限线"。每个业务都有自己的预算上限，比如年度 GPU 不能超过 50 万。**决策树不是找最便宜的方案，而是在预算上限内找效果最优。** 这是把"无限追求降价"扭回"够用就行"的关键——不是花最少的钱，而是花得最有价值。

张工那家公司后来定下了自己的决策原则：每季度评估一次模型价格变化，每年评估一次自建 vs API 的盈亏平衡点，每次新模型上线之前必须做 A/B 测试。**AI 推理选型不是一次定终身，而是和电价一样要跟着市场调。** 模型迭代太快，决策树也得跟着迭代。

## 八 真实成本对比表

这一节我们用一组对比，给三道闸 + 五类菜谱的效果一个直观感受：

**同一套客服业务（约 10 亿 token/月），6 种部署方案的月度账单：**

| 方案 | 模型 | 月度账单 | 节省率 | 关键技术 |
|---|---|---|---|---|
| 1. 纯 GPT-5.5 直接调用 | gpt-5.5 | 25 万 | 0%（基线） | 不优化 |
| 2. 路由分流 + 缓存 | gpt-5.5 + gpt-5.4-mini + Prompt Caching | 14 万 | -44% | 模型路由 + 缓存 |
| 3. vLLM + PagedAttention + 连续批 | 自建 Llama-3.1-70B-FP8 + vLLM | 11 万 | -56% | 三道闸全套 |
| 4. 量化到 Q4_K_M + 蒸馏 8B | 自建 Llama-3.1-8B-Q4_K_M + 蒸馏 | 6.5 万 | -74% | 量化 + 蒸馏 |
| 5. vLLM + 5 类优化全上 + 国产芯片 | 自建 Qwen3-7B + vLLM + 缓存 + 路由 | 4.8 万 | -81% | 三道闸 + 5 菜谱 |
| 6. 张工闭门会方案 | 实时 gpt-5.4-nano + 离线 Batch + Prompt Caching | 8 万 | -68% | 三道闸 + Batch |

**6 种方案的关键不是"哪个最便宜"，是"哪个组合匹配你的业务场景"。** 张工那家 AI 客服创业公司走的是方案 6——实时任务用 gpt-5.4-nano（输入 0.20 美元/MTok），离线质检用 Batch API（-50%），所有长 system prompt 走 Prompt Caching（-90%）。最后算下来 8 万。

8 万这件事不是奇迹，是**三道闸 + 五类菜谱 + 一份预算表**共同起作用的产物。**AI 推理成本不是省出来的，是设计出来的——每一笔 token 都被看见、被预测、被拦截、被复盘。**

方案 5 的 4.8 万看上去最便宜，但它牺牲的是"不上 AI 5.5 那一档旗舰模型"——业务能不能接受这个质量妥协，要看场景。方案 6 的 8 万看上去不是最便宜，但它平衡了"质量 + 成本 + 运维难度"三个维度，是大多数创业公司能走的路。

再看一组工程视角对比：同样 8 万这一个最终数字，方案 6 走的是"API + 缓存 + Batch"——基本没有 GPU 运维，工程师花在"电"上的时间占比 10% 以下；方案 5 走的是"自建 + Q4_K_M + 国产芯片"——省钱但要养一支工程团队负责运维。**不是"省钱省出来的"省钱，而是"运营复杂度换出来"的省钱。** 这条路上，工程师的角色从"调模型"变成"养电"——调度、监控、扩容、备份，全是电力调度的活儿。

这一节讲的对比表，看上去是 6 个价格，骨子里是 6 种工程哲学。**选哪个方案，看的不是账上能省多少，而是团队能不能扛住"养电"的复杂度。** 钱能省，但团队时间和精力不能省；账上少 2 万、团队多耗 1 个工程师月薪，省下来的就不是省下来的。

## 九 金句收束

写到这里，这篇要收束了。

回到第一篇的张工——那个被凌晨 3 点电话叫醒的 CTO。25 万的账单敲了 23 天，复盘会又开了三天。三道闸焊死、五类菜谱配齐，月度账单从 25 万降到 8 万。

这不是省钱，是把 AI 推理系统从"月光族"切到"精打细算的电力调度"。**三道闸不是把 AI 训得更快，而是把 GPU 的每一秒钟都用在刀刃上。** 五类菜谱不是五张底牌，而是五道工程纪律。**缓存不是技术，而是把每一笔 token 都看得见、可预测、可拦截。**

这是第一篇的反差开场留下的悬念——"三道闸怎么把电费降到 1/3"。答案是：第一道闸 GPU 利用率从 20% 提到 70%、第二道闸显存分时复用节省 80%、第三道闸批处理填充空闲再涨 30%、五类菜谱在底下压成本、模型选型决策树把对的病送到对的医院。**这些动作叠起来，账单降 68%。**

AI 时代的成本治理比电费治理更难，因为 GPU 看不见、电表没人装、漏电保护器要在生产挂掉那一天才想起来装。**把家用电表装上、把分时家电配上、把节能灯泡换上、把家庭药盒整理好、把漏电保护器装上、把路由策略写好、把 fallback 备好。** 这一整套做下来，AI 系统才像一个能经营的家。

这是经营一个家、搬进新家过日子这条主线的延续(本组覆盖从运维到 AI 计费)——从消防演习到备份恢复，从多环境治理到配置中心，从定时任务到长期陪客户，再到 AI 推理的电费治理。**AI 时代把家经营好的每一步，都是工程纪律的一步；工程纪律不到位，账单就来敲门。**

回过头看 上一篇那句"接 AI 进业务之前，先问问自己：电表装了没有"，这一期把它升级成"电表装了之后，再问问自己：电闸焊死没有、储能调好没有、错峰接通没有、节能灯换上没有、漏电保护器装上没有、路由写到配置中心没有、备品备件准备齐全没有。"——**AI 系统的工程纪律不是一件一件分开的，是一整套连在一起、电不通、家不安、账单不断。**

这就是我们从这一期开始想传达给工程师的核心命题——**AI 推理成本从来不是省出来的，是设计出来的。** 设计这个词，听上去像"图漂亮"，其实它的意思是"让账单可预测、可拦截、可复盘"。把这三件事都做下来了，AI 才能从"定时账单炸弹"变成"能放心经营的家"。

经营一个家不是一件易事，AI 推理系统也不是——它要养、要调、要更新，还要在你下班之后独自撑过整个夜晚。**这个家好不好过，不取决于你买了多大的 GPU，而取决于你愿不愿意花时间去理解每一笔电费背后的工程动作。** 张工团队在闭门会最后一天记下的日记里写过这么一句话——"电表装好了不再看，等于没装。" 这句话，跟 上一篇那句"接 AI 进业务之前，先问问自己：电表装了没有"是同一条线的两端——一边是"装"，一边是"看"，中间这条主线，是工程纪律的命。

最后送你一句——

**敢点火是勇气，能收住才是本事。**
